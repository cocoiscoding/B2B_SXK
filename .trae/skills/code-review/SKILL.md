# Code Review / 项目代码审查指南

> 本 Skill 用于指导大模型对一个**已经写好代码的项目**，按照**标准化、严格化、可追溯**的流程进行专业 Code Review。以系统架构师与资深工程师的视角，**严谨、全面、分级、不臆测**地审查代码，发现潜在的安全、正确性、性能、并发、可维护性等问题，最终输出一份**结构化、带严重程度分级、带证据定位**的代码审查报告。
>
> 一句话定位：**代码审查的本质是"多维度缺陷发现 + 风险定级 + 证据归档"——以工程标准与最佳实践为标尺，系统性地扫描代码库，找出偏离标准之处，并按危害程度排序，而非泛泛点评或逐行朗读。**

---

## 一、技能定位与适用场景

### 1.1 适用场景
- 项目代码已基本完成（或某模块/某次提交已完成），需要在交付、合并、上线前进行**质量把关**。
- 用户提出形如："帮我 review 一下这个项目的代码""对当前代码做一次 code review""审查一下这次提交的代码""上线前帮我过一遍代码质量"。
- 需要从**安全、正确、性能、并发、可维护、架构**等多个维度系统性发现缺陷与风险。
- 既要找到"有问题的那一行"，更要给出"问题等级 + 证据 + 修复建议"。

### 1.2 不适用场景（边界）
- **定位某个具体 Bug 的根因**（以现象+报错找根因）→ 交给 `bug-localization`。
- **理解某个功能怎么实现的**（只读解析）→ 交给 `feature-localization`。
- **启动项目 / 跑起来**→ 交给 `project-bootstrap-guide`。
- **生成项目全景上下文文档**→ 交给 `project-init-context`。
- **开发新功能 / 重构 / 性能优化（写代码）**→ 非本 Skill 范畴；本 Skill 只输出审查报告与建议，不直接改代码。
- 纯风格/格式自动检查（如 ESLint/Prettier/Checkstyle 能自动修复的）→ 建议直接跑工具，本 Skill 聚焦需人工判断的深层问题。

> **与 bug-localization 的关键区别**：bug 定位是"已知有问题，逆向找根因"；代码审查是"未知是否有问题，正向全面扫描风险"。前者收窄聚焦，后者全面覆盖。

### 1.3 目标产物（代码审查报告）
经过本 Skill 的审查后，应输出一份结构化**代码审查报告**，至少包含：
1. **审查概述**（审查范围、规模、技术栈、总体质量结论）。
2. **问题汇总表**（按严重程度分级统计：Critical / Major / Minor / Info）。
3. **逐条问题清单**（每条含：编号、严重程度、维度类别、问题描述、证据 `文件:行号` + 代码片段、影响分析、修复建议）。
4. **维度覆盖矩阵**（确认每个审查维度都已覆盖，无遗漏）。
5. **亮点与合规项**（做得好的地方，避免纯批评）。
6. **置信度（高/中/低）**与待确认项 / 未能覆盖的部分。

### 1.4 核心原则（心智模型）
- **只读审查（Read-Only）**：审查阶段**不修改任何项目代码**，审查与修复职责分离。报告中的"修复建议"明确标注为**建议**，是否实施由用户决定。
- **证据驱动，拒绝臆测**：每一个"这里有问题"的结论都必须附**具体 `文件:行号` + 代码片段**作为证据；没有证据的推断必须标注为"疑似/待验证"。
- **多维度全面覆盖**：安全、正确、健壮、性能、并发、可维护、架构、数据、API、可测试——**十个维度逐项过**，不可只盯单一维度。
- **分级客观，抓大放小**：按**危害程度（而非个人偏好）**定级，Critical/Major 优先，Minor/Info 不淹没重点。
- **标准为尺，避免主观口水**：以**行业最佳实践、安全规范、语言社区规约、项目自身约定**为判断依据，而非"我觉得这样不好看"。
- **相关性 ≠ 缺陷**：搜到某个关键词或模式不代表它有问题，必须确认它在该上下文下确实构成风险。
- **不教条，权衡语境**：警惕"过度设计""为安全而安全"的反向问题；建议需考虑项目实际阶段与约束。

## 二、第一步：审查需求的结构化解析（Input Triage）

拿到审查请求后，**先不要急着逐文件阅读**。先把模糊的审查意图拆成结构化字段，并对齐范围与标准。这一步决定了审查的方向与深度是否正确。

### 2.1 必须澄清的 7 个要素

| 要素 | 说明 | 示例 / 追问话术 |
|------|------|----------------|
| **审查范围** | 审查哪些代码 | 全项目 / 某模块（`src/auth`）/ 某次提交（`git diff`）/ 某些文件 |
| **审查目的** | 为什么审查 | 上线前把关 / 合并前把关 / 安全审计 / 技术债评估 / 新人代码把关 |
| **关注维度** | 是否有侧重点 | 全面 / 重点看安全 / 重点看性能 / 重点看并发 |
| **严格程度** | 到什么颗粒度 | 只报 Critical+Major / 含 Minor / 连风格也报 |
| **技术栈上下文** | 项目是什么栈 | 已知则复用；未知则阶段0侦察 |
| **项目约定/规范** | 有无团队规范 | 是否有编码规范、lint 配置、安全基线可对照 |
| **已知线索** | 用户已知的疑点 | "我觉得这块并发有问题""上次泄露过密钥" |

### 2.2 审查范围分类（决定覆盖策略）
将审查目标归入以下类别之一，后续覆盖策略（第三章、第四章）据此调整：

- **A. 全量项目审查**：整个代码库首次/周期性全量过一遍。
- **B. 模块/子系统审查**：聚焦某个模块（如认证模块、支付模块）。
- **C. 增量/提交审查（Diff-based）**：只审查某次提交/分支相对基线的变更（合并前最常见）。
- **D. 单文件/小组审查**：审查指定的一组文件。
- **E. 主题专项审查**：只审一个主题（如安全专项、性能专项、并发专项）。

### 2.3 信息完备度评估与追问清单（健壮性保障）
现实中用户的审查请求往往**范围模糊、目的不清**。遇到关键缺失，**必须先主动追问**，而非基于猜测硬审。追问时给出具体、可执行的指引：

- 范围不明 → 请用户明确"是全项目、某模块，还是某次提交/分支的变更？"
- 严格程度不明 → 请用户确认"只要 Critical+Major，还是连 Minor/风格也要？"
- 无团队规范 → 告知"未发现团队编码规范/lint 基线，我将以语言社区通用最佳实践为标准；若有团队规范请提供。"
- 增量审查但无版本控制 → 请用户指明基线（哪个分支/tag），或切换到全量审查。
- 关注维度不明 → 默认全面覆盖十大维度，并说明"我会全面审查，若要聚焦请告知"。

> **原则**：宁可先对齐一句"你要审什么、审到什么程度"，也不要在错误范围或错误深度上跑很远。范围跑偏 = 大量无效劳动。

## 三、核心方法论：六阶段代码审查流水线

面对任何代码审查请求，严格按以下六个阶段推进。**每个阶段都有明确的"要做什么、得到什么结论、不满足则回退"**。

```
审查请求 ──► [阶段0 技术栈与架构侦察] ──► [阶段1 范围界定与入口清点]
                                                        │
              [阶段5 报告生成] ◄── [阶段4 定级与确认] ◄── [阶段2 多维度逐项审查]
                                      ▲                    │
                                      └──── [阶段3 交叉验证与一致性检查] ◄──┘
```

### 阶段 0：项目技术栈与架构侦察（Recon）
**目标**：在审查前，先建立"项目地图"与"审查基线"，确定该用什么标准去衡量。**若已有同项目的前序分析（project-init-context / project-bootstrap-guide 产出），优先复用，避免重复侦察。**

**侦察动作**：
1. 识别技术栈指纹（`package.json` / `pom.xml` / `go.mod` / `requirements.txt` / `composer.json` / `Cargo.toml` 等），确定**语言、框架、版本**——这决定了适用哪些安全规范与最佳实践（如 Java 适用 OWASP、Eslint/Checkstyle 规约；Go 适用 Effective Go；Python 适用 PEP8）。
2. 判定**分层架构与目录职责**：识别 controller / service / repository / model / module / domain 等分层位置，建立"每层该做什么、不该做什么"的职责基线（用于架构维度审查）。
3. 识别**项目自身的约定/规范**：
   - Lint / 格式化配置（`.eslintrc` / `.prettierrc` / `checkstyle.xml` / `.golangci.yml` / `ruff.toml`）→ 作为代码质量基线。
   - 安全配置（依赖扫描配置、安全头配置、认证中间件）→ 作为安全基线。
   - 测试目录（`test/` / `__tests__/` / `src/test/`）→ 用于评估测试覆盖。
4. 判定**关键横切关注点的实现位置**：认证/授权（守卫/拦截器/中间件）、异常处理（全局异常过滤器/`@ControllerAdvice`）、日志、配置加载——这些是高频缺陷区，需提前定位。
5. 识别**数据层与外部依赖**：ORM/数据库、缓存、消息队列、第三方服务——用于数据一致性、性能、安全维度的审查。

**结论产出**：一份"审查基线地图"——技术栈与适用规范、分层职责、横切关注点位置、关键依赖。后续阶段据此选择审查策略与判断标准。

> **关键**：不同技术栈有不同的高危模式（如 PHP 的 SQL 注入、Node.js 的原型链污染、Java 的反序列化）。阶段0决定了你"该找什么"。

---

### 阶段 1：审查范围界定与入口清点（Scope & Entry）
**目标**：精确界定本次要审查哪些文件/变更，建立"审查清单"，确保**无遗漏、无越界**。

**按审查范围分类（2.2）确定清点策略**：

| 范围类型 | 清点策略 |
|----------|----------|
| **A 全量** | 列出全部源码文件（排除 `node_modules`/`vendor`/`dist`/`build`/第三方生成代码），按模块/分层分组，形成审查清单 |
| **B 模块** | 列出该模块目录下全部源码文件 + 其直接依赖的跨模块文件（交叉引用） |
| **C 增量** | 用 `git diff <base>..<head> --stat` 列出变更文件清单；**新增/修改/删除**分别标记 |
| **D 单文件组** | 直接以用户指定文件为清单 |
| **E 主题专项** | 根据主题筛选相关文件（如安全专项聚焦：认证、SQL、输入处理、密钥、文件操作） |

**增量审查（C）的特殊处理**：
- 不仅要看"改了什么"，还要看"改动是否破坏了未改动代码的假设"（如改了函数签名，调用方是否同步）。
- 删除/重命名需确认"是否有遗留引用"。
- 变更行数少 ≠ 风险低（一行 SQL 拼接改动可能是 Critical）。

**关键产出**：
- 一份**审查清单**（文件列表 + 各文件审查优先级标注）。
- 优先级标注原则：**涉及认证/授权/支付/数据写入/外部输入/并发/加密的文件 = 高优先级**；纯展示/工具/配置文件 = 中低优先级。

---

### 阶段 2：多维度逐项审查（Multi-Dimensional Scan）
**目标**：这是审查的核心。对照第四章的**十大审查维度**，对清单中的每一处代码逐维度检查，收集所有"发现项（Finding）"。**每个发现项必须记录：维度、文件:行号、代码片段、问题描述。**

**审查执行策略（两条线并行）**：

1. **逐文件精读（纵向）**：对清单中的每个文件，**逐函数/逐方法**阅读，对照十大维度检查。
2. **维度专项扫描（横向）**：对**高危维度**（尤其安全、并发），用全局搜索做"模式扫描"，避免逐文件阅读时遗漏。

**高危模式快速扫描清单（横向扫描抓手）**：

| 维度 | 高危模式（全局搜索命中后逐一核对） |
|------|------|
| 安全-SQL注入 | `SELECT.*\+`、`execute\(.*%`、`query\(.*\+`、字符串拼接 SQL、`@Query` 中含 `+` 或 `${}` |
| 安全-XSS | `innerHTML`、`v-html`、`dangerouslySetInnerHTML`、`document.write`、未转义输出 |
| 安全-敏感信息 | `password`、`secret`、`token`、`apiKey`、`privateKey` 出现在源码/日志/硬编码中 |
| 安全-命令注入 | `exec\(`、`Runtime.exec`、`child_process`、`os.system`、`subprocess` 接收拼接输入 |
| 安全-反序列化 | `pickle.loads`、`ObjectInputStream`、`unserialize`、`eval(` |
| 安全-路径遍历 | 文件操作中拼接用户输入路径、未校验 `..` |
| 并发-竞态 | check-then-act 模式（先查后改）、`if` 后紧跟写操作、共享可变状态无同步 |
| 并发-死锁 | 嵌套 `synchronized`/`Lock`、多个锁的获取顺序不一致 |
| 性能-N+1 | 循环内执行 DB 查询、循环内 `fetch`/远程调用 |
| 性能-阻塞 | 主线程/事件循环中的同步 IO、循环内大计算 |
| 健壮性-异常吞咽 | `catch.*\{\s*\}`、`except:` 空体、`catch(e){}` 无任何处理 |
| 资源泄漏 | `open(` / `new FileInputStream` 无 `try-with-resources` / `finally` 关闭 |

> **重要**：搜索命中 ≠ 确诊。每一条命中都要**回到上下文确认**它在该场景下是否真的构成风险（阶段3交叉验证）。

**逐文件精读的关注点（纵向）**：
- 读函数签名：**入参是否校验？返回值是否可能为空/出错？**
- 读函数体：**主逻辑是否正确？边界条件覆盖了吗？异常路径处理了吗？**
- 读数据操作：**是否在事务内？是否有并发风险？是否有 N+1？**
- 读外部交互：**超时？重试？降级？鉴权？**
- 读依赖使用：**框架/库的 API 是否被正确使用？版本是否有已知漏洞？**

**产出**：一个"原始发现项池（Finding Pool）"，每项含：维度类别、文件:行号、代码片段、问题描述、初步严重程度。

### 阶段 3：交叉验证与一致性检查（Cross-Verify）
**目标**：对阶段2收集的发现项做**去伪存真**与**全局一致性**检查，避免误报、漏报、矛盾。

**3.1 发现项真伪验证（去误报）**
对每个发现项，逐条确认：
1. **是否真实成立**：回到完整上下文，确认该代码路径**确实可达**、该风险**确实存在**（而非已被上游防护、已被框架处理、属测试桩代码）。
2. **是否已被缓解**：检查是否存在该文件之外的防护层（如全局参数校验拦截器、ORM 自动转义、网关 WAF）已经消解了该风险——若有，降级或标注"已有缓解"。
3. **是否适用当前语境**：判断该"问题"在项目的**实际阶段与约束**下是否成立（如内部工具无须苛刻鉴权、原型阶段可容忍部分债）。

**3.2 全局一致性检查（防矛盾、补遗漏）**
1. **同类一致**：同一种问题在不同文件的处理是否一致？不一致本身就是问题（如有的接口做了参数校验、有的没做）。
2. **跨文件调用链**：改动/审查是否破坏了跨文件的契约（如改了函数签名但调用方未同步、改了返回结构但前端未适配）。
3. **配置与代码一致**：配置项是否被代码正确消费？代码依赖的配置是否存在？
4. **文档与实现一致**：接口文档/注释/README 描述的行为是否与代码实际一致？不一致要报。

**3.3 漏报补查**
对照第四章维度矩阵，确认**每个维度都已实际审查**，而非因为"没找到问题"就跳过。对高优先级文件做二次复查。

**产出**：经过验证的发现项清单（剔除误报，补充遗漏，标注一致性差异）。

---

### 阶段 4：严重程度定级与确认（Severity Grading）
**目标**：对每个确认的发现项给出**客观、一致**的严重程度定级，使报告聚焦真正重要的风险。

**四级定级标准**：

| 级别 | 定义 | 典型示例 |
|------|------|----------|
| 🔴 **Critical（致命）** | 可直接导致**安全事故、数据丢失/泄露、生产崩溃、资金损失**；必须立即修复，阻断上线 | SQL 注入、硬编码生产密钥、越权访问、未鉴权的资金操作、可导致数据覆盖的并发缺陷 |
| 🟠 **Major（严重）** | 会导致**功能错误、数据不一致、严重性能问题、高概率崩溃**；上线前应修复 | 逻辑错误、缺失事务导致脏数据、N+1 严重性能问题、异常被吞导致静默失败、未处理的边界条件 |
| 🟡 **Minor（一般）** | **不影响正确性但降低健壮性/可维护性**；建议修复，可带病上线 | 命名不清、重复代码、缺少注释、魔法数字、缺少参数校验（低风险场景）、资源未及时释放（低风险） |
| 🔵 **Info（建议）** | **优化空间或最佳实践建议**；非缺陷 | 可用更优算法、可抽取公共方法、风格统一、可增加测试 |

**定级规则**：
- 以**实际危害**为准，不以"代码难看程度"为准。
- 同类问题在不同**风险面**下定级不同（如"缺参数校验"在公开接口=Critical/Major，在内部纯函数=Minor）。
- 若不确定定级，取**保守偏高**并在报告中说明依据，请用户复核。

**产出**：分级后的发现项清单，按 Critical → Major → Minor → Info 排序。

---

### 阶段 5：报告生成（Report）
**目标**：按第九章的输出规范，生成结构化、可执行、可追溯的代码审查报告。

**报告质量要求**：
- **可定位**：每条问题附 `文件:行号` + 代码片段。
- **可执行**：每条问题附明确的修复建议（而非"这里有问题你自己看着办"）。
- **分级清晰**：严重程度一眼可辨，Critical 不被 Minor 淹没。
- **覆盖透明**：附维度覆盖矩阵，让用户知道"哪些审了、哪些没审"。
- **客观平衡**：既报问题，也肯定合规项与亮点。

## 四、代码审查维度矩阵（Pattern Library）

这是本 Skill 的核心知识库。**审查时逐维度对照**，每个维度给出**缺陷候选清单 + 判断要点**。这是资深工程师经验沉淀的"速查表"。

### A. 安全性（Security）——最高优先级
**缺陷候选**：
1. **注入类**：SQL 注入（拼接 SQL / 未参数化）、命令注入（`exec` 拼接）、NoSQL 注入、LDAP/XPath 注入、模板注入。
2. **XSS**：未转义的 HTML 输出（`innerHTML` / `v-html` / `dangerouslySetInnerHTML`）、反射型/存储型 XSS。
3. **认证/授权缺陷**：缺失鉴权、鉴权可绕过、越权（水平/垂直）、权限校验遗漏、会话固定、token 不校验过期/签名。
4. **敏感信息泄露**：密钥/密码/token 硬编码、日志打印敏感字段、错误响应暴露堆栈/SQL、源码/配置进入版本控制（`.git`、`.env` 提交）、调试接口未关闭。
5. **加密误用**：弱算法（MD5/SHA1 用于密码、DES/ECB）、硬编码盐、随机数不安全（用 `Math.random` 生成 token）、密钥管理不当。
6. **不安全反序列化**：`pickle` / `ObjectInputStream` / `unserialize` / `eval` 处理不可信输入。
7. **路径遍历**：文件操作拼接用户输入、未校验 `..`、未限制根目录。
8. **SSRF**：服务端发起的请求 URL 来自用户输入且未做白名单。
9. **文件上传**：未校验类型/大小/内容、上传路径可控、可执行文件上传。
10. **CSRF**：状态变更操作缺失 CSRF 防护 / SameSite。
11. **不安全依赖**：`package.json` / `pom.xml` 中依赖含已知 CVE（高危版本）。
12. **安全配置**：CORS 过宽（`*` + 凭证）、安全头缺失（CSP/HSTS/X-Frame-Options）、HTTPS 未强制。

**判断要点**：**所有来自外部（用户输入、第三方回调、消息体）的数据，在进入"危险操作"（SQL/命令/文件/网络/反序列化）前，是否经过校验/参数化/转义？**

### B. 功能正确性（Correctness）
**缺陷候选**：
1. **逻辑错误**：条件判断反向、`==` 与 `===` 误用、短路求值误用、`&&`/`||` 用反、负数/边界比较错误、off-by-one。
2. **边界条件遗漏**：空集合/空字符串/`null`/`0`/负数/最大值/超大输入/越界访问未处理。
3. **类型问题**：隐式类型转换、浮点精度丢失（金额用 float）、整型溢出、JSON 数字精度。
4. **空值处理缺失**：未防御 `null`/`undefined`、可选链缺失导致 `Cannot read of undefined`、后端返回缺字段前端直接解构。
5. **返回值/状态未检查**：忽略函数返回值（如写入返回 false、map.put 覆盖）、状态机状态转移错误。
6. **算法/计算错误**：业务计算（折扣/积分/税率）公式错误、舍入方向错误、时区/时间戳换算错误。
7. **魔法值**：散落的硬编码数字/字符串含义不明，易改错。

### C. 健壮性与异常处理（Robustness）
**缺陷候选**：
1. **异常吞咽**：空 `catch`/`except`、`catch(e){}` 无处理无日志、`.catch(()=>{})` 静默。
2. **过宽捕获**：`catch(Exception)` 吞掉所有（含本不该吞的）、`except:` 裸捕获。
3. **异常信息丢失**：抛新异常未保留原异常（cause）、日志无堆栈、错误码丢失。
4. **错误恢复缺失**：失败后状态不一致（部分成功）、无补偿/回滚。
5. **外部调用无防护**：无超时、无重试/无限重试、无熔断/限流、无降级、无重试退避。
6. **降级策略错误**：失败时返回错误数据而非明确报错（更危险）。
7. **资源管理**：流/连接/锁/文件句柄未在 `finally`/`try-with-resources`/`using` 中释放 → 泄漏。

### D. 性能（Performance）
**缺陷候选**：
1. **N+1 查询**：循环内逐条查 DB / 远程调用（最常见、最致命的性能问题）。
2. **循环内 IO/重操作**：循环内打开连接、读文件、发起 HTTP。
3. **内存问题**：全量加载大结果集（无分页）、无界缓存/列表增长、大对象常驻、内存泄漏（闭包/监听器未移除）。
4. **算法复杂度**：O(n²) 嵌套在循环中、对大集合做线性查找（应用 Map/Set/索引）、重复计算未缓存。
5. **阻塞**：主线程/事件循环中的同步 IO、长任务阻塞请求线程、未来任务串行等待本可并行。
6. **缓存**：该缓存的热点数据未缓存、缓存无失效导致脏数据、缓存击穿/穿透/雪崩无防护。
7. **序列化开销**：不必要的深拷贝/序列化、大对象反复序列化。
8. **数据库**：缺失索引（WHERE/JOIN/ORDER BY 字段）、`SELECT *`、事务过大、锁粒度过粗。

### E. 并发与线程安全（Concurrency）
**缺陷候选**：
1. **竞态条件**：check-then-act（先查存在再操作，中间被插入）、读-改-写非原子（如 `count++`）。
2. **共享可变状态无同步**：多线程/协程访问的可变状态无锁/无 `volatile`/无原子操作。
3. **死锁**：嵌套锁且获取顺序不一致、锁内等待外部回调、持锁等待 IO。
4. **线程不安全的单例/共享对象**：懒加载单例非线程安全、共享 `SimpleDateFormat`/`ArrayList` 等非线程安全对象。
5. **集合并发修改**：遍历时修改、多线程读写非并发集合。
6. **异步问题**：Promise/async 未 await、异步回调未处理错误、异步任务泄漏（未取消）。
7. **并发数据一致性**：并发写覆盖、乐观锁版本未校验、缓存与 DB 双写不一致。

### F. 可维护性与代码质量（Maintainability）
**缺陷候选**：
1. **命名**：无意义命名（`a`/`tmp`/`data`/`flag`）、误导性命名、命名与行为不符、中英混用不一致。
2. **复杂度**：超长函数（圈复杂度过高）、深层嵌套（>4 层）、超大类（上帝对象）、超大文件。
3. **重复代码**：复制粘贴的逻辑（应抽取公共方法）、散落的相同校验。
4. **死代码**：未使用的变量/函数/导入/字段、被注释掉的代码、不可达分支。
5. **注释问题**：关键逻辑无注释、注释过时与代码不符、注释冗余（复述代码）、TODO/FIXME 未跟踪。
6. **职责混乱**：单一职责违反（一个函数干多件事）、分层穿透（Controller 含业务逻辑、Service 含展示逻辑）。
7. **硬编码**：配置/常量/URL 硬编码（应提取到配置/常量）。

### G. 架构与设计（Architecture）
**缺陷候选**：
1. **分层违规**：Controller 直接访问 DAO（绕过 Service）、UI 层直接操作 DB、循环依赖。
2. **耦合问题**：核心逻辑硬依赖具体实现（应依赖接口/抽象）、模块间直接依赖实现细节、违反依赖倒置。
3. **抽象问题**：抽象泄漏（底层细节暴露到上层）、过度设计（为不存在的扩展性增加复杂度）、设计模式误用。
4. **配置/环境耦合**：环境相关值硬编码、不同环境逻辑用 if-else 堆砌（应策略模式/配置注入）。
5. **错误处理架构**：异常处理散落各处无统一策略、错误码无统一定义。

### H. 数据一致性与持久化（Data）
**缺陷候选**：
1. **事务边界**：跨多操作的数据变更无事务、事务范围过大（含远程调用）、事务嵌套错误。
2. **幂等性**：重复提交/重试导致重复扣款/重复创建（关键写操作必须幂等）。
3. **数据完整性**：缺约束（唯一/非空/外键）、软删除逻辑不一致、级联缺失导致孤儿数据。
4. **并发写**：扣减/累加非原子导致超卖/余额错乱（应乐观锁/原子更新/行锁）。
5. **缓存一致性**：先删缓存还是先更新 DB 的顺序错误、双写不一致。
6. **迁移**：数据库迁移脚本不可逆/无回滚、迁移与代码版本不匹配。

### I. API 设计（API Design）
**缺陷候选**：
1. **入参校验缺失**：API 入口未校验必填/类型/长度/格式（尤其是对外接口）。
2. **契约不清**：返回结构不统一、错误码无规范、无文档或文档与实现不符。
3. **破坏性变更**：删除/重命名对外字段、改参数语义、未做版本兼容（合并前重点查）。
4. **RESTful/规范**：HTTP 方法误用（GET 改数据）、状态码滥用、URL 命名不一致。
5. **安全**：敏感字段在响应中明文返回、列表接口无鉴权/无分页/无上限。

### J. 可测试性（Testability）
**缺陷候选**：
1. **测试缺失**：核心业务逻辑/复杂计算无单元测试、关键路径无集成测试。
2. **可测性差**：硬依赖具体类（无法 mock）、方法内 `new` 依赖、依赖全局状态/当前时间（不可控）。
3. **测试质量**：断言不充分（只断言不报错）、测试间共享状态、测试依赖执行顺序、mock 过度（测了 mock 而非逻辑）。
4. **边界未测**：关键边界条件（空/0/负/超长/并发）无测试覆盖。

## 五、审查工具与命令速查（均为只读安全）

> 优先使用 IDE / 大模型的内置搜索与阅读能力（Glob / Grep / Read）。**禁止执行任何写入、删除、构建部署、网络写操作。**

### 5.1 范围界定（增量审查必备）
- **变更文件清单**：`git diff --stat <base>..<head>`、`git diff --name-only <base>..<head>`。
- **具体变更内容**：`git diff <base>..<head> -- <file>`。
- **提交历史**：`git log -n <N> --oneline`、`git log -p <file>`。

### 5.2 全局模式扫描（横向扫描武器）
- **安全高危模式**：搜索 SQL 拼接、`innerHTML`、硬编码密钥、`exec`/`eval`、反序列化、路径拼接。
- **并发高危模式**：搜索 `synchronized`/`Lock`/`atomic`、共享可变状态、check-then-act。
- **性能高危模式**：搜索循环内的 DB/远程调用、`SELECT *`、全量加载。
- **健壮性高危模式**：搜索空 `catch`、裸 `except`、缺失 `finally`/资源释放。

### 5.3 结构化检索（定位而非阅读）
- **Glob 找文件**：`**/*controller*`、`**/*service*`、`**/auth/**`、`**/config/**`、`**/test/**`。
- **依赖清单**：读 `package.json` / `pom.xml` / `go.mod` / `requirements.txt` 评估依赖与版本。
- **配置基线**：读 `.eslintrc` / `checkstyle` / 安全配置 / 测试配置。

### 5.4 辅助证据来源（只读）
- **接口文档**：Swagger / OpenAPI / 注解（佐证契约一致性）。
- **数据库 schema**：`*.prisma` / `migrations/` / `*.sql` / `@Entity`（佐证数据约束）。
- **CI 配置**：`.github/workflows` / `.gitlab-ci.yml`（佐证是否有自动化检查门禁）。

## 六、信息不完备时的降级策略（健壮性核心）

现实审查中，项目信息、运行时环境往往不全。本 Skill 必须能优雅降级，而非卡死或乱猜。

| 缺失项 | 降级策略 |
|--------|----------|
| 无团队编码规范 | 以**语言社区通用最佳实践**为标准（Java: 阿里/Google 规约；JS: Airbnb/Standard；Go: Effective Go；Python: PEP8），并明确声明所采用标准 |
| 不确定某调用是否可达 | 标注"待确认可达性"，列出调用方分析，若无法确认则保守按"可达"处理并降一级报告 |
| 无法运行静态分析工具 | 以人工代码审查 + 全局模式扫描替代，在报告中声明"未运行自动化扫描，建议补充" |
| 部分文件无法读取 | 在维度覆盖矩阵中标注"未覆盖文件清单"与原因，绝不假装已审 |
| 不确定依赖是否有 CVE | 列出依赖版本，建议用户用 `npm audit` / `snyk` / OWASP Dependency-Check 扫描，不臆断 |
| 业务规则不明确导致无法判断正确性 | 对**结构性/通用性**问题（安全/性能/并发/可维护）正常报告；对**业务正确性**标注"需业务方确认预期" |
| 增量审查但改动触及大量调用方 | 聚焦改动本身 + 直接调用方，标注"间接影响范围较大，建议补充全量回归" |

**置信度标注规则**：
- **高**：有明确代码证据 + 风险路径可达已确认 + 符合公认标准。
- **中**：有代码证据，但可达性/实际危害需运行时或业务确认，或标准属"建议"非"强制"。
- **低**：基于模式匹配命中但上下文证据不足——必须显式声明为"疑似，建议复核"。

## 七、防误判：常见陷阱（严谨性核心）

代码审查最怕"自信地报错问题"或"漏掉真问题"。以下陷阱必须主动规避：

1. **误报（假阳性）**：看到 `innerHTML` 就报 XSS，却没看输入是否来自可信源；看到拼接就报注入，却没看是否已参数化。**每条命中必须回到上下文确认。**
2. **漏报（假阴性）**：只看"明显的问题"，忽视**跨文件、跨层、时序性**的深层问题；或只看"改动的行"忽视"改动对未改动代码的影响"。
3. **标准错配**：用 Java 的规范去要求 Python，用生产级标准去苛责原型代码，用大型团队规范去套个人项目。**标准必须匹配技术栈与项目阶段。**
4. **教条主义**：把"建议/最佳实践"当成"缺陷/必须改"，制造噪音淹没真正的问题。**区分"缺陷"与"可优化"。**
5. **忽视已有防护**：框架/中间件/上游已提供防护（如 ORM 自动参数化、全局校验拦截器），却仍报该问题为 Critical。**要先确认是否已被缓解。**
6. **只报不赞**：纯批评式 review 令人沮丧且不公平。**合规项、亮点、设计巧妙处也要肯定。**
7. **修复建议不可执行**：只说"这里不安全"却不给具体改法，或给的建议脱离项目实际。**建议要具体、可落地、考虑影响面。**
8. **严重程度虚高/虚低**：把风格问题报成 Critical（噪音），把 SQL 注入报成 Minor（危险）。**严格按实际危害定级。**
9. **忽视测试与文档**：只审"功能代码"，漏掉测试质量与文档一致性。**可测试性与文档一致性也是审查维度。**
10. **越权改动**：在审查未完成/未经同意时就动手改代码。**审查与修复职责分离（合规约束）。**

## 八、安全与合规约束（Read-Only）

- **只读**：审查全程**不得修改、删除、新增**项目文件；不得执行写库、写文件、构建部署、发外部请求的命令。
- **最小授权**：只访问审查所必需的代码与配置；**不主动读取与审查无关的敏感数据**（如真实生产数据、用户隐私数据）。审查中发现的硬编码密钥等敏感信息，在报告中**打码脱敏**呈现，不完整泄露。
- **职责分离**：报告中的"修复建议"明确标注为**建议**，是否实施由用户决定并另起流程（可交由 `modification-plan-application` 执行）。
- **不越权**：不重启服务、不改配置、不触碰生产环境；如需运行时验证（如复现漏洞），由用户在其受控环境内完成并反馈。
- **可追溯**：所有结论附 `文件:行号` 或代码片段证据，便于用户复核。
- **诚实标注边界**：当审查范围/维度因信息不足未能完整覆盖时，**明确告知未覆盖部分**，不伪造"已全面覆盖"。
- **保密**：审查报告可能含安全漏洞细节，提醒用户**妥善保管报告**，勿公开泄露未修复的漏洞信息。

## 九、代码审查报告输出规范

完成审查后，按以下结构输出。**语言与用户保持一致**（用户用中文则中文输出）。报告应**分级清晰、证据确凿、建议可执行**。

```markdown
## 📋 代码审查报告

### 1. 审查概述
- **审查范围**：<全项目 / 模块 / 增量提交 / 文件组>
- **审查规模**：<N 个文件，约 M 行代码>

## 十、工作流总览（速记）

```
1. 解析需求（7要素 + 范围分类 + 标准对齐，模糊则追问）
        │
2. 阶段0  技术栈与架构侦察 ──► 建立审查基线（复用已有分析）
        │
3. 阶段1  范围界定与入口清点 ──► 审查清单（文件 + 优先级）
        │
4. 阶段2  多维度逐项审查（纵向逐文件精读 + 横向高危模式扫描）──► 发现项池
        │
5. 阶段3  交叉验证（去误报 + 一致性检查 + 漏报补查）──► 确认的发现项
        │
6. 阶段4  严重程度定级（Critical/Major/Minor/Info，按实际危害）──► 分级清单
        │
7. 阶段5  报告生成（概述 + 汇总 + 详情 + 覆盖矩阵 + 亮点 + 置信度）
```

**贯穿全程的红线**：只读、证据驱动、十维度全面覆盖、分级客观抓大放小、标准匹配语境、相关性≠缺陷、每条建议可执行、信息不全先追问。
