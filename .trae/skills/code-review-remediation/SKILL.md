# Code Review Remediation / 代码审查修复指南

> 本 Skill 用于指导大模型在**已经完成 Code Review 并拿到审查报告**之后，**根据报告中的发现项逐项修复代码**，并**严格保证修复不破坏原有功能（无回归）**。以系统架构师与资深工程师的视角，**严谨、分级、可验证、最小侵入**地完成"审查发现项 → 修复方案 → 执行修复 → 回归验证"的全过程，最终输出一份**逐项可追溯、功能保全有据**的修复报告。
>
> 一句话定位：**代码审查修复的本质是"在'把审查报告的问题改全'与'绝对不改坏现有功能'之间取得精确平衡"——以审查发现项为修复依据，以功能保全为不可逾越的红线，用最小有效变更消除缺陷，每一处修复都配以验证。**

---

## 一、技能定位与适用场景

### 1.1 适用场景
- 项目代码已**完成 Code Review**（无论是由 `code-review` Skill 产出、人工 review 留下的评论、还是 PR/MR 的 review comments），现在需要**根据审查结果修改代码**。
- 用户提出形如："根据 review 结果改一下代码""把审查报告里的问题都修了""按这些 review 意见修复代码，但别改坏功能""CR 反馈的这些问题帮我处理掉"。
- 既要"改全"（审查报告中的每个发现项都要有明确的处置：修复 / 延后 / 驳回），更要"改稳"（修复后原有功能不能回归）。
- 审查报告给出的通常是**问题定位 + 修复建议**（而非精确的 before→after 代码），因此修复需要"设计"——把建议转化为具体、正确的代码变更。

### 1.2 不适用场景（边界）
- **还没有做 Code Review，直接想提升代码质量** → 先用 `code-review` 出报告，再回到本 Skill。
- **有一份精确的修改方案/diff，需要忠实复刻到另一个项目** → 交给 `modification-plan-application`（它复刻精确变更，本 Skill 是"根据建议设计并执行修复"）。
- **定位某个具体 Bug 的根因**（以现象找根因）→ 交给 `bug-localization`。
- **理解某个功能怎么实现的**（只读解析）→ 交给 `feature-localization`。
- **功能 A 仿照功能 B 的思路改造** → 交给 `feature-pattern-porting`。
- **开发全新功能 / 大规模重构**（非缺陷修复导向）→ 非本 Skill 范畴（属于通用编码）。

> **与 modification-plan-application 的关键区别**：modification-plan-application 的输入是**精确的修改方案**（diff / before-after 代码块），任务是**忠实复刻、零偏差**，方案怎么写就怎么改；本 Skill 的输入是**审查发现项 + 修复建议**（建议是方向性的，非精确代码），任务是**设计正确的修复并执行**，核心约束是**功能保全**。前者是"执行蓝图"，后者是"据图施工并自证不伤结构"。
>
> **与 code-review 的衔接关系**：code-review 是**只读的缺陷发现**（产出报告，不动代码）；本 Skill 是 code-review 的**下游执行环节**（消费报告，修复代码）。两者可串联使用：先 review 出报告，再 remediation 修复。

### 1.3 目标产物（修复报告）
经过本 Skill 的修复流程后，应输出一份结构化**修复报告**，至少包含：
1. **修复概览**（审查报告共 N 项发现项，处置分布：已修复 / 延后 / 驳回 各多少）。
2. **逐项修复明细**（每项含：发现项编号与严重程度、问题描述摘要、修复位置 `文件:行号`、修复策略、改前→改后对照、影响域评估、验证结果）。
3. **功能保全验证结果**（构建/类型检查/测试的通过情况；无回归的确认）。
4. **延后项与驳回项清单**（为什么延后/驳回、建议的后续处理）。
5. **连带变更声明**（为修复某问题而必须连带修改的部分，如更新调用方、补充 import）。
6. **置信度（高/中/低）**与待确认项 / 未能验证的部分。

### 1.4 核心原则（心智模型）
- **功能保全是第一优先级（Do No Harm）**：任何修复都**不得破坏原有功能**。一个引入回归的"修复"比原来的问题更危险——原来的问题至少是"已知的"，回归是"未知的"。**当"修复彻底性"与"功能保全"冲突时，功能保全优先。**
- **修复有据可循**：每一处代码变更都必须**追溯到审查报告中的某个具体发现项**（含编号、位置）。无发现项不改码；有发现项必须有明确处置（修复/延后/驳回），不可"视而不见"。
- **先理解后修复**：**绝不在未充分理解原始代码意图、数据流、行为边界的情况下动手修复。** 修复前先搞清楚"这段代码现在在做什么、为什么这么写、改了会牵动什么"，再设计修复。
- **最小有效变更**：用**最小的、最精准的变更**消除缺陷，不做"顺手优化""顺手重构""顺手清理"。修复的范围严格限定在发现项所指出的问题，不扩大爆炸半径。
- **修复与验证配对**：每一处修复**必须配以验证**——构建/类型检查/测试/心智执行至少择其一。未经验证的修复不算"完成"。
- **契约守恒**：默认**不改变代码的对外契约**（API 签名、返回结构、DB schema、事件名、公开方法签名）。必须改变时，做涟漪分析并显式声明、获用户确认。
- **分级处理、抓大放小**：优先修复 Critical/Major；Minor/Info 可批量处理或合理延后。在"修复紧迫度"与"回归风险"之间做权衡。
- **变更可回溯**：每处改动记录"对应哪个发现项 + 改前改后 + 验证结果"，便于用户逐项复核。

## 二、第一步：审查报告的结构化解析（Report Triage）

拿到审查报告（或 review 意见）后，**先不要急着搜代码、更不要动手改**。先把报告彻底读懂、拆透，并评估修复约束。这一步决定了后续修复的方向是否正确——**误读一个发现项，整处修复都会跑偏。**

### 2.1 审查报告的常见形态识别

现实中，审查结果**形态多样、粒度不一**。必须先识别其形态，再选择对应的解析策略：

| 形态 | 特征 | 示例 | 可执行性 |
|------|------|------|----------|
| **① 结构化报告** | 有编号、分级、位置、建议的结构化报告（`code-review` Skill 产出） | "CRITICAL-1：SQL注入，位置 user.ts:42，建议参数化" | 🟢 高 |
| **② PR/MR 评论** | 代码评审平台上的行内评论，附带文件行号 | GitHub/GitLab PR 上对某行的 review comment | 🟢 高 |
| **③ 散列 review 意见** | 用户口头/聊天转述的审查意见，可能无行号 | "reviewer 说这块并发有问题，还有异常处理太粗糙" | 🟡 中（需定位） |
| **④ 清单式意见** | 一条条列出的修改要求，有/无定位 | "1.加参数校验 2.修空指针 3.SQL参数化" | 🟡 中 |
| **⑤ 混合形态** | 以上多种组合 | 主体是结构化报告 + 几条口头补充意见 | 🟡 按条目分别处理 |

> **原则**：无论哪种形态，都必须把每一条审查意见**结构化为标准发现项**（见 2.2），统一管理、逐项处置。散落的、未结构化的意见最容易漏改。

### 2.2 必须提取的结构化字段

对审查报告中的**每一条独立发现项（Finding）**，提取以下字段，组装成"修复任务单元"：

| 字段 | 说明 | 必填 |
|------|------|------|
| **发现项编号** | 报告中的编号或自分配序号（F1, F2, ...），用于追踪 | ✅ |
| **严重程度** | 报告标注的 Critical / Major / Minor / Info；未标注则由本 Skill 评估 | ✅ |
| **审查维度** | 安全 / 正确 / 健壮 / 性能 / 并发 / 可维护 / 架构 / 数据 / API / 可测试 | ✅ |
| **问题位置** | 报告指出的 `文件:行号` / 函数名 / 代码片段 | ✅ |
| **问题描述** | 一句话概括"问题是什么" | ✅ |
| **修复建议** | 报告给出的修复方向（可能是具体代码，也可能是方向性描述） | ✅ |
| **修复风险预估** | 该修复的回归风险：低 / 中 / 高（本 Skill 评估，见 2.4） | ✅ |
| **处置决策** | 修复 / 延后 / 驳回（待阶段 2-3 确定后回填） | 待定 |

> **"发现项"的划分标准**：一条审查意见 = 一个发现项 = 一个独立的修复任务单元。即使多条意见指向同一文件，也各自独立编号，便于逐项追踪与核对。

### 2.3 必须澄清的修复约束要素

修复前，必须与用户对齐以下**约束条件**——它们决定了修复的边界与策略：

| 要素 | 说明 | 追问话术 / 默认策略 |
|------|------|---------------------|
| **修复范围** | 全部修 / 只修 Critical+Major / 只修指定几项 | "审查报告共 N 项，您要全部修复，还是只修 Critical+Major？"
| **契约可变性** | 能否改 API 签名 / 返回结构 / DB schema | "有些修复需要改对外接口，是否允许？还是必须保持契约不变？"默认：契约守恒
| **功能保全验证手段** | 有无测试可跑 / 能否构建 / 仅靠推理验证 | "项目有没有测试套件？能否执行构建？"默认：以构建+类型检查+心智执行验证
| **已知约束** | 不能引入新依赖 / 不能改某模块 / 有时间限制 | "修复有什么限制条件？"
| **连带变更授权** | 修复某问题时是否允许连带改调用方/共享代码 | 默认：必要的连带变更允许，但需显式声明

### 2.4 修复风险预估（回归风险评估）—— 本 Skill 独有的评估维度

审查报告的**严重程度**衡量的是"问题本身的危害"；而修复的**回归风险**衡量的是"改这里有多容易改坏别的东西"。两者**正交**，必须同时评估，共同决定修复优先级与策略：

| 回归风险 | 定义 | 典型场景 | 默认策略 |
|----------|------|----------|----------|
| 🟢 **低风险** | 改动**局部、独立、无外部调用方依赖**，改完不影响其他代码 | 参数化一个独立 SQL、修正一个局部比较运算符、补充一个空值检查 | 可放心修复，轻量验证 |
| 🟡 **中风险** | 改动**被多处调用**或涉及**共享逻辑**，改后需确认调用方行为不变 | 修改一个被多处调用的工具函数的逻辑、改变某方法的内部处理顺序 | 修复后必须搜索全部调用方逐一确认 + 测试验证 |
| 🔴 **高风险** | 改动涉及**对外契约**、**核心数据流**、**共享基础设施**、**复杂并发/事务逻辑** | 改 API 签名/返回结构、改 DB schema、改共享基类、改事务边界 | 必须做完整影响域分析 + 涟漪修复 + 用户确认 + 全面回归验证 |

> **关键洞察**：一个 Minor 问题可能位于高风险区（如改一个被 50 处调用的公共工具函数），一个 Critical 问题可能位于低风险区（如独立模块里的 SQL 注入）。**严重程度决定"修不修"，回归风险决定"怎么修、修完怎么验"。**

### 2.5 信息完备度评估与追问清单（健壮性保障）
现实中，审查报告可能**不完整、有歧义、建议不可行**。遇到以下情况，**必须先主动追问或显式标注**，而非基于猜测硬改：

| 问题 | 表现 | 处置 |
|------|------|------|
| **定位不明** | 意见只说"这块有问题"但没给文件/行号 | 在代码中搜索定位最可能的位置，标注假设请用户确认 |
| **建议不可行** | 建议用了项目没有的依赖/框架特性 | 标注"建议依赖项目不具备"，提出等效替代方案请用户选择 |
| **意见有歧义** | "改一下这个判断"——改哪个判断？怎么改？ | 列出 2-3 种理解请用户确认 |
| **意见间冲突** | 两条意见对同一处代码给出矛盾建议 | 显式列出冲突，请用户裁决 |
| **业务规则不确定** | 意见涉及业务正确性但预期行为不明 | 对结构性问题（安全/性能/并发）正常修复；对业务正确性标注"需业务方确认预期" |
| **范围模糊** | 不知道要修全部还是部分 | 用 2.3 的追问话术确认 |

> **原则**：宁可先对齐一处歧义，也不要在错误理解上改出一批错误。审查意见理解错一处，到修复阶段会被放大成系统性偏差。

## 三、核心方法论：六阶段修复流水线

面对任何"根据审查结果修复代码"的请求，严格按以下六个阶段推进。**每个阶段都有明确的"要做什么、得到什么结论、不满足则回退"**。

```
审查报告 ──► [阶段0 项目技术与代码现状侦察]
                        │
            [阶段1 发现项解析与修复分流] ──► 修复任务清单(逐项+风险标注)
                        │
            [阶段2 修复方案设计与影响域分析] ──► 每项的修复方案 + 涟漪分析
                        │
            [阶段3 修复优先级排序与确认] ──► [用户确认 Gate]
                        │
            [阶段4 修复执行] ──► 逐项落地 + 即时验证
                        │
            [阶段5 功能验证与回归保障] ──► 不满足则回退/补改
                        │
                   输出修复报告
```

### 阶段 0：项目技术与代码现状侦察（Recon）
**目标**：在修复前，先建立"项目地图"与"修复基线"，确定技术栈约定、构建/测试能力、版本控制状态。**若已有同项目的前序分析（project-init-context / project-bootstrap-guide / code-review 产出），优先复用，避免重复侦察。**

**侦察动作**：
1. 识别技术栈指纹（`package.json` / `pom.xml` / `go.mod` / `requirements.txt` / `composer.json` / `Cargo.toml` 等），确定**语言、框架、版本**——这决定了修复时该用什么惯用法（如参数化查询用什么 API、校验用什么装饰器）。
2. 判定**分层架构与目录职责**（controller / service / repository / model / module），确认审查报告引用的文件路径都能对上。
3. 确认**构建与测试能力**：
   - 构建命令是什么？（`npm run build` / `mvn compile` / `go build` / `tsc`）→ 用于修复后的编译验证。
   - 测试命令是什么？（`npm test` / `mvn test` / `go test` / `pytest`）→ 用于回归验证。
   - 有无类型检查？（TypeScript 的 `tsc --noEmit`、Java 的编译期检查、Python 的 mypy）。
   - 有无 lint？（`.eslintrc` / `checkstyle` / `golangci-lint`）→ 可辅助验证修复质量。
4. 确认**版本控制状态**（是否干净的 working tree？建议用户先 commit/建分支，确保可回退）。

**结论产出**：一份"修复基线"——技术栈与惯用法、分层结构、构建/测试命令、版本控制状态。后续阶段据此选择修复策略与验证手段。

> **关键**：技术栈决定了"怎么修才对"（如 Java 用 `PreparedStatement`、Node.js 用参数绑定、Python 用参数化 cursor），构建/测试能力决定了"修完怎么验"。阶段 0 确保修复既正确又可验证。

---

### 阶段 1：发现项解析与修复分流（Parse & Triage）—— 本 Skill 的起点
**目标**：把审查报告**彻底拆解**为结构化的修复任务清单，并对每项做**修复可行性评估与风险标注**。**这是后续一切修复的基础——漏拆一个发现项就会漏修，误判一个风险就会改坏。**

#### 1.1 逐条提取发现项（遵循第二章 2.2 字段表）
- 对审查报告中的**每一条意见**，提取一个"修复任务单元"。
- **不遗漏**：报告的每一段、每一条评论、每个附注都要检查。常见漏改来源：报告正文中"顺便……""另外……"的附带说明，或被折叠的评论。
- **不臆造**：报告中没有的问题，不自行新增（即使你发现了别的问题）——本阶段只消费报告，不产出新发现。

#### 1.2 逐项修复可行性评估
对每个发现项，回答：
1. **问题位置是否可定位？**（报告给了精确位置 → 可定位；只给了文字描述 → 需搜索定位，标注置信度）
2. **修复建议是否明确可行？**（给了具体改法 → 直接用；给了方向 → 需在阶段 2 具体化；未给建议 → 需自行设计）
3. **修复是否会触及契约/共享物？**（不改契约/不碰共享 → 低风险；改 → 高风险，需特殊处理）
4. **修复是否有前置依赖？**（如需要先加 DB 字段、先引入依赖 → 标注前置工作）

#### 1.3 修复任务清单（关键产出）
| 编号 | 严重程度 | 维度 | 问题位置 | 问题摘要 | 建议摘要 | 回归风险 | 可行性 | 处置(待定) |
|------|----------|------|----------|----------|----------|----------|--------|------------|
| F1 | 🔴 Critical | 安全 | user.ts:42 | SQL注入 | 参数化查询 | 🟢 低 | ✅ 明确 | 待定 |
| F2 | 🟠 Major | 健壮性 | pay.ts:88 | 异常吞咽 | 补日志+处理 | 🟡 中 | ✅ 明确 | 待定 |
| F3 | 🟡 Minor | 可维护 | util.ts:15 | 魔法数字 | 提取常量 | 🟢 低 | ✅ 明确 | 待定 |
| F4 | 🔵 Info | 性能 | order.ts:60 | N+1查询 | 批量查询 | 🔴 高 | ⚠️ 改动大 | 待定 |

**自检**：清单提取完毕后，**反向核对**——报告中的每一条意见是否都对应到了清单中的至少一个发现项？有无遗漏？

---

### 阶段 2：修复方案设计与影响域分析（Design & Impact）—— 正确性与功能保全的核心
**目标**：为每个待修复的发现项，**设计具体的修复方案**（把建议转化为正确、可执行的代码变更），并**分析每处修复的影响域**（会不会牵动其他代码）。**这是整个流水线最考验工程功底的一步：修复既要消除缺陷，又不能引入新缺陷。**

#### 2.1 修复前必做：理解原始代码（先理解后修复）
**对每个发现项，动手设计修复前，必须先读懂目标代码：**
1. **这段代码当前在做什么？**（功能意图、数据流向、输入输出）
2. **为什么会写成这样？**（是有意为之的权衡，还是确实的疏忽？——避免把"有意的设计"当 bug 修）
3. **改动会影响什么？**（谁调用这段代码？改动后调用方的假设是否还成立？）
4. **这段代码的边界行为是什么？**（空值、边界值、异常路径——修复后这些路径是否仍被覆盖）

> **红线**：**未理解原始代码意图就动手修复 = 盲改**。盲改是回归的最主要来源。宁可多花时间读懂，也不要"看起来这里不对，我直接改"。

#### 2.2 设计修复方案（把建议转化为正确的代码变更）
审查报告的"修复建议"通常是**方向性的**（如"建议参数化""建议加校验""建议用乐观锁"），而非可直接粘贴的代码。必须结合**项目的技术栈、编码约定、上下文**，设计出具体、正确的修复代码：

- **遵循项目既有约定**：项目已有的参数化方式、校验库、异常体系、日志框架——优先复用，不另造一套。
- **遵循技术栈惯用法**：Java 用 `PreparedStatement`/JPA 绑定、Node.js 用 `?`/`$1` 占位符、Python 用参数化 cursor、Go 用 `db.Query(sql, args...)`。
- **修复要完整**：不只改"发现项指出的那一行"，还要改**同一问题在同一文件/同类场景下的所有出现处**（审查报告可能只举了一例，但同类问题可能有多处）。
- **修复要正确**：改完的代码逻辑必须在项目的业务语境下成立——代入实际数据流做"心智执行"，确认改后行为符合预期。

#### 2.3 影响域分析（功能保全的保障）
对每处修复，回答**"改了这里，还会牵动什么"**：

| 影响源 | 含义 | 排查手法 |
|--------|------|----------|
| **调用方影响** | 改了某方法的行为，调用方是否依赖旧行为？ | 全局搜索该方法名/函数名的所有调用处，逐一判断改后是否仍正确 |
| **契约影响** | 改动是否触及 API 签名/返回结构/DB schema/事件名？ | 对比改前改后的对外契约，有变化则升级为高风险 |
| **共享物影响** | 改的是否是被多处引用的共享工具/组件/基类？ | 搜索共享物的所有使用方，确认改后使用方行为不变 |
| **数据流影响** | 改动是否影响数据流向、时序、事务边界？ | 追踪数据流，确认改后数据的读写顺序/事务范围仍正确 |
| **副作用影响** | 改动是否影响异步事件/消息/缓存的产生与消费？ | 检查事件/消息/缓存的产生点与消费点是否受影响 |

**影响域分析的产出**：每个修复方案的**影响域清单**（波及哪些文件/调用方/共享物）+ **是否需要连带变更**（如改了函数签名，需同步更新所有调用方）。

> **核心原则**：影响域分析的目的是**确保修复不产生意外的涟漪**。任何牵动调用方/共享物/契约的修复，必须把涟漪范围内的连带变更一并纳入修复方案，显式声明。

#### 2.4 连带变更识别
有些修复**必然牵连其他代码**，必须一并处理（否则修复不完整、会引入编译/运行错误）：
- 修改了函数签名 → 所有调用方需同步适配。
- 删除/重命名了变量/函数 → 所有引用处需同步更新。
- 修改了返回结构 → 消费该结构的代码需同步适配（或保持向后兼容）。
- 引入了新依赖 → 需更新依赖清单。
- 新增/修改了 DB 字段 → 可能需更新 ORM 实体定义。

> 这些连带变更**不是"顺手改"，而是"修复完整性的必要部分"**。但必须在修复报告中显式声明为"为修复 F-X 所需的连带变更"。

---

### 阶段 3：修复优先级排序与确认（Prioritize & Confirm）
**目标**：基于"严重程度 × 回归风险"，对所有发现项排序，确定修复批次与顺序，并在执行前获得用户确认。

#### 3.1 修复优先级矩阵（严重程度 × 回归风险）

| | 🟢 低回归风险 | 🟡 中回归风险 | 🔴 高回归风险 |
|------|----------|----------|----------|
| **🔴 Critical** | **P0：立即修复**（安全漏洞且改动安全） | **P0：立即修复 + 调用方验证** | **P1：修复 + 完整影响域分析 + 用户确认** |
| **🟠 Major** | **P1：优先修复** | **P2：修复 + 调用方验证** | **P2：修复 + 影响域分析 + 用户确认** |
| **🟡 Minor** | **P3：批量修复** | **P3：修复 + 轻量验证** | **P4：建议延后**（风险收益比低） |
| **🔵 Info** | **P4：可选修复** | **P4：可选/延后** | **延后或跳过** |

> **排序原则**：先修"高严重 × 低风险"（性价比最高、见效最快）；"高严重 × 高风险"的要慎修（务必做好影响域分析）；"低严重 × 高风险"的建议延后（为改一个 Info 问题而冒回归风险不值得）。

#### 3.2 延后与驳回机制
并非所有发现项都必须立即修复。以下情形可**合理延后或驳回**，但必须在报告中显式记录：
- **延后**：高回归风险且当前无充分验证条件（如无测试套件）→ 标注"建议补充测试后修复"。
- **延后**：修复需要较大的前置工作（如 DB 迁移、引入新框架）→ 标注"需排期专项处理"。
- **驳回**：审查意见有误（经核实不构成问题）→ 标注"经核实不成立，原因：……"。
- **驳回**：意见与项目实际约束冲突（如内部工具无须苛刻鉴权）→ 标注"在当前语境下不适用"。
- **降级**：意见的严重程度被高估（如把风格问题报成 Critical）→ 调整级别后处理。

> **原则**：每一项发现项都必须有**明确的处置决策**（修复/延后/驳回/降级），不可"忘记"或"跳过而不说"。处置透明是修复报告可信度的基础。

#### 3.3 用户确认 Gate（执行前必须确认）
在进入阶段 4 执行前，向用户呈现：
1. **修复任务清单总览**（N 项发现项，处置分布：修复 x / 延后 y / 驳回 z）。
2. **修复优先级排序**（按 P0→P4 排列）。
3. **高风险修复项的风险提示**（哪些修复回归风险高、影响域大、可能改契约）。
4. **连带变更声明**（哪些修复会牵动调用方/共享物/契约）。
5. **前置工作**（是否有需要先完成的 DB 迁移/依赖安装）。
6. 请用户确认修复范围与优先级后执行。

> **红线**：涉及**契约变更、DB schema 变更、共享基础设施变更、删除代码/文件**的修复，**必须获得用户显式确认**后方可执行。

---

### 阶段 4：修复执行（Apply）—— 逐项落地 + 即时验证
**目标**：在方案设计无误、用户确认后，**逐项、有序**地执行修复，**每项修复后立即验证**。**执行顺序有讲究：先改被依赖项，再改依赖方；先低风险后高风险；同文件的批量处理。**

#### 4.1 执行顺序原则
- **依赖先行**：若 F-A 的修复依赖 F-B 的结果（如 F-A 引用 F-B 新建的方法），先执行 F-B。
- **同文件聚合**：同一文件的多处修复聚合执行，避免反复读写同一文件导致行号漂移。
- **低风险先行**：先执行低回归风险的修复，建立"修复-验证"的节奏与信心，再处理高风险项。
- **批量与逐项的平衡**：低风险且独立的修复可批量执行后统一验证；高风险修复**必须逐项执行逐项验证**。

#### 4.2 逐项执行原则
- **一次一个发现项**（或一组同文件的发现项）：执行后立即验证，确认无误再进行下一项。
- **忠实执行设计方案**：按阶段 2 设计的修复方案执行，**不在执行中临时扩大范围或改变策略**。若执行中发现方案需调整（如发现实际代码与预期不同），暂停并回到阶段 2 重新设计。
- **精确界定变更范围**：只改修复方案指定的代码范围，**不多删不多加不改周边**。替换时精确匹配 before → after。
- **保持文件格式约定**：遵循项目的缩进风格、换行符、引号约定，不引入格式噪音。
- **即时验证**：每项（或每批）修复后立即执行验证（见 4.3）。
- **记录可回溯**：每处改动记录"发现项编号 + 文件:行号 + 改前摘要 + 改后摘要 + 验证结果"。

#### 4.3 即时验证（每项修复后的验证）
每项（或每批）修复后，根据回归风险级别执行验证：

| 风险级别 | 即时验证手段 |
|----------|-------------|
| 🟢 低风险 | 改后 Read 确认代码正确 + 心智执行确认逻辑成立 |
| 🟡 中风险 | 上述 + 搜索全部调用方确认行为不变 + （有测试则）跑相关测试 |
| 🔴 高风险 | 上述 + 全量构建/类型检查 + 全量测试 + 重点回归验证 |

> **原则**：验证不通过的修复，**不得进入下一项**——要么修正修复方案重新执行，要么回退该修复并标注"修复失败，待处理"。

---

### 阶段 5：功能验证与回归保障（Verify）—— "不破坏功能"的最终保障
**目标**：全部修复执行完毕后，进行**系统性的功能验证**，确保所有修复**未引入回归**，原有功能完整保留。**不做整体验证就交付 = 无法保证功能保全。**

#### 5.1 验证层次（从局部到全局）

| 层次 | 验证内容 | 手段 | 覆盖什么 |
|------|----------|------|----------|
| **L1 编译/类型** | 修复后的代码能否通过编译/类型检查 | 构建命令 / `tsc --noEmit` / 编译 | 语法错误、类型不匹配、引用断裂 |
| **L2 单元测试** | 修复是否破坏既有测试 | `npm test` / `mvn test` / `pytest` | 既有功能的逻辑回归 |
| **L3 影响域回归** | 受修复影响的调用方/共享物行为是否不变 | 针对影响域清单中的调用方做专项验证 | 涟漪范围的回归 |
| **L4 心智执行** | 对无法自动测试的部分，人工推演修复前后的行为差异 | 逐场景代入数据流推演 | 业务逻辑正确性、边界条件、异常路径 |
| **L5（可选）运行时** | 实际运行确认 | 用户在受控环境执行 | 运行时行为、集成正确性 |

#### 5.2 无回归确认清单
修复完成后，逐项确认：
1. **构建通过**：项目能正常构建/编译/类型检查通过？
2. **测试通过**：既有测试全部通过？（若有失败的，是因为修复改变了预期行为——需评估是否合理）
3. **调用方无损**：影响域清单中的所有调用方，行为是否与修复前一致？
4. **契约未破**（若承诺契约守恒）：对外 API 签名/返回结构/DB schema 是否未变？
5. **无越界**：抽查未被审查报告涉及的文件，确认未被改动？
6. **无残留**：修复未留下半成品代码、未删除导致编译错误的引用？

#### 5.3 不满足时的处理
- **构建/测试失败** → 分析是否修复引入的问题：若是，修正修复方案重新执行；若测试本身需更新（因修复改变了合理行为），更新测试并说明。
- **发现回归** → 定位是哪个修复引起的，回退或修正该修复。
- **影响域验证不通过** → 补充连带修复，直到调用方行为恢复一致。
- **无法自动验证**（无测试/无构建条件）→ 以 L4 心智执行验证为主，在报告中显式声明"未执行自动化验证，建议用户补充"，不伪造"已验证"。

## 四、按修复策略类型的执行矩阵（Pattern Library）

针对不同性质的修复，给出**修复要点 + 回归风险 + 易错点**。这是经验沉淀的"速查表"。

### ① 精确替换型修复（Precise Replacement）
- **场景**：问题定位明确，修复就是"把 A 改成 B"，逻辑清晰、范围局限。如：SQL 拼接改参数化、`==` 改 `===`、硬编码密钥改为从配置读取、补一个 `null` 检查。
- **修复要点**：精确替换目标代码，**顺带检查同类问题在其他位置的出现**（审查可能只报了一处）。
- **回归风险**：🟢 低（局部、独立）。
- **易错**：替换范围过大波及周边；只改了报告指出的一处，漏了同类其他处。

### ② 逻辑修正型修复（Logic Correction）
- **场景**：修复涉及**业务逻辑的调整**，如 off-by-one、条件判断反向、计算公式错误、状态转移遗漏。
- **修复要点**：**先彻底理解原有逻辑意图**，再修正；修正后必须代入**正常路径 + 边界路径 + 异常路径**做心智执行验证。
- **回归风险**：🟡 中（逻辑变化可能影响调用方对返回值/副作用的预期）。
- **易错**：没理解原意就改，把"有意的设计"当 bug 修；只改了正常路径，边界/异常路径没覆盖。

### ③ 防护增强型修复（Protective Enhancement / Additive）
- **场景**：在既有代码上**增加防护**，如补参数校验、补异常处理（try-catch + 日志）、补超时/重试/熔断、补资源释放（finally/try-with-resources）。
- **修复要点**：**添加型修复是回归风险最低的**——原有主逻辑不变，只在外围加防护。但要注意：新增的校验/异常**是否改变了正常路径的返回行为**（如校验失败直接返回错误，而原来不校验时能通过）。
- **回归风险**：🟢 低（增量为主，原有逻辑不动）。但若新增校验**拦截了原本能通过的输入**，则风险上升为 🟡。
- **易错**：新增的异常处理过于宽泛，吞掉了不该吞的异常；新增校验过严，误拦合法输入。

### ④ 性能优化型修复（Performance）
- **场景**：修复 N+1 查询、消除循环内 IO、补索引、加分页、优化算法复杂度。
- **修复要点**：性能优化**容易引入功能回归**（如批量查询改变了数据顺序、缓存引入脏读），**必须确认优化前后功能行为完全一致**。
- **回归风险**：🔴 中-高（性能优化常伴随数据流/时序变化）。
- **易错**：批量查询改变了结果顺序（原来逐条有序，批量后无序）；缓存失效遗漏导致脏数据；优化后分页逻辑改变。

### ⑤ 并发安全型修复（Concurrency Safety）
- **场景**：补同步（锁/原子操作/volatile）、改 check-then-act 为原子操作、修正线程不安全的单例/共享对象。
- **修复要点**：并发修复**必须验证不引入死锁、不降低并发度、锁粒度合理**。修复后做并发场景的心智执行。
- **回归风险**：🔴 高（并发逻辑细微变化都可能导致死锁/性能退化/新的竞态）。
- **易错**：加锁顺序不一致引入死锁；锁粒度过粗导致性能退化；用了非线程安全的替代品。

### ⑥ 结构/重构型修复（Structural Refactor）
- **场景**：修复分层违规（controller 含业务逻辑下沉到 service）、抽取重复代码为公共方法、消除上帝对象。
- **修复要点**：重构型修复**回归风险最高**，必须做完整影响域分析 + 全量验证。**优先采用"移动代码而非重写代码"**——把逻辑原封不动搬到正确的位置，不在搬运中"顺手优化"。
- **回归风险**：🔴 高（波及面大）。
- **易错**：搬运过程中"顺手"改了逻辑导致行为变化；抽取公共方法后参数/返回值与原 inline 代码不一致；重构波及了共享代码。

### ⑦ 配置/依赖型修复（Config / Dependency）
- **场景**：修复有漏洞的依赖版本、补安全配置（CORS/安全头）、修正错误的配置值。
- **修复要点**：依赖升级**务必检查版本兼容性**（有无 breaking change）；配置修改**确认不影响其他环境/功能**。
- **回归风险**：🟡 中（依赖升级可能引入不兼容；配置改动影响全局）。
- **易错**：依赖升级到不兼容版本；只改了配置文件没同步改消费配置的代码；环境差异导致配置在本地正常生产异常。

### ⑧ 新增测试型修复（Add Tests）
- **场景**：审查发现关键逻辑缺测试，需要补充单元测试/集成测试。
- **修复要点**：新增测试本身**不改变功能代码**（回归风险最低），但测试要**测对东西**（断言充分、覆盖边界、不测 mock）。
- **回归风险**：🟢 低（只加测试不改功能代码）。但若为了"可测性"而修改了功能代码（如提取方法、注入依赖），则该部分功能代码的修改需另行评估风险。
- **易错**：测试断言不充分（只断言不报错）；测试依赖执行顺序；为提升可测性改了功能代码却未验证。

## 五、修复工具与操作速查

> 阶段 0-3 为**只读分析**（理解代码 + 设计方案 + 影响域排查），优先使用内置搜索能力（Glob / Grep / Read）。阶段 4 进入**写入执行**，须在用户确认后执行修复。阶段 5 为**验证**（只读 + 可选的构建/测试命令）。

### 5.1 只读阶段（阶段 0-3）的工具
- **Glob**：按模式找文件，核对审查报告引用的路径（`**/user.service.*`、`**/auth/**`）。
- **Grep**：搜索目标代码位置、查找同类问题、排查调用方引用（搜函数名/方法名/事件名/表名）。
- **Read**：读取目标文件，理解原始代码意图、确认修复方案的可行性。
- **LS**：列目录结构，确认分层与模块组织。

### 5.2 影响域排查武器（阶段 2-3 核心）
- **搜调用方**：全局搜索要修改的方法名/函数名 → 列出所有调用处 = 潜在受影响方。
- **搜同类问题**：审查报了一处 SQL 注入，全局搜同类拼接模式，找到所有同类问题。
- **识别共享物**：判断要改的文件/方法是否被多处引用（搜 import / require / 调用）。
- **契约比对**：对比修复前后的 API 签名/返回结构/DB schema，确认是否破坏契约。

### 5.3 写入阶段（阶段 4）的工具
- **文件编辑**：使用 SearchReplace（精确替换 before → after）/ Write（新增文件），执行修复。
- **命令执行**（仅在验证时、用户确认后）：构建（`npm run build`）、测试（`npm test`）、类型检查（`tsc --noEmit`）。

### 5.4 验证阶段（阶段 5）的工具
- **构建/类型检查**：确认修复后代码可编译、类型正确。
- **测试**：运行既有测试套件，确认无回归。
- **Grep 复查**：修复后重新搜索高危模式，确认问题已被消除（如修复 SQL 注入后，确认该处不再有拼接）。

### 5.5 操作安全约定
- **版本控制意识**：建议用户在修复前 commit/建分支，确保可回退。大范围修复前主动提示。
- **逐项执行 + 即时验证**：一项改完、验证后，再改下一项。
- **禁止危险操作**：不执行 `rm -rf`、不强制覆盖、不碰 `.git` 目录、不重启服务、不改生产配置。

## 六、信息不完备时的降级策略（健壮性核心）

现实中，审查报告、项目信息、验证条件往往不全。本 Skill 必须能优雅降级，而非卡死或乱改。

| 缺失项 | 降级策略 |
|--------|----------|
| 审查意见定位不明（无行号） | 在代码中搜索最可能的位置，标注"定位置信度"，执行前请用户确认 |
| 修复建议不可行（缺依赖/框架不支持） | 标注"原建议不可行"，提出等效替代方案（用项目已有的手段实现同等防护）请用户选择 |
| 无测试套件（无法跑回归测试） | 以"构建验证 + 类型检查 + 心智执行 + 影响域逐项排查"替代，在报告中显式声明"未执行自动化回归测试，建议补充" |
| 无构建条件（无法编译验证） | 以"逐项 Read 确认代码正确 + 心智执行"替代，标注"未执行编译验证" |
| 不确定改动是否影响某调用方 | 全局搜索该调用方的引用做穷举；搜不到标注"未发现外部依赖（待确认）" |
| 业务规则不明确导致无法确认修复正确性 | 对结构性问题（安全/性能/并发/健壮性）正常修复；对业务正确性标注"修复逻辑需业务方确认预期行为" |
| 修复必然破坏契约且无法兼容 | 显式声明风险，提供"契约兼容过渡方案"（双写/灰度/版本号）供用户选择，不擅自破坏 |
| 高风险修复但无充分验证条件 | 标注"建议延后至具备验证条件后修复"，不冒险盲改 |
| 同一问题有多处出现但报告只报了一处 | 修复报告指出的那一处 + 全局搜索同类问题一并修复，在报告中声明"额外发现并修复了同类问题 N 处" |

**置信度标注规则**：
- **高**：所有发现项定位精确 + 修复方案明确且已验证（构建/测试通过）+ 影响域已排查 + 无回归。
- **中**：大部分发现项修复并验证，个别项定位需推断或验证手段有限（如无测试套件，仅心智执行验证）。
- **低**：多个发现项定位不精确/修复方案有不确定性/无法执行自动化验证 —— 必须显式声明"待验证"并列出所有不确定点。

## 七、防误判：常见陷阱（严谨性核心）

代码审查修复最怕"修了旧问题、引入新问题"或"改了不该改的、漏了该改的"。以下陷阱必须主动规避：

1. **盲改（未理解就改）**：没读懂原始代码的意图和行为边界就动手——**这是回归的头号来源**。**必须先理解再修复。**
2. **扩大范围（顺手改）**：修 SQL 注入时顺手"优化"了查询逻辑、重构了周边代码——扩大了爆炸半径。**只改发现项指定的问题。**
3. **只改一处（漏同类）**：审查报了一处 SQL 注入，代码里有 5 处同类拼接，只改了报的那 1 处——**必须全局搜同类问题一并修复。**
4. **修复引入回归**：加了参数校验但过严误拦合法输入、优化 N+1 但改变了结果顺序——**每处修复后必须验证功能不变。**
5. **忽视调用方**：改了某方法的内部行为/返回值，但没检查调用方是否依赖旧行为——**必须做影响域分析。**
6. **破坏契约不声明**：修着修着改了 API 签名/返回结构，却不告知用户——**契约变更必须显式声明并获确认。**
7. **污染共享物**：改了被多处引用的公共工具/基类，波及所有使用方——**共享物优先"扩展而非修改"。**
8. **臆造修复**：审查建议是"加校验"，但没给校验规则，自己编了一套不合理的规则——**建议不明确时应追问/按项目约定，不臆造。**
9. **不做最终验证**：逐项改完就交付，不跑构建/测试——**阶段 5 整体验证不可省略。**
10. **修复与建议脱节**：改出来的代码与审查建议不一致（如建议用参数化，实际改成了 ORM 转义）——虽可能等效，但应在报告中说明"采用了等效替代方案及原因"。
11. **漏处置发现项**：报告有 10 项，只处理了 8 项，另外 2 项"忘了"——**每项发现项都必须有明确处置（修复/延后/驳回），报告透明记录。**
12. **越权执行高风险操作**：未经确认就删代码/改依赖/改 DB schema——**高风险操作必须先确认。**
13. **为可测性改功能代码未验证**：补测试时为了让代码可测而提取了方法/注入了依赖，却未验证这些功能代码改动不引入回归——**可测性改造部分需同等验证。**

## 八、安全与合规约束

- **先只读、后执行、再验证**：阶段 0-3 全程**只读**（不得修改项目文件）；阶段 4 须在**用户确认后**执行修复；阶段 5 只读验证（可选执行构建/测试命令）。
- **功能保全是红线**：任何修复**不得破坏原有功能**。当修复彻底性与功能保全冲突时，**功能保全优先**；宁可延后一个高风险修复，也不冒险引入回归。
- **最小有效变更**：只修改发现项所指出的问题及相关连带部分；不做审查报告未涉及的"顺手优化/重构"。
- **破坏性操作显式确认**：删除代码/文件、改 API 契约、改 DB schema、改依赖清单、改共享基础设施——执行前必须向用户显式提示并获得确认。
- **最小授权**：只访问修复所必需的代码与配置；不主动读取无关敏感数据（密钥、隐私数据）。
- **不越权**：不重启服务、不改生产配置、不触碰生产数据；不执行 `rm -rf` 等危险命令；如需运行时验证，由用户在其受控环境内执行并反馈。
- **可追溯**：每处改动附"发现项编号 + 文件:行号 + 改前改后对照 + 验证结果"，便于用户逐项复核。
- **连带变更透明化**：所有为修复而必须的连带改动（调用方适配、import 更新等），必须在报告中显式声明。
- **延后/驳回透明化**：所有延后/驳回的发现项及原因，必须在报告中显式记录，不隐瞒。
- **诚实标注边界**：当验证条件不足、修复正确性无法确认、或功能保全无法保证时，明确告知，不编造"已修复且无回归"的假象。
- **建议版本控制**：修复前主动建议用户 commit/建分支，确保可回退。

## 十、工作流总览（速记）

```
1. 解析报告（形态识别 + 逐项提取发现项 + 约束对齐 + 风险预估，歧义则追问）
        │
2. 阶段0  项目技术与代码现状侦察 ──► 修复基线（技术栈/构建测试能力/版本控制）
        │
3. 阶段1  发现项解析与修复分流 ──► 修复任务清单（逐项 + 可行性 + 风险标注）
        │
4. 阶段2  修复方案设计与影响域分析 ──► 每项方案（先理解原始代码 + 设计修复 + 涟漪分析 + 连带变更）
        │
5. 阶段3  优先级排序与确认 ──► [用户确认 Gate]（严重×风险矩阵 + 高风险提示 + 契约变更确认）
        │
6. 阶段4  修复执行 ──► 逐项落地 + 即时验证（依赖先行/同文件聚合/低风险先行）
        │
7. 阶段5  功能验证与回归保障 ──► L1-L5 分层验证 + 无回归确认 ──► 不满足则回退/补改
        │
8. 输出修复报告（逐项明细 + 连带声明 + 延后驳回 + 验证结果 + 置信度）
```

**贯穿全程的红线**：功能保全第一（Do No Harm）、修复有据可循、先理解后修复、最小有效变更、修复与验证配对、契约守恒、分级处理抓大放小、变更可回溯、破坏性操作必确认、信息不全先追问。
