# Bug Localization / 用户反馈型 Bug 根因定位指南

> 本 Skill 用于指导大模型在**一个已经启动、相对完善的项目**中，根据**用户反馈的 UI 操作 + 现象/报错**，以系统架构师视角，**严谨、可追溯、不臆测**地反向定位到代码中的**真正根因（Root Cause）**，而非停留在表面症状。
>
> 一句话定位：**Bug 定位的本质是"因果溯源"——从可观测的现象（UI 表现 + 报错）逆向追溯到产生该现象的代码根因。**

---

## 一、技能定位与适用场景

### 1.1 适用场景
- 项目已能运行，用户反馈形如："在 XX 页面点击了 XX 按钮，没反应 / 弹窗报错 / 数据不对 / 页面白屏"。
- 已拿到用户的操作描述与（部分）报错信息，需要在代码库中定位问题出处。
- 需要区分**前端问题、后端问题、全链路问题**或**环境/数据问题**。
- 既需要找到"出错的那一行"，更需要回答"为什么出错（根因）"。

### 1.2 不适用场景（边界）
- 项目尚未启动 / 无法运行的初始化问题 → 交给"项目启动"类 Skill。
- 纯需求类问题（"这里应该多一个功能"）→ 非本 Skill 范畴。
- 性能优化、重构、新功能开发 → 非本 Skill 范畴。

### 1.3 目标产物（诊断报告）
经过本 Skill 的分析后，应输出一份结构化**诊断报告**，至少包含：
1. **一句话根因结论**（Root Cause）。
2. 现象复述与触发条件。
3. **完整证据链**（每一步均标注 `文件:行号` 或 `代码片段`）。
4. 涉及文件清单（前端 / 后端 / 配置）。
5. 根因分类（参见第六章高频清单）。
6. **置信度（高/中/低）**与剩余不确定点。
7. 修复建议（作为**只读分析**的产出，明确标注为"建议"，不擅自动手改）。

### 1.4 核心原则（心智模型）
- **只读分析（Read-Only）**：定位阶段**不修改任何项目代码**，诊断与修复职责分离。需要改动时另起流程并征得用户同意。
- **证据驱动，拒绝臆测**：每一个结论都必须有代码或运行时证据支撑；没有证据的推断必须标注为"假设"。
- **症状 ≠ 根因**：报错弹窗是"症状"，要顺着调用链往上找到"最早偏离预期的那一点"才是根因。
- **证据链闭合**：现象 → 入口 → 调用链 → 根因，环环相扣，任一环断裂则结论不可信。
- **全链路视角**：前端 / 后端 / 数据 / 网络 / 环境都可能是个环节，不要只盯一处。
- **相关性 ≠ 因果性**："最近改过 A"不代表"A 就是原因"；不要被时间巧合误导。

## 二、第一步：用户反馈的结构化解析（Input Triage）

拿到用户反馈后，**先不要急着搜代码**。先把非结构化的描述拆成结构化字段。这一步决定了后续定位的方向是否正确。

### 2.1 必须提取的 7 个字段

| 字段 | 说明 | 示例 |
|------|------|------|
| **操作路径** | 用户在哪个页面 / 路由 / 模块 | "用户管理页" `/system/user` |
| **操作动作** | 具体对哪个元素做了什么 | 点击"删除"按钮 |
| **操作对象标识** | 元素的文案 / id / icon | 文案"删除" |
| **期望行为** | 用户预期发生什么 | 该用户应被删除并列表刷新 |
| **实际现象** | 真实发生了什么（精确分类） | 无反应 / 弹错 / 数据错误 / 白屏 / 跳转错误 |
| **报错信息** | 文案 / 堆栈 / 状态码 / 控制台输出 | `500 Internal Server Error`、`Cannot read property 'id' of undefined` |
| **触发条件** | 必现 / 偶发 / 特定数据触发 | 仅当该用户有订单时才报错 |

### 2.2 现象分类（决定排查方向）
将"实际现象"归入以下类别之一，后续排查矩阵（第五章）按此分发：

- **A. 无响应（点击/提交后无任何变化）**
- **B. 弹窗/提示报错**
- **C. 数据异常（不显示 / 显示错误 / 部分丢失）**
- **D. 崩溃 / 白屏 / 跳转到错误页**
- **E. 偶发 / 间歇性问题**

### 2.3 信息完备度评估与追问清单（健壮性保障）
现实中用户反馈往往**信息不全**。遇到缺失项，**必须先主动追问**，而不是基于残缺信息硬猜。追问时给出具体、可执行的指引：

- 缺报错信息 → 请用户：① 按 F12 打开浏览器开发者工具 → Console 复制红色报错；② 切到 Network 重新操作，找变红的请求，截图状态码与响应体。
- 缺操作路径 → 请用户说明完整页面路径，或截图地址栏 URL。
- 缺触发条件 → 请用户确认是否每次必现，是否与某条特定数据相关。
- 后端项目 → 请用户提供操作时间点对应的后端日志片段（含 ERROR/堆栈）。

> **原则**：宁可多问一句，也不要在错误的方向上跑很远。信息残缺时定位成本会指数级上升。

## 三、核心方法论：五阶段因果溯源流水线

面对任何反馈，严格按以下五个阶段推进。**每个阶段都有明确的"要做什么、得到什么结论、不满足则回退"**。

```
现象 ──► [阶段0 技术栈侦察] ──► [阶段1 UI→代码入口] ──► [阶段2 入口→业务逻辑]
                                                                       │
       [阶段4 根因确认] ◄── [阶段3 报错溯源/全链路串联] ◄──────────────────┘
```

### 阶段 0：项目技术栈与结构侦察（Recon）
**目标**：在搜代码前，先建立"项目地图"，确定用什么手段去搜。

**侦察动作**：
1. 识别技术栈指纹文件（沿用项目启动 Skill 的指纹表：`package.json` / `pom.xml` / `go.mod` / `requirements.txt` 等）。
2. 判定**前端框架**（React / Vue / Angular / 原生）与**路由约定**：
   - 文件式路由：Next.js（`app/`、`pages/`）、Nuxt（`pages/`）→ 文件路径即路由。
   - 配置式路由：Vue Router / React Router 的路由表（搜 `routes`、`path:`、`createRouter`）。
3. 判定**后端框架与路由分发方式**（Spring 的 `@RequestMapping`、Express 的 `app.get`、Django 的 `urls.py`、Gin 的 `router.GET` 等）。
4. 判定**API 风格**（REST / GraphQL / RPC）与**前后端通信约定**（请求路径前缀、统一返回结构 `{code, msg, data}`）。
5. 判定**状态管理**（Redux / Pinia / Vuex / Context / Zustand）。

**结论产出**：一份"项目地图"——前端入口在哪、后端入口在哪、路由如何组织、API 如何命名。后续阶段据此选择搜索策略。

> 若已有同项目的前序分析记录（如项目启动 Skill 的产出），优先复用，避免重复侦察。

---

### 阶段 1：UI 元素 → 代码入口映射
**目标**：把"用户操作的 UI 元素"映射到"渲染它的源码文件 + 事件处理器"。这是整个流水线最关键的起点，**入口找错，全盘皆错**。

按**可靠性从高到低**依次尝试以下定位策略，命中即停：

| 优先级 | 策略 | 手法 | 适用 |
|--------|------|------|------|
| 1 | **文案搜索** | 用 UI 上可见文字（按钮文案、标题、提示）全局精确搜索 | 通用，最直接 |
| 2 | **路由反查页面** | 由 URL/路由定位页面组件，再在页面内找元素 | 文件路由 / 配置路由 |
| 3 | **元素标识** | 搜索 `id` / `class` / `data-*` / `name` / `test-id` | 文案为动态/图标时 |
| 4 | **资源名** | 搜索图标文件名、图片 alt、icon class | 纯图标按钮 |
| 5 | **结构推断** | 由相邻元素/页面区块结构反推组件 | 文案为通用词（如"提交"） |

**关键产出**：
- 渲染该 UI 元素的**组件文件**。
- 绑定在该元素上的**事件处理器**（`onClick` / `@click` / `onSubmit` / `addEventListener` / 框架指令）。

**常见陷阱**：
- 文案可能来自**国际化 i18n 文件**（搜不到原文）→ 先到 `locales/` / `i18n/` 找 key，再用 key 反查使用处。
- 元素可能在**通用组件 / 高阶组件 / 循环渲染**中 → 沿 props/slot 向上追溯到具体调用方。
- 文案为通用词（"确定"/"提交"）会命中过多 → 叠加路由或相邻文案缩小范围。

---

### 阶段 2：入口 → 业务逻辑追踪
**目标**：从事件处理器出发，沿着**控制流与数据流**追踪，看操作到底触发了什么。

**追踪要点**：
1. 进入事件处理器函数，逐行阅读：
   - 是否有**早期 return / 条件不满足**导致"什么都没做"（解释"无响应"）。
   - 是否调用了 API（`fetch` / `axios` / `request` / `useQuery`）、跳转（`router.push`）、弹窗（`Modal` / `message`）、状态更新（`setState` / `dispatch`）。
2. 追踪**状态变更**：`state` / `context` / `store` 的读写是否正确。
3. 追踪**副作用**：API 请求的 URL、参数、method、请求体如何构造。
4. 识别**异常处理**：是否被 `try/catch` 吞掉、`.catch` 是否静默、是否 `console.error` 了但无 UI 反馈（典型"无响应"根因之一）。

**产出**：一条"前端侧因果链"——从点击到发起请求/改变状态的完整路径，标注每一步 `文件:行号`。

---

### 阶段 3：报错信息溯源 / 全链路串联
**目标**：让"报错"与"代码"对上号。根据现象类别（第二章 2.2）分流处理。

#### 3.1 前端 JS 异常（控制台红色报错）
- 读**调用堆栈**，从栈顶往下，逐帧定位到**项目代码**（跳过 `node_modules` / 第三方库帧）。
- 找到抛错的那一行，回看上下文：是空值访问？类型错误？越界？
- 堆栈是定位的**最强证据**，务必完整利用。

#### 3.2 框架警告 / 校验提示
- React/Vue/Angular 的开发警告（如 key 缺失、effect 依赖、表单校验）→ 按提示文案定位校验逻辑或表单 schema。
- 业务校验弹窗（"手机号格式不正确"）→ 搜索校验文案，定位 `rules` / 校验函数。

#### 3.3 网络请求错误（按状态码分流）
| 状态码 | 性质 | 排查方向 |
|--------|------|----------|
| **4xx** | 客户端构造问题 | 请求 URL/路径、参数、请求体字段名、鉴权头（token/cookie）、权限不足（403） |
| **5xx** | 后端处理异常 | 进入后端日志与堆栈，定位 Controller → Service |
| **超时 / pending** | 阻塞型 | 后端慢查询、死锁、外部依赖（第三方 API/邮件）卡住、未 await 的长任务 |
| **CORS / 网络错误** | 环境/配置 | 后端跨域配置、代理配置、CSP、证书 |

> **关键**：4xx 不要急着甩锅前端，也可能是后端接口签名变了前端没同步；5xx 不要只看前端，必须深入后端。

#### 3.4 后端日志 / 堆栈（反向定位）
- 拿到后端堆栈 → 跳过框架帧，定位到**业务代码**帧（Service / Controller / Mapper）。
- 顺着 Controller → Service → Repository/DAO → SQL 逐层检查：
  - 参数接收是否正确（参数名、类型、必填）。
  - 业务逻辑是否抛了被捕获后转成错误码的异常。
  - SQL 是否报错（字段不存在、约束冲突、空指针）。
  - 事务/并发是否导致数据不一致。

#### 3.5 全链路因果串联（前端 ↔ 后端）
最终要能画出一条**贯穿前后端的完整因果链**：

```
点击按钮(前端) → onClick handler → 构造请求(URL/参数) → HTTP 请求
   → 后端路由 → Controller → Service → Repository → DB
                                                          │
前端渲染 ← 处理响应 ← HTTP 响应 ←（异常在此抛出并向上冒泡）───┘
```

报错点可能出现在任意环节。**根因 = 这条链上"最早偏离预期"的点**。

---

### 阶段 4：根因确认
**目标**：从"怀疑点"升级为"已确认的根因"，避免把中间症状当根因。

**确认标准（必须同时满足）**：
1. **可解释全部现象**：该根因能逻辑自洽地推出用户看到的全部症状（无响应 + 报错 + 数据异常等）。
2. **代码确实如此**：在源码中找到具体 `文件:行号`，证明代码逻辑确实会走到该错误路径。
3. **无更深层原因**：再往上追一层，确认上层逻辑本身是合理的（否则当前点只是症状）。

**验证手法（只读）**：
- 沿数据流代入**具体触发数据**（用用户反馈中的那条数据）做"心智执行"，看是否复现错误。
- 检查**边界与分支**：空值、0、空数组、特殊字符、并发、首次进入等是否被覆盖。
- 若条件允许，建议用户在**本地按特定条件复现**一次以验证假设（而非改代码）。

**不满足时的处理**：
- 若假设只能解释部分现象 → 标注"部分确认"，列出无法解释的点，回到阶段 2/3 继续。
- 若多个可疑点并存 → 列出假设清单，按"可解释的现象数 × 证据强度"排序，逐一验证排除（见第十章）。

## 四、按现象分类的排查矩阵（Pattern Library）

针对第二章的现象分类，给出**根因候选清单 + 对应排查手法**。这是经验沉淀的"速查表"，遇到对应现象优先按此排查。

### A. 点击按钮"没响应"
**根因候选**（按概率）：
1. **事件未绑定 / 绑定到错误元素**：`onClick` 缺失、绑定在父级但被遮挡。
2. **元素被遮挡或不可点击**：`z-index` / 绝对定位覆盖 / 透明遮罩层 / `pointer-events:none`。
3. **按钮处于 disabled 状态**但样式未体现（loading 态未恢复、条件置灰）。
4. **handler 内早期 return / 条件不满足**：某个 `if` 拦截了逻辑。
5. **异步请求卡在 pending**：请求未返回，后续逻辑（如刷新、跳转）未触发；或 loading 态阻塞了二次操作。
6. **异常被 try/catch 静默吞掉**：`.catch(()=>{})` 空实现、`catch` 后无任何反馈。
7. **防抖/节流**：连续点击只生效一次，或冷却期内无反应。
8. **事件冒泡 / 默认行为**冲突：被 `stopPropagation` / `preventDefault` 在错误层级拦截。

**排查手法**：① 确认事件绑定 → ② 进 handler 入口打点/看是否执行 → ③ 看 Network 是否有请求 → ④ 看 Console 是否有被吞的异常 → ⑤ 检查拦截条件分支。

### B. 弹窗/提示报错
**分流**：
- **前端业务校验弹窗**（提交前即弹）→ 搜索文案，定位 `rules` / 校验函数 / 表单 schema。
- **后端错误码弹窗**（请求返回后弹）→ 拿错误码到前端全局响应拦截器找映射，再到后端找抛该码的位置。
- **框架/库原生报错**（带库名）→ 识别来源库，查其触发条件，定位调用方式是否误用。
- **系统/网络级**（"网络异常""服务开小差"）→ 看 Network 状态码与响应体，按 3.3 处理。

### C. 数据异常（不显示 / 显示错误 / 部分丢失）
**根因候选**：
1. 接口字段名与渲染字段不匹配（后端 `userName`，前端取 `username`）。
2. 渲染条件错误（`v-if` / `&&` 短路 / 权限判断）。
3. 数据格式转换错误（时间戳、枚举、嵌套对象解构）。
4. 分页 / 过滤 / 排序逻辑错误。
5. 接口成功但 `data` 为空（后端查询条件问题）→ 跨到后端排查。
6. **前端缓存 / 旧状态**：列表未重新拉取、store 未清空。

### D. 崩溃 / 白屏 / 跳错误页
**根因候选**：
1. 渲染期异常（访问 `undefined` 的属性）→ 看控制台堆栈，常发生在 `.map` / 解构 / 访问深层字段。
2. 路由配置错误 / 守卫抛错 → 跳到 404 或兜底页。
3. 资源加载失败（chunk 404、CDN 失效、版本不一致）→ Network 看 JS/CSS 加载。
4. 内存溢出 / 死循环（少见但致命）。

### E. 偶发 / 间歇性问题（最难）
**根因候选**：
1. **竞态条件**：多个异步请求乱序返回、快速连点、`useEffect` 依赖不当。
2. **缓存 / 脏状态**：首次与二次行为不同。
3. **时序依赖**：依赖某个尚未就绪的数据 / DOM。
4. **后端并发 / 事务**：同时操作导致数据不一致或锁。
5. **外部依赖不稳定**：第三方服务、网络抖动。
**排查手法**：要求用户记录**精确复现步骤与上下文**，对比"成功 vs 失败"两次的差异，定位变量。

## 五、高频根因清单（Root Cause Checklist）

当顺着调用链一时找不到明确错误时，用此清单**逐项核对**，命中率极高。把它当作"根因的候选项池"。

| 类别 | 典型根因 | 典型现象 |
|------|----------|----------|
| 空值 | `null/undefined` 未防御、可选链缺失、后端返回缺字段 | 白屏、`Cannot read property of undefined` |
| 异步 | 忘记 `await`、Promise 链断裂、竞态 | 数据时有时无、顺序错乱 |
| 类型/字段 | 字段名大小写、类型不匹配、序列化差异 | 数据显示错误 |
| 权限 | token 失效/未带、角色无权限、越权被拒 | 401/403、操作被拒 |
| 边界 | 0 / 空集合 / 超长 / 特殊字符 / 负数 | 特定数据才报错 |
| 缓存 | 旧数据、store 未清、HTTP 缓存、CDN 旧版本 | 改了没生效、偶发 |
| 配置 | 环境变量缺失、API 前缀错误、域名/端口 | 整块功能不可用 |
| 并发/事务 | 并发写、未加锁、事务回滚 | 数据不一致 |
| 事件 | 冒泡未阻、默认行为未阻、重复绑定 | 行为异常、触发多次 |
| 跨域/安全 | CORS、CSP、CSRF token | 请求失败、资源加载失败 |
| 依赖 | 库版本不兼容、peer 依赖缺失、锁定旧版 | 启动/构建/运行报错 |

---

## 六、定位工具与命令速查（按技术栈，均为只读安全）

> 优先使用 IDE / 大模型的内置搜索能力。以下是常用的只读分析手段，**禁止执行任何写入、删除、网络写操作**。

### 6.1 全局精确搜索（定位入口的核心武器）
- **按文案找组件**：精确匹配按钮/提示文案（注意 i18n key 转译）。
- **按路由找页面**：搜路由 path（如 `/system/user`）、路由 name。
- **按 API 路径串联前后端**：前端搜请求路径（如 `/api/user/delete`）→ 后端搜同一路由定义，**这是打通前后端因果链的最强抓手**。
- **按错误码/错误文案**：前端搜错误码常量、后端搜抛出该码的位置。

### 6.2 运行时信息（需引导用户配合提供）
大模型只能做**静态代码分析 + 因果推理**，运行时信息必须由用户从以下渠道获取并反馈：
- **浏览器 DevTools**：Console（异常堆栈）、Network（请求/响应/状态码）、Elements（DOM 结构、事件监听）、Sources（断点）、Application（localStorage/cookie/token）。
- **后端日志**：操作时间点对应的 ERROR/WARN 与堆栈。
- **数据库**：可疑数据的实际值（只读查询）。

### 6.3 引导用户收集信息的标准话术
- "请按 F12 → Console，重新操作一次，把红色报错完整复制给我。"
- "请切到 Network 标签，重新点击，找到那条变红的请求，告诉我：请求方法、URL、状态码、响应内容。"
- "请提供大约 XX:XX 时间点的前后端日志片段。"

## 七、信息不完备时的降级策略（健壮性核心）

现实反馈几乎总是不完备的。本 Skill 必须能优雅降级，而非卡死或乱猜。

| 缺失项 | 降级策略 |
|--------|----------|
| 无报错信息 | 按"无响应"路径走静态分析（阶段 1→2），列出最可能的 2-3 个根因候选，并明确请用户补充 Console/Network 信息以收敛 |
| 无操作路径 | 从用户能描述的最小元素切入，反向推断所属页面/模块 |
| 无法确定前后端 | 先按"最短路径"假设（前端优先），给出验证点；同时列出"若是后端问题"的判断依据 |
| 多个可疑点并存 | 列假设清单 → 按"可解释现象数 × 证据强度"排序 → 从最高分开始逐一**只读验证/排除** |
| 无法复现 | 聚焦偶发根因（竞态/缓存/并发，第四章 E），要求用户提供成功与失败两次的对比上下文 |

**置信度标注规则**：
- **高**：有堆栈/日志直接指认 + 代码逻辑自洽 + 能解释全部现象。
- **中**：有较强代码证据，但缺一项运行时佐证，或只能解释主要现象。
- **低**：仅基于现象推断，无直接代码/运行时证据——必须显式声明为"待验证假设"并列出验证步骤。

## 八、防误判：常见陷阱（严谨性核心）

定位最怕"自信地指错地方"。以下陷阱必须主动规避：

1. **相关性 ≠ 因果性**：不要因为"刚改了 A 就出问题"就断定是 A；可能是巧合、或 A 改动暴露了既有的 B 问题。
2. **症状当根因**：报错的那一行往往是"受害者"而非"凶手"。例：`Cannot read 'x' of undefined`，报错行是受害者，凶手是上游让该变量变 `undefined` 的逻辑。
3. **"最近改动"偏见**：优先看代码逻辑本身是否正确，而非谁最近碰过。
4. **环境差异盲区**：开发正常 ≠ 生产正常。注意环境变量、数据量、版本、域名、HTTPS 等差异。
5. **只见树木不见森林**：确认一个可疑点后，务必回看是否还有更深层原因（阶段 4 第 3 条）。
6. **过度依赖单一证据**：仅凭一条日志就下结论风险高，尽量用代码逻辑交叉印证。
7. **忽视第三方/配置层**：Nginx、网关、拦截器、中间件、CDN 层的问题常被误判为应用代码问题。
8. **修复≠诊断**：本 Skill 的产出是**诊断**，不要在定位未完成时就动手改代码（合规约束）。

---

## 九、安全与合规约束（Read-Only）

- **只读**：定位全程**不得修改、删除、新增**项目文件；不得执行写库、写文件、发外部请求的命令。
- **最小授权**：只访问定位所必需的代码与配置；不主动读取与问题无关的敏感数据（密钥、个人隐私数据）。
- **职责分离**：诊断报告中的"修复建议"明确标注为**建议**，是否实施由用户决定并另起流程。
- **不越权**：不重启服务、不改配置、不触碰生产环境数据；如需运行时信息，由用户在其受控环境内获取并反馈。
- **可追溯**：所有结论附 `文件:行号` 或代码片段证据，便于用户复核。

## 十、诊断报告输出规范

完成定位后，按以下结构输出。**语言与用户保持一致**（用户用中文则中文输出）。

```markdown
## 🔍 Bug 诊断报告

### 1. 根因结论（一句话）
<例：删除用户接口在后端 Service 中未对"用户存在关联订单"做防御，导致外键约束抛 500，前端 catch 静默吞错，表现为点击无反应。>

### 2. 现象与触发条件
- 操作：<页面 + 动作>
- 现象分类：A 无响应 / B 弹错 / C 数据异常 / D 崩溃 / E 偶发
- 触发条件：<必现 / 特定数据>

### 3. 证据链（因果溯源，每环附 文件:行号）
1. 入口：`前端组件:行` 渲染按钮，绑定 `onDelete`
2. handler：`xxx.tsx:行` 构造请求 `DELETE /api/user/:id`
3. 网络：返回 500（用户提供）
4. 后端：`UserController:行` → `UserService:行` 执行删除 → DB 抛外键约束异常
5. 根因点：`UserService:行` 缺少关联数据检查

### 4. 涉及文件清单
- 前端：`xxx.tsx`
- 后端：`UserController.java`、`UserService.java`
- 配置：—

### 5. 根因分类
<空值 / 异步 / 权限 / 边界 / 并发 / …，见第五章>

### 6. 修复建议（仅供参考，未实施）
1. <后端：删除前校验/级联/软删>
2. <前端：catch 中给出明确错误提示，避免静默吞错>

### 7. 置信度
高 / 中 / 低 —— <不确定点与待验证项>
```

## 十一、工作流总览（速记）

```
1. 解析反馈（7字段 + 现象分类 + 完备度评估，缺则追问）
        │
2. 阶段0  技术栈/结构侦察 ──► 建立项目地图
        │
3. 阶段1  UI→代码入口（文案/路由/标识…）──► 组件 + 事件处理器
        │
4. 阶段2  入口→业务逻辑（控制流/数据流/异常处理）──► 前端因果链
        │
5. 阶段3  报错溯源（堆栈/状态码/后端日志）+ 全链路串联
        │
6. 阶段4  根因确认（解释全部现象 + 代码坐实 + 无更深层）──► 不满足则回退
        │
7. 输出诊断报告（根因 + 证据链 + 置信度 + 修复建议）
```

**贯穿全程的红线**：只读、证据驱动、症状≠根因、信息不全先追问、相关性≠因果性。
