# Feature Pattern Porting / 功能思路仿照改造指南

> 本 Skill 用于指导大模型在一个**庞大、相对陌生（或部分陌生）的项目代码库**中，处理一类高频却高风险的需求：**"功能 A 要仿照功能 B 的思路来进行修改/改造"**。以系统架构师视角，**严谨、可追溯、不臆测**地完成"理解两个功能 → 提炼 B 的思路 → 映射到 A → 设计正确改造 → 隔离影响域"的全过程，最终产出一份**逻辑正确、且不会牵连系统中其他功能**的改造方案。
>
> 一句话定位：**功能仿照改造的本质是"跨功能的模式迁移"——从参照功能 B 中抽象出可复用的"怎么做（机制/思路）"，而非照搬"做什么（业务语义）"，再把它正确、最小侵入地嫁接到目标功能 A 上。**

---

## 一、技能定位与适用场景

### 1.1 适用场景
- 用户提出形如："功能 A 仿照功能 B 的思路来改""参照订单退款的逻辑改一下会员注销""导出报表那边是怎么做的，把这个导出也改成那样"。
- 用户**看到了 B 的良好实现**（设计优雅、健壮、符合规范），希望 A 也具备同样的机制/思路，但**不知道改造牵扯什么、有多复杂**。
- 需要在改动前**厘清改造范围、识别风险、保证不破坏其他功能**，而非盲目动手。
- 既要"改对"（改造后的逻辑在 A 的业务语境下成立），更要"改稳"（不引发回归）。

### 1.2 不适用场景（边界）
- **只想理解单个功能怎么实现**（无改造意图）→ 交给 `feature-localization`。
- **修 Bug / 排障**（以现象找根因）→ 交给 `bug-localization`。
- **凭空开发全新功能**（无参照对象）→ 非本 Skill 范畴（属于通用编码）。
- **纯重构 / 性能优化 / 升级框架**（目标是改善而非迁移思路）→ 非本 Skill 范畴。
- 纯概念解释、非本项目代码问题 → 直接回答，无需本 Skill。

> **与 feature-localization 的关键区别**：功能定位是"把一个功能的实现全貌搞清楚"（只读、单功能、理解导向）；本 Skill 是"以 B 的实现为参照，设计并落地对 A 的改造"（跨功能、模式迁移、改造导向）。本 Skill 的前两个阶段会**复用功能定位的方法论**作为子步骤。

### 1.3 目标产物（改造方案）
经过本 Skill 的分析后，应输出一份结构化**改造方案**，至少包含：
1. **一句话改造目标**（A 要学 B 的什么、改成什么样）。
2. **参照功能 B 的实现解析**（B 是怎么做的，附 `文件:行号`）。
3. **目标功能 A 的现状解析**（A 现在是怎么做的，附 `文件:行号`）。
4. **"思路"提炼清单**（从 B 抽象出的可移植要素，并标注"可移植 / 需适配 / 不可移植"）。
5. **改造映射表**（B 的每个思路要素 → 落到 A 的哪个文件/哪一层/怎么改）。
6. **前提缺口与前置工作**（A 缺少 B 思路所依赖的前提时，需先补齐什么）。
7. **影响域分析**（改动波及哪些对外契约、调用方、共享依赖；哪些是安全的、哪些有回归风险）。
8. **隔离与防护策略**（如何确保不牵连其他功能）。
9. **改造后的正确性验证方案**（怎么确认改对了、没改坏）。
10. **置信度（高/中/低）**与不确定点 / 待确认项。

### 1.4 核心原则（心智模型）
- **迁移"机制"，不照搬"业务"**：要搬的是 B 解决问题的**思路/模式/架构手法**，而非 B 的领域逻辑、字段、文案。A 有 A 的业务语义，照抄必然错位。
- **证据驱动，拒绝臆测**：B 的每个思路要素、A 的每处现状，都必须有 `文件:行号` 支撑；无证据的推断必须标注为"假设"。
- **理解与改动职责分离**：先彻底读懂 A 与 B，再设计改造；读懂之前不落笔改码。
- **契约优先，最小侵入**：尽量做"内部实现型"改动，**绝不擅自破坏 A 的对外契约**（API 签名、返回结构、DB schema、事件名）；必须破坏时做涟漪分析与显式声明。
- **不污染共享物**：A、B 共用的工具/组件/基类，宁可新建也不轻易改动——改一处共享物会波及所有使用方。
- **正确性要在 A 的语境下重新验证**：B 的思路搬到 A 后，前提条件可能不成立，必须重新走一遍逻辑，不能"因为 B 没问题就认为 A 也没问题"。
- **全链路视角**：前端 / 后端 / 数据 / 中间件 / 异步任务，思路可能要落到多个环节，不要只改一层。

## 二、第一步：改造需求的结构化解析（Intent Triage）

拿到"功能 A 仿照功能 B"的描述后，**先不要急着搜代码**。先把模糊的改造意图拆成结构化字段，并对齐边界。这一步决定了后续提炼的"思路"方向是否正确——**学错了对象的某个面，整份方案都会跑偏**。

### 2.1 必须澄清的 7 个要素

| 要素 | 说明 | 示例 / 追问话术 |
|------|------|----------------|
| **目标功能 A** | 要被改造的是哪个功能 | "会员注销"指注销流程整体，还是注销时的某个环节？ |
| **参照功能 B** | 要仿照思路的是哪个功能 | "订单退款"指退款的什么——并发控制？状态机？审批流？还是审计日志？ |
| **"思路"的具体所指** | A 要学 B 的**哪个方面** | 是学它的"乐观锁防并发"？还是"操作流水审计"？还是"分步状态机"？ |
| **改造后的预期** | A 改完应该达到什么效果 | "会员注销也带上版本号并发控制和操作日志" |
| **改造边界** | 改哪些层、不改哪些层 | 只改后端逻辑，还是前端交互也要对齐？ |
| **已知线索** | A、B 的文件/类/API 路径 | 任何已知锚点都能大幅加速定位 |
| **约束条件** | 不能动的东西 | "A 的对外接口不能变""不能停机""不能引入新依赖" |

### 2.2 "思路"层面的歧义消解（最关键）
用户说"仿照 B 的思路"，往往**只说了结果没说维度**。必须把"思路"落到具体的工程维度，否则无从下手。向用户提供可选维度让其确认：

> "您说 A 要仿照 B 的思路，具体是指以下哪一类（可多选）：
> ① 架构分层与代码组织方式；② 数据流与调用链路；③ 参数校验方式；④ 异常处理与错误码体系；⑤ 事务与并发一致性控制；⑥ 状态管理/状态机；⑦ 鉴权与权限校验；⑧ 缓存策略；⑨ 异步/事件/消息副作用；⑩ 审计/日志/可观测性；⑪ 返回结构与 API 契约风格；⑫ 扩展点/可配置化设计。"

### 2.3 信息完备度评估与追问清单（健壮性保障）
现实中用户的描述往往**模糊、指代不清**。遇到歧义，**必须先主动追问**：

- A、B 名称有歧义 → 给出 2-3 种可能解释让用户选择（"这里的'注销'是账号注销还是登出？"）。
- "思路"维度不明 → 用 2.2 的维度清单让用户勾选。
- 不知道改前端还是后端 → 请用户明确，或声明"我会先做 A、B 全链路对比，再圈定改动范围"。
- A、B 给了中文名但项目用英文 → 先翻译/猜测英文关键词，结合搜索验证。
- 存在"约束未明" → 主动确认能否改契约、能否加依赖、能否改 DB。

> **原则**：宁可先对齐一句"你要学 B 的什么、改 A 的什么"，也不要在错误的"思路维度"里深入很远。学错维度，提炼出的全是无效内容。

## 三、核心方法论：七阶段模式迁移流水线

面对任何"功能 A 仿照功能 B"的请求，严格按以下七个阶段推进。**每个阶段都有明确的"要做什么、得到什么结论、不满足则回退"**。

```
改造意图 ──► [阶段0 双功能意图对齐]
                         │
        [阶段1 参照功能B 逆向解析] ──► [阶段2 目标功能A 现状解析]
                         │                       │
                         └──────► [阶段3 思路提炼与抽象] ◄──────┘
                                          │
                                  [阶段4 差异分析与映射设计]
                                          │
                                  [阶段5 影响域分析与隔离设计]
                                          │
                                  [阶段6 方案验证与确认] ──► 不满足则回退
                                          │
                                  输出改造方案（→ 实施）
```

### 阶段 0：项目技术栈与结构侦察（Recon）
**目标**：在搜代码前，先建立"项目地图"，确定去哪搜、怎么搜。**若已有同项目的前序分析（project-init-context / project-bootstrap-guide / feature-localization 产出），优先复用，避免重复侦察。**

**侦察动作**：
1. 识别技术栈指纹（`package.json` / `pom.xml` / `go.mod` / `requirements.txt` / `composer.json` / `Cargo.toml` 等）。
2. 判定**前端框架与路由约定**（文件式路由 vs 配置式路由）、**后端框架与路由分发方式**（`@RequestMapping` / `app.get` / `APIRouter` / `@Controller` 等）。
3. 判定**分层架构与目录职责**（controller / service / repository / model / domain / module）。
4. 判定**API 风格、统一返回结构、鉴权方式、事务约定、状态管理方案**。

**结论产出**：一份"项目地图"——后续阶段据此选择 A、B 各自的切入层与搜索策略。

---

### 阶段 1：参照功能 B 逆向解析（Reference Analysis）
**目标**：彻底读懂"被仿照的 B 是怎么实现的"，建立 B 的完整实现地图。**这是提炼"思路"的原料，原料不全则提炼必然失真。**

> 复用 `feature-localization` 的五阶段方法论（技术栈侦察 → 锚定入口 → 核心逻辑追踪 → 架构数据流串联 → 验证确认），对 B 做一次完整的功能解析。

**必须解析清楚（围绕用户选定的"思路维度"，重点深挖对应环节）**：
1. **入口与调用链**：B 的前端入口 / 后端入口 / 触发器，沿 Controller → Service → Repository 逐层下钻，每项附 `文件:行号`。
2. **数据流**：请求/响应字段、涉及的实体/表、读还是写、是否有缓存、是否有异步副作用。
3. **机制细节**（按用户关注的维度）：B 用了什么并发控制？什么校验？什么异常处理？什么事务边界？什么状态流转？什么鉴权？什么审计？

**关键产出**：B 的实现地图（入口 + 调用链 + 数据流 + 机制清单），每项有 `文件:行号`。

**常见陷阱**：
- 只看 B 的"主线"而漏掉 B 之所以健壮的"暗线"（如重试、幂等、补偿、审计）——**要仿照的思路往往藏在暗线里**。
- 把 B 的**业务逻辑**当成"思路"——业务逻辑是领域专属的，不可移植，需在阶段 3 剔除。

---

### 阶段 2：目标功能 A 现状解析（Target Baseline）
**目标**：彻底读懂"A 现在是怎么实现的"，建立 A 的现状基线。**不知道 A 现状，就无法判断"要改什么、改了会不会破"。**

> 同样复用 `feature-localization` 方法论对 A 做完整解析。

**必须解析清楚**：
1. A 的**入口、调用链、数据流**（同阶段 1）。
2. A 当前的**机制现状**：A 现在有没有并发控制？怎么校验？怎么处理异常？有没有事务？有没有审计？——逐项与 B 对照。
3. A 的**对外契约清单**（改造时不可轻易破坏的"红线"）：
   - API 签名（方法、路径、入参、出参字段）。
   - 对外公开的方法 / 函数 / 组件 props。
   - 返回的数据结构 / DTO 形态。
   - DB 表结构 / 字段 / 约束。
   - 对外发出的事件 / 消息名与载荷。
   - 配置项 / 环境变量键名。
4. A 的**调用方/依赖方**（谁会用到 A）：谁调 A 的 API？谁依赖 A 的公开方法？谁订阅 A 的事件？

**关键产出**：A 的现状地图 + **对外契约清单 + 依赖方清单**（后者直接喂给阶段 5 的影响域分析）。

---

### 阶段 3：思路提炼与抽象（Pattern Extraction）—— 本 Skill 的核心
**目标**：从 B 的具体实现中，**抽象出可移植的"思路/机制"**，并明确区分"可移植 / 需适配 / 不可移植"。**这是整个流水线最考验架构素养的一步：抽象层级太高（"写好代码"）等于废话，太低（照抄代码）必然错位。**

#### 3.1 把 B 的实现拆成"机制层"与"语义层"
对 B 解析出的每一项，问：**这是"通用机制"（怎么做的手法）还是"领域语义"（做什么的业务）？**

| 判定 | 含义 | 处理 |
|------|------|------|
| **通用机制（可移植）** | 与业务无关的工程手法：乐观锁、分步状态机、统一异常→错误码、操作流水审计、DTO 校验装饰器、事件发布解耦、重试/幂等、缓存读写策略 | ✅ 提炼为"思路要素" |
| **需适配** | 机制通用，但参数/触发条件是 B 专属：如"退款金额校验"机制可移植，但"金额上限 5000"是 B 的业务规则 | ⚠️ 移植机制骨架，参数换成 A 的领域值 |
| **领域语义（不可移植）** | B 独有的业务逻辑：退款流程、库存扣减规则、积分计算公式、B 的文案/字段名 | ❌ 坚决不搬，A 保留自己的语义 |

> **抽象的检验标准**：一句话能说清的、脱离 B 具体业务仍成立的，才是"思路"；换一个功能它依然适用的，才是好的抽象。

#### 3.2 可移植"思路"维度清单（对照用户在 2.2 勾选的维度）
| 维度 | 典型可移植思路 | 识别线索（在 B 中找） |
|------|----------------|----------------------|
| 架构分层 | controller 只做编排、service 装业务、repository 只碰 DB；依赖接口而非实现 | 分层是否清晰、是否用接口/依赖注入 |
| 数据流 | 请求→校验→业务→持久化→（发事件）→返回 的编排顺序；DTO 与 Entity 分离转换 | 调用链的顺序与转换点 |
| 参数校验 | DTO + `class-validator`/`zod`/`pydantic`/`@Valid`；校验集中在入口 | 校验发生在哪一层、用什么手段 |
| 异常处理 | 自定义业务异常 + 全局过滤器/拦截器统一转错误码；不裸抛、不静默吞错 | `try/catch`、全局过滤器、错误码枚举 |
| 事务/一致性 | `@Transactional`/unitOfWork 包裹写操作；乐观锁（版本号）；悲观锁；幂等键 | 事务边界、版本字段、唯一约束 |
| 状态管理 | 状态机（有限状态 + 合法迁移校验）；前端 store 单向流 | 状态枚举、迁移校验、`status` 字段 |
| 鉴权/权限 | 守卫/拦截器/装饰器（`@PreAuthorize`/`@UseGuard`）；数据权限过滤 | 横切组件、注解、SQL where 注入 |
| 缓存策略 | 读时先查缓存→回源→回填；写后失效/双删；缓存键命名约定 | 缓存读写顺序、失效点、键格式 |
| 异步/事件 | 主流程只做核心写，副作用（通知/日志/统计）走事件/消息异步解耦 | `publish`/`emit`/消息发送点、消费者 |
| 审计/可观测 | 关键操作落操作流水；结构化日志；链路 traceId | 审计表写入、日志埋点、trace 注入 |
| 契约风格 | 统一返回 `{code,msg,data}`；分页结构；错误码分段 | 全局响应包装、响应拦截器 |
| 扩展点 | 策略模式/可配置化/钩子，使新增分支不改主干 | 策略注册表、配置项、钩子函数 |

#### 3.3 提炼产出（思路要素表）
| 思路要素 | 在 B 的出处 | 可移植性 | 一句话描述（脱离 B 业务） |
|----------|------------|----------|--------------------------|
| 乐观锁防并发 | `refund.service.ts:77` `@Version` | 可移植 | 写操作前比对版本号，冲突则拒绝 |
| 操作流水审计 | `refund.service.ts:95` 写 `op_log` | 可移植 | 关键写操作落一条不可变流水 |
| 退款金额上限校验 | `refund.service.ts:60` `<=5000` | 需适配 | 金额类入参做上限校验（上限值换 A 的） |
| 退款业务流程 | `refund.service.ts:40-120` | 不可移植 | — |

### 阶段 4：差异分析与映射设计（Gap & Mapping）
**目标**：把提炼出的每个"思路要素"，**具体映射到 A 要改的地方**，并识别 A 缺少的前提。**这是从"分析"过渡到"设计"的桥梁。**

#### 4.1 逐要素映射（改造映射表）
对阶段 3 每个可移植要素，回答：
1. **A 现状**：A 这方面现在是什么样（有/没有/不一样）？出处 `文件:行号`。
2. **落差（Gap）**：A 相比 B 的思路，缺什么 / 哪里不同？
3. **落地位置**：这个思路要加到/改到 A 的哪个文件、哪一层、哪个方法？
4. **改法（抽象到 A 的语境）**：用 A 的实体/字段/语义重述这个机制（**不是照抄 B 的代码**）。
5. **前提缺口**：B 的思路依赖某些前提（如版本号字段、审计表、事件总线、错误码段），A 是否具备？不具备则列为"前置工作"。

| 思路要素 | A 现状 | 落差 | 落地位置 | 改法（A 语境） | 前提缺口 |
|----------|--------|------|----------|----------------|----------|
| 乐观锁 | `member.service.ts:50` 直接 update，无版本控制 | 缺并发防护 | `member.service.ts:注销方法` | 给 `member` 表加 `version` 字段，update 时带 `WHERE version=?` 并自增 | 需 DB 加字段 + 迁移 |
| 操作流水 | 无 | 缺审计 | 新建 `member_op_log` 记录，注销后写一条 | 复用 `op_log` 表结构，type 用 A 的"注销" | 需建表/复用表 |

#### 4.2 前提缺口汇总与前置工作
- DB 改动：加字段/加表/加约束/加索引（注意是否影响存量数据与迁移）。
- 依赖缺失：需引入事件总线、缓存客户端、校验库等。
- 配置/常量：需新增错误码段、配置项、状态枚举。
- 共享基础设施：若 A、B 共用某个基类/工具，确认是否需要扩展（扩展需评估影响域）。

> **原则**：前置工作本身也要纳入影响域分析——加 DB 字段、改共享基类都可能波及其他功能。

### 阶段 5：影响域分析与隔离设计（Impact & Isolation）—— "不影响其他功能"的保障
**目标**：回答"这次改造会动到哪些对外契约、波及哪些调用方/共享依赖"，并设计**最小侵入、零回归**的隔离策略。**这是本 Skill 区别于普通"功能开发"的核心价值：改造可以激进，但影响必须可控。**

#### 5.1 三类影响源（逐项排查）
| 影响源 | 含义 | 风险等级 | 排查手法 |
|--------|------|----------|----------|
| **对外契约破坏** | 改了 A 的 API 签名 / 返回结构 / DB schema / 事件名 / 公开方法 | 🔴 高（直接回归调用方） | 全局搜索 A 的 API 路径、公开方法名、事件名、表名的所有引用处 |
| **调用方/依赖方回归** | A 的现有调用方依赖了 A 的旧行为，改造后行为变了 | 🟠 中 | 找到所有调用 A 的入口（前端调用、其他 service、定时任务、消息消费者） |
| **共享物污染** | 改了 A、B 共用的工具/组件/基类/常量/类型 | 🔴 高（波及所有使用方） | 改动若是共享物，列出其全部使用方 |

#### 5.2 影响分级与处置策略
| 改动类型 | 是否破坏契约 | 处置 |
|----------|------------|------|
| **纯内部实现型**（只改 service 内部逻辑，契约不变） | 否 | ✅ 最安全，优先采用 |
| **契约兼容扩展型**（新增可选字段、新增接口、返回结构只增不减） | 否（向后兼容） | ✅ 安全，注意调用方是否需感知新字段 |
| **契约破坏型**（改字段名/类型、删字段、改语义、改事件载荷） | 是 | ⚠️ 慎用：必须做涟漪分析 → 通知所有调用方同步改 → 评估能否版本兼容过渡（双写/灰度） |
| **共享物改动型** | 视情况 | ⚠️ 优先"扩展而非修改"（新增重载/新增方法/新增配置项），避免改既有签名 |

#### 5.3 隔离设计原则（确保不牵连其他功能）
1. **契约守恒**：除非用户明确授权，**默认保持 A 的对外契约不变**——把改造藏在内部实现层。
2. **扩展优先于修改**（开闭原则）：需要新行为时新增方法/分支/配置，不改既有方法签名。
3. **不碰共享物的既有签名**：若必须增强共享工具，走"新增重载/新增可选参数带默认值"路线，保证旧调用方零感知。
4. **分支隔离**：A 的新逻辑尽量落在 A 自己的文件/方法内，不侵入通用路径。
5. **副作用收敛**：新增的异步事件/消息/缓存键，使用 A 专属命名空间，不与 B 或其他功能冲突。
6. **配置开关**（可选）：高风险改造可加配置开关/特性开关，便于灰度与回滚。
7. **回归用例**：对每个调用方/共享物使用方，准备"改造前后行为一致"的验证点。

#### 5.4 影响域报告（产出）
| 被改动项 | 改动性质 | 影响的调用方/使用方 | 是否破坏契约 | 隔离策略 |
|----------|----------|---------------------|-------------|----------|
| `member.service.注销()` | 内部实现 | — | 否 | 安全 |
| `member` 表加 `version` | schema 扩展 | 读 member 的查询需注意 | 否（只增列） | 新列设默认值，存量数据兼容 |
| 复用 `op_log` 表 | 共享物（只写不） | 其他写 op_log 的功能 | 否 | 用 A 专属 type 值，不碰既有数据 |

### 阶段 6：方案验证与确认
**目标**：把"设计稿"升级为"经校验的可落地方案"，避免改造逻辑在 A 语境下不成立或遗漏环节。

**确认标准（尽量同时满足）**：
1. **思路已正确映射**：每个可移植要素都有 A 的落地位置与 A 语境下的改法，无悬空。
2. **前提已补齐**：所有前提缺口都有对应的前置工作，无遗漏。
3. **逻辑在 A 语境成立**：B 的思路搬到 A 后，代入 A 的实体/流程做"心智执行"，每一步衔接成立（如乐观锁要求 A 的表有版本字段——已补齐）。
4. **契约可控**：所有对外契约改动已显式列出，调用方影响已评估，隔离策略到位。
5. **无越界**：未把不可移植的 B 业务语义硬塞进 A；未污染共享物既有签名；未破坏系统约定一致性。

**验证手法（仅设计层面，未实施）**：
- 用**一条典型 A 操作**代入改造后的设计，做端到端心智执行，看是否走通、是否引入副作用。
- 检查**边界与分支**：A 的并发、空值、异常路径、状态流转在新思路下是否都被覆盖。
- 交叉验证：从"调用方视角"反向检查——A 的调用方在改造后行为是否依旧符合预期。

**不满足时的处理**：
- 若某思路要素在 A 无法落地（前提缺失且不可补）→ 标注"该要素暂不可移植"，说明原因，请用户决策是否降级或放弃。
- 若改造必然破坏契约且无法隔离 → 显式声明风险，提供"契约兼容过渡方案"（双写/灰度/版本号）供用户选择。

## 四、按"思路维度"的改造策略矩阵（Pattern Library）

针对第二章 2.2 的思路维度，给出**提炼要点 + 移植要点 + 易错点**。这是经验沉淀的"速查表"。

### ① 架构分层思路
- **提炼**：B 是否严格分层（controller 编排、service 业务、repository 数据）。
- **移植**：把 A 改造成同样的分层，逻辑从"挤在 controller"下沉到 service。
- **易错**：下沉时漏掉原本 controller 里的校验/鉴权；破坏 A 既有分层约定。

### ② 数据流思路
- **提炼**：B 的"请求→校验→业务→持久化→事件→返回"编排顺序、DTO/Entity 转换点。
- **移植**：让 A 的数据流与 B 同构，补齐缺失环节（如 A 没有事件解耦，加上）。
- **易错**：照搬 B 的字段名；顺序变了导致事务边界/副作用时序错乱。

### ③ 参数校验思路
- **提炼**：B 的 DTO + 校验装饰器/库，校验集中在入口。
- **移植**：给 A 的入参补同款校验，规则值换 A 的领域约束。
- **易错**：把 B 的业务上限值原样搬来；校验放错层（应在入口而非散落 service）。

### ④ 异常处理思路
- **提炼**：B 的"自定义业务异常 → 全局过滤器统一转错误码"。
- **移植**：A 也走同一套异常体系（复用既有异常类/过滤器，不另造）。
- **易错**：新增错误码段与他人冲突；异常被吞导致"无响应"。

### ⑤ 事务/一致性思路
- **提炼**：B 的事务边界、乐观锁版本号、幂等键。
- **移植**：A 的写操作纳入事务；并发场景加版本号/唯一约束。
- **易错**：A 的表没有版本字段却照搬乐观锁；事务范围过大锁表。

### ⑥ 状态机思路
- **提炼**：B 的状态枚举 + 合法迁移校验（非法迁移直接拒）。
- **移植**：给 A 建状态机，定义 A 自己的合法迁移图。
- **易错**：照搬 B 的状态名；迁移图与 A 业务不符。

### ⑦ 鉴权/权限思路
- **提炼**：B 的守卫/装饰器/数据权限过滤（行级、字段级）。
- **移植**：给 A 套同款守卫，权限点换成 A 的。
- **易错**：权限点硬编码 B 的标识；越权风险。

### ⑧ 缓存策略思路
- **提炼**：B 的"读查缓存→回源→回填"与"写后失效/双删"。
- **移植**：A 采用同款读写顺序与失效策略，键用 A 命名空间。
- **易错**：键名冲突导致串数据；失效遗漏导致脏读。

### ⑨ 异步/事件思路
- **提炼**：B 把副作用（通知/日志/统计）从主流程剥离到事件/消息。
- **移植**：A 的副作用也走事件解耦，事件名用 A 专属。
- **易错**：事件载荷照搬 B；消费失败无补偿。

### ⑩ 审计/可观测思路
- **提炼**：B 的关键写操作落操作流水、结构化日志、traceId。
- **移植**：A 在同等关键点埋审计/日志，type 用 A 的业务类型。
- **易错**：审计写法破坏既有审计表约定；日志含敏感信息。

### ⑪ 契约风格思路
- **提炼**：B 的统一返回结构 `{code,msg,data}`、分页结构、错误码分段。
- **移植**：A 对齐同一套契约风格（应复用项目既有包装器，保持一致）。
- **易错**：为 A 单独造一套返回结构，反而破坏系统一致性。

### ⑫ 扩展点/可配置化思路
- **提炼**：B 用策略模式/配置项/钩子，使新增分支不改主干。
- **移植**：把 A 的硬编码分支重构为可扩展结构。
- **易错**：过度设计；改造范围失控波及共享代码。

## 五、改造工具与分析动作速查（前期为只读安全）

> 前期解析与设计阶段**优先使用内置搜索能力（Glob / Grep / Read）**，**禁止执行任何写入、删除、网络写操作**。只有进入"实施"阶段、且方案经用户确认后，才按方案动码。

### 5.1 定位 A、B 两功能的入口与实现（复用 feature-localization）
- 按 API 路径串联前后端（搜 `/api/xxx`）。
- 按文案 / 路由 / 领域关键词 / 触发器 / 注解决策找入口。
- 沿 Controller → Service → Repository 追踪调用链，每项附 `文件:行号`。

### 5.2 影响域排查（阶段 5 核心武器）
- **搜 A 的对外契约引用**：全局搜 A 的 API 路径、公开方法名、事件名、表名、配置键名 → 列出所有引用处即"潜在受影响方"。
- **识别共享物**：判断要改的文件/方法是否被 A、B 之外的功能引用（搜 import / 调用）。
- **调用方逆向**：从 A 的入口反查"谁会触发 A"（前端调用、其他 service、定时任务、消息消费）。

### 5.3 前提缺口核查
- 查 A 的实体/表是否具备 B 思路所需字段（读 schema/migration/entity）。
- 查 A 是否具备所需基础设施（事件总线/缓存/审计表/错误码段）。

## 六、信息不完备时的降级策略（健壮性核心）

现实请求中，A、B 的信息往往不全。本 Skill 必须能优雅降级，而非卡死或乱猜。

| 缺失项 | 降级策略 |
|--------|----------|
| A、B 含义有歧义 | 给出 2-3 种解释请用户选择，或先按"最常见含义"做概览并标注假设 |
| "思路"维度不明 | 用第二章 2.2 的维度清单请用户勾选，未确认前不深入提炼 |
| 不知 A 还是 B 改谁 | 先确认方向（"是把 A 改得像 B，对吗？"） |
| B 的实现找不到入口 | 换关键词变体 / 从数据实体反查 / 检查是否异步触发型 |
| A 的契约/调用方不清 | 全局搜索 A 的对外标识做穷举；搜不到则标注"未发现外部依赖（待确认）" |
| 前提缺口无法补齐 | 标注该思路要素"暂不可移植"，说明阻塞点，请用户决策 |
| 改造必然破坏契约 | 显式声明风险，提供契约兼容过渡方案供选择，不擅自破坏 |

**置信度标注规则**：
- **高**：A、B 均完整解析 + 思路映射无悬空 + 影响域已穷举 + 逻辑在 A 语境验证通过。
- **中**：主线清晰，但个别前提/调用方缺证据。
- **低**：A 或 B 仅部分解析，思路映射有缺口——必须显式声明为"待验证"并列出待确认项。

## 七、防误判：常见陷阱（严谨性核心）

改造最怕"自信地把 B 的东西生硬塞进 A"或"改了 A 却牵连一片"。以下陷阱必须主动规避：

1. **照搬业务语义**：把 B 的退款流程/字段/文案塞进 A——A 有 A 的业务，照抄必错位。**只搬机制，不搬语义。**
2. **抽象层级失当**：太低（复制代码）→ 领域不符；太高（"要健壮"）→ 无从落地。**以"换一个功能仍成立"为标尺。**
3. **移植过度**：把 B 专属的业务规则（上限、阈值、流程步骤）当成通用思路搬来。
4. **移植不足**：只复制表面代码，漏掉 B 之所以健壮的暗线（重试/幂等/补偿/审计）。
5. **破坏对外契约**：擅自改 A 的 API 签名/返回结构/DB schema，回归所有调用方。
6. **污染共享物**：改了 A、B 共用的工具/基类，波及所有使用方。**优先扩展而非修改。**
7. **忽视前提缺口**：B 的思路依赖版本字段/审计表/事件总线，A 没有，却照搬导致运行即错。
8. **逻辑未在 A 语境重验**：默认"B 没问题，A 也没问题"——A 的并发量/数据特征/状态不同，必须重走逻辑。
9. **副作用命名冲突**：新增事件/缓存键用了通用名，与其他功能串数据。
10. **破坏系统一致性**：为 A 单独造一套异常/返回/分层风格，与项目既有约定背道而驰。
11. **越界扩大改动**：借改造之名顺手重构无关代码，扩大爆炸半径。
12. **解读≠动手**：方案未经确认前不动码；解读职责与改动职责分离。

## 八、安全与合规约束

- **先只读、后实施**：阶段 0-6 全程**不得修改、删除、新增**项目文件；仅产出改造方案。实施须在方案经用户确认后、严格按方案执行。
- **最小侵入**：改动尽量收敛在 A 的内部实现；共享物优先扩展不改既有签名；契约默认守恒。
- **最小授权**：只访问解析所必需的代码与配置；不主动读取无关敏感数据（密钥、个人隐私数据）。
- **不越权**：不重启服务、不改生产配置、不触碰生产数据；如需运行时验证，由用户在其受控环境内执行并反馈。
- **可追溯**：所有结论附 `文件:行号` 或代码片段证据，便于用户复核。
- **风险显式化**：任何契约破坏、共享物改动、前置 DB/依赖变更，必须在方案中显式标注风险等级与缓解措施。
- **诚实标注边界**：当信息不足、改造不可行或必然破坏契约时，明确告知，不编造"看似可行实则有害"的方案。

## 九、改造方案输出规范

完成解析与设计后，按以下结构输出。**语言与用户保持一致**（用户用中文则中文输出）。方案应**清晰、可执行、风险透明**——用户要的是"知道怎么改、改了安全"，而非一份含糊的"建议"。

```markdown

### 1. 改造目标（一句话）
<例：会员注销仿照订单退款的思路，补齐"乐观锁防并发 + 操作流水审计"，且不改变注销接口对外契约。>

### 2. 参照功能 B 的实现解析（被学的对象）
```

## 十、工作流总览（速记）

```
1. 解析意图（7要素 + 思路维度对齐 + 边界/约束确认，歧义则追问）
        │
2. 阶段0  技术栈/结构侦察 ──► 建立项目地图（复用已有分析）
        │
3. 阶段1  参照功能B 逆向解析（入口/调用链/数据流/机制，附行号）──► B 实现地图
        │
4. 阶段2  目标功能A 现状解析（同上 + 对外契约清单 + 依赖方清单）──► A 现状基线
        │
5. 阶段3  思路提炼（拆机制层 vs 语义层，区分可移植/需适配/不可移植）──► 思路要素表
        │
6. 阶段4  差异分析与映射（每要素→A 落点 + 前提缺口）──► 改造映射表 + 前置工作
        │
7. 阶段5  影响域分析与隔离（契约/调用方/共享物三类排查 + 隔离策略）
        │
8. 阶段6  方案验证（映射无悬空 + 前提已补 + A语境逻辑成立 + 契约可控）──► 不满足则回退
        │
9. 输出改造方案（映射 + 前置 + 影响域 + 验证 + 实施步骤 + 置信度）
```

**贯穿全程的红线**：迁移机制不照搬语义、证据驱动不臆测、契约守恒最小侵入、不污染共享物、逻辑在 A 语境重验、信息不全先追问、解读与动手职责分离。
