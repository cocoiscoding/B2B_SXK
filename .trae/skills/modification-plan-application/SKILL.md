# Modification Plan Application / 修改方案应用执行指南

> 本 Skill 用于指导大模型在一个**与项目 A 初始完全相同的项目代码库 B** 中，**忠实、精确、完整、可验证**地应用一份**已由项目 A 验证过的修改方案（Markdown 格式）**，使 B 修改后的结果与 A 修改后的结果完全一致。
>
> 一句话定位：**修改方案应用的本质是"忠实复刻"——把项目 A 上已验证的变更，精确、完整、无偏差地复刻到项目 B，做到"方案即蓝图，B 即 A"，既不少改漏改、也不擅自加戏。**

---

## 一、技能定位与适用场景

### 1.1 适用场景
- 用户手上有**两份初始相同的项目代码 A 和 B**；已在 A 上完成修改，并产出了一份**修改方案（md 格式）**。
- 现在需要把这份修改方案**应用到 B 上**，使 B 与修改后的 A 一致。
- 修改方案可能包含：新增文件、删除文件、修改文件内容、重命名/移动、配置变更、依赖变更等。
- 既需要"改对"（方案里的每个改动点都在 B 上精确落地），更需要"改全"（不遗漏方案中的任何一处改动），还必须"不越界"（不改动方案未涉及的任何代码）。

### 1.2 不适用场景（边界）
- **只有改动意图、没有成型方案**（如"帮我把 B 也改成支持导出 Excel"）→ 这属于需求开发，应先用 `feature-localization` 理解 B、再设计方案，非本 Skill 范畴。
- **跨功能模式迁移**（如"功能 A 仿照功能 B 的思路来改"）→ 交给 `feature-pattern-porting`，它产出的是"设计改造方案"，而非"复刻既有方案"。
- **修 Bug / 排障**（以现象找根因）→ 交给 `bug-localization`。
- **只读理解代码**（不涉及任何改动）→ 交给 `feature-localization`。
- 纯概念解释、非本项目代码问题 → 直接回答，无需本 Skill。

> **与 feature-pattern-porting 的关键区别**：feature-pattern-porting 是"以 B 的实现为参照，**设计并产出**一份对 A 的改造方案"（创造方案，迁移的是"机制思路"而非具体代码）；本 Skill 是"**拿一份现成的修改方案，忠实执行**到 B 上"（执行方案，复刻的是"具体变更"而非抽象思路）。前者重在设计与影响域分析，后者重在精确匹配与零偏差复刻。

### 1.3 目标产物（执行报告）
经过本 Skill 的流程后，应输出一份结构化**执行报告**，至少包含：
1. **方案解析摘要**（方案描述了多少处变更、涉及哪些文件、什么类型的改动）。
2. **原子变更清单**（把方案拆解为最小可执行单元，每项标注类型、目标文件、定位锚点）。
3. **锚点匹配结果**（每个原子变更在 B 中定位到的具体位置，附 `文件:行号`；匹配失败/有歧义的单独列出）。
4. **偏差与决策记录**（方案描述与 B 实际代码的差异，及如何处置）。
5. **执行结果**（已应用的变更清单，每项标注成功/跳过/失败及原因）。
6. **一致性校验结果**（逐项核对方案中的每个改动点是否已在 B 落地；整体是否与 A 一致）。
7. **置信度（高/中/低）**与待确认项 / 未解决的偏差。

### 1.4 核心原则（心智模型）
- **方案即蓝图，忠实第一**：方案怎么写就怎么改。**绝不擅自"优化"、"重构"、"顺手改"**方案未涉及的代码。忠实复刻是唯一目标，自作主张是最大禁忌。
- **不漏不改不错**（三不红线）：不少改（方案每一处改动都必须落地）、不多改（方案未提及的代码一个字符都不动）、不改错（每处改动的"改前→改后"与方案描述精确一致）。
- **先定位后动手**：先在 B 中精确定位每处变更的目标位置，确认锚点匹配无误、偏差已消解后，再执行修改。**禁止"边读方案边改"——极易漏改错改。**
- **原子化拆解**：把方案拆成最小、独立、可逐一核对的"原子变更"，逐项执行、逐项校验，杜绝"整体感觉改完了"的笼统判断。
- **偏差显式化**：当方案描述的代码与 B 实际代码不完全吻合时（即使 B 初始等于 A，也可能因编辑器格式化、隐藏字符、版本漂移产生细微差异），**必须显式标注并决策，绝不默默"自适应"**。
- **执行可回溯、可校验**：每处改动附方案出处、B 中定位、改前改后对照，便于用户逐项复核与最终一致性确认。
- **最小授权、安全变更**：只改方案涉及的文件；不动无关文件；不执行方案未要求的命令（如额外安装依赖、跑迁移）；涉及破坏性操作（删除文件、改 DB、改依赖）必须显式确认。

## 二、第一步：修改方案的结构化解析（Plan Triage）

拿到修改方案（md）后，**先不要急着搜 B 的代码、更不要动手改**。先把方案彻底读懂、拆透。这一步决定了后续执行的准确性——**方案理解错一处，整批改动都会跑偏。**

### 2.1 修改方案的常见形态识别

现实中，md 格式的修改方案**形态多样、粒度不一**。必须先识别方案的"描述形态"，再选择对应的解析策略：

| 形态 | 特征 | 示例 | 可靠性 |
|------|------|------|--------|
| **① 精确 diff / patch 格式** | 含 `@@` hunk 头、`+`/`-` 行号标记 | `@@ -42,5 +42,7 @@` 标准 unified diff | 🟢 最高（机器可应用） |
| **② before/after 代码块对** | 成对给出"原代码"与"新代码" | ` ```before ` / ` ```after ` 两个代码块 | 🟢 高（锚点明确） |
| **③ 带行号/锚点的精确描述** | 给出文件路径 + 行号或函数名 + 改动说明 | "在 `user.service.ts:88` 的 `login()` 内，将 `bcrypt` 改为 `argon2`" | 🟡 中（需在 B 定位确认） |
| **④ 纯自然语言描述** | 只用文字说明改了什么，无代码块 | "给登录接口加了参数校验和限流" | 🔴 低（高度依赖推理，易歧义） |
| **⑤ 文件级新增/删除说明** | 描述整文件新增或删除 | "新增 `src/utils/rateLimit.ts`，内容如下：……" | 🟢 高（整文件操作） |
| **⑥ 混合形态** | 以上多种组合 | 大文件改动用 diff，小改动用文字，新文件用完整代码 | 🟡 按子段落分别处理 |

> **原则**：一份方案里可能**混合多种形态**。必须**逐段落、逐改动点**地识别其形态，不能用统一策略粗暴处理。形态判断错误（如把"示意性代码"当成"精确 before 块"），会导致锚点匹配失败或错误替换。

### 2.2 必须提取的结构化字段

对方案中的**每一处独立改动**，提取以下字段，组装成"原子变更"：

| 字段 | 说明 | 必填 |
|------|------|------|
| **变更编号** | 自增序号（C1, C2, ...），用于追踪与核对 | ✅ |
| **变更类型** | 新增文件 / 删除文件 / 修改内容(替换) / 插入代码 / 删除代码 / 重命名·移动 / 配置变更 / 依赖变更 | ✅ |
| **目标文件** | 方案中指明的文件路径（相对/绝对） | ✅ |
| **定位锚点** | 用于在 B 中精确定位的内容：before 代码块 / 函数名 / 类名 / 行号 / 唯一标识字符串 | 修改类必填 |
| **改前内容** | 方案描述的"原样"代码（before 块 / diff 的 `-` 行） | 修改类必填 |
| **改后内容** | 方案描述的"目标"代码（after 块 / diff 的 `+` 行 / 新文件完整内容） | ✅ |
| **方案出处** | 方案 md 中的段落标题 / 小节编号，便于回溯 | ✅ |
| **备注/约束** | 方案中附带的注意事项、前置条件、特殊说明 | ⚪ 可选 |

> **"原子"的划分标准**：一个原子变更 = 一次不可再分的、独立的修改单元（一处替换 / 一个文件的新增 / 一段代码的插入）。一处改动若涉及同一文件多个不连续位置，拆为多个原子变更。**宁可拆细，不可合并**——拆细才能逐项核对不遗漏。

### 2.3 方案完备度评估与追问清单（健壮性保障）

现实中，修改方案**未必完整、清晰、无歧义**。遇到以下情况，**必须先主动追问或显式标注**，而非基于猜测硬改：

| 问题 | 表现 | 处置 |
|------|------|------|
| **锚点缺失** | 只说"改了登录逻辑"但没给 before 代码也没给行号 | 追问：请提供改动前的原代码片段或精确行号/函数名 |
| **目标文件不明确** | 只说"修改了 service 层"但没指明哪个文件 | 追问：请明确文件路径 |
| **改后内容不完整** | 只说了"加了校验"但没给加了什么校验的代码 | 追问：请提供改动后的完整代码 |
| **存在歧义描述** | "把那个判断改一下"——改哪个判断？怎么改？ | 列出 2-3 种可能理解请用户确认 |
| **方案与 B 结构不符** | 方案引用的文件/路径在 B 中不存在或结构不同 | 显式标注偏差，按第六章降级策略处理 |
| **隐含依赖未说明** | 方案用了新依赖/新工具函数但未声明引入步骤 | 追问：是否需要安装依赖/新增工具文件 |
| **删改不可逆且无确认** | 方案要求删除文件或大段删除代码但无说明 | 执行前显式确认 |

> **原则**：宁可先对齐一处歧义，也不要在错误理解上改出一批错误。方案理解阶段的一个小歧义，到执行阶段会被放大成系统性偏差。

## 三、核心方法论：六阶段忠实执行流水线

面对任何"把修改方案应用到 B"的请求，严格按以下六个阶段推进。**每个阶段都有明确的"要做什么、得到什么结论、不满足则回退"**。

```
修改方案(md) ──► [阶段0 B的技术栈/结构侦察]
                         │
                [阶段1 方案解析与原子变更提取] ──► 原子变更清单
                         │
                [阶段2 B 中锚点定位与匹配] ──► 每个变更的精确位置(文件:行号)
                         │
                [阶段3 偏差检测与对齐] ──► 偏差清单 + 处置决策
                         │
                [阶段4 变更应用执行] ──► 逐项落地(用户确认后)
                         │
                [阶段5 一致性校验] ──► 不满足则回退/补改
                         │
                   输出执行报告
```

### 阶段 0：项目 B 技术栈与结构侦察（Recon）
**目标**：在执行方案前，先建立 B 的"项目地图"，确认 B 的结构、技术栈与方案描述的环境吻合。**若已有 B 的前序分析（project-init-context / project-bootstrap-guide / feature-localization 产出），优先复用。**

**侦察动作**：
1. 识别 B 的技术栈指纹（`package.json` / `pom.xml` / `go.mod` / `requirements.txt` / `composer.json` / `Cargo.toml` 等）。
2. 确认 B 的**目录结构与分层**（与方案中引用的路径对得上吗？）。
3. 确认 B 的**版本控制状态**（是否干净的 working tree？是否有未提交改动？——若 B 已被改过，"初始相同"的前提可能不成立）。
4. 粗扫方案中**涉及的所有文件路径**，确认它们在 B 中存在（新增文件除外）。

**结论产出**：一份"B 项目地图"+ **方案路径核对结果**（方案引用的文件是否都在 B 中找到）。若发现路径不匹配，进入阶段 3 的偏差处理。

**常见陷阱**：
- B 与 A 的**目录结构有细微差异**（如 A 用 `src/` 而 B 用 `app/`）→ 必须以 B 的实际结构为准，方案路径需做映射。
- B **不是真正的"初始相同"**（已被其他人改过）→ 需评估既有改动是否会与方案冲突。

### 阶段 1：修改方案解析与原子变更提取（Parse）—— 本 Skill 的起点
**目标**：把整份方案 md **彻底读懂**，拆解为**结构化的原子变更清单**。**这是后续一切执行的基础——拆解不全或有误，执行必然漏改错改。**

#### 1.1 通读全文，建立全局认知
- 完整阅读方案 md，理解：**改了什么、为什么改、涉及哪些文件、改动总量**。
- 记录方案的**整体改动主线**（如"本次方案为登录模块增加了限流和参数校验"）。
- 标注方案的**形态分布**（哪些段落是 diff、哪些是 before/after、哪些是文字描述——见第二章 2.1）。

#### 1.2 逐段落提取原子变更
对方案中的**每一处改动描述**，按第二章 2.2 的字段表提取一个"原子变更"。注意：
- **不遗漏**：方案的每个代码块、每段改动描述都要检查是否对应一个变更。常见漏改来源：方案正文中的"顺便……""另外……"等容易被忽略的附带说明。
- **不臆造**：方案没说的改动，不自行补充。如方案说"加了校验"但没给校验代码，应标注"改后内容缺失"并追问，而不是自己编一段校验代码。
- **不合并**：同一文件的多处不连续改动，拆为多个原子变更（便于逐项核对）。

#### 1.3 原子变更清单（关键产出）
| 编号 | 类型 | 目标文件 | 定位锚点 | 改前 | 改后 | 方案出处 | 备注 |
|------|------|----------|----------|------|------|----------|------|
| C1 | 修改(替换) | `user.service.ts` | before 块: `const hash = bcrypt.hashSync(...)` | (旧代码) | (新代码) | §2.1 | — |
| C2 | 插入代码 | `user.controller.ts` | 函数 `login()` 开头 | — | (限流中间件) | §2.3 | 新增依赖 |
| C3 | 新增文件 | `src/utils/rateLimit.ts` | — | — | (完整文件内容) | §3 | — |
| C4 | 删除代码 | `auth.module.ts` | `providers: [oldProvider]` | (旧行) | (新行) | §4 | — |

**自检**：原子变更清单提取完毕后，**反向核对**——方案 md 中的每一段改动描述，是否都对应到了清单中的至少一个原子变更？有无遗漏？

---

### 阶段 2：B 中锚点定位与匹配（Locate）—— 忠实执行的核心
**目标**：为每个原子变更，在 B 中**精确定位**到要修改的目标位置（具体到 `文件:行号`）。**锚点定错，整处改动必错。这是"忠实复刻"能否成功的关键关卡。**

#### 2.1 锚点定位策略（按可靠性从高到低）

| 优先级 | 策略 | 手法 | 适用 |
|--------|------|------|------|
| 1 | **精确代码块匹配** | 用方案给出的 before 代码块，在 B 中精确搜索（逐字符匹配） | 形态①②③，**首选** |
| 2 | **函数/类名 + 上下文** | 先定位函数/类，再用 before 片段的上下文缩小到具体行 | 锚点是函数名时 |
| 3 | **唯一标识字符串** | 用代码中的唯一字符串（注释、特殊变量名、字符串字面量）定位 | before 块有唯一特征时 |
| 4 | **行号参考（需校验）** | 方案给出的行号**仅作参考**，必须用代码内容校验该行是否确实匹配 | 形态③，行号可能漂移 |
| 5 | **语义推断 + 用户确认** | 方案只有文字描述时，推断最可能的位置，**标注假设并请用户确认** | 形态④，可靠性最低 |

#### 2.2 匹配结果分类与处置

| 匹配结果 | 含义 | 处置 |
|----------|------|------|
| **✅ 唯一精确匹配** | before 代码块在 B 中有且仅有一处完全匹配 | 直接采用，进入阶段 4 执行 |
| **⚠️ 多处匹配** | before 代码块在 B 中出现多次 | 用上下文（所在函数/类/相邻代码）消歧；无法消歧则列出所有匹配位置请用户指定 |
| **❌ 零匹配（近似存在）** | B 中有"很像但不一样"的代码（空格/格式/版本差异） | 进入阶段 3 偏差处理 |
| **❌ 零匹配（完全不存在）** | before 代码块在 B 中完全找不到 | 可能该改动已在 B 中存在（已改过）/B 与 A 不同源/锚点过时 → 阶段 3 处理 |

**常见陷阱**：
- **格式差异导致匹配失败**：A 和 B 的缩进风格、引号类型、换行符（CRLF vs LF）、末尾空格不同，导致"看似相同"的代码精确匹配失败。**对策**：匹配时对空白字符做归一化（忽略尾部空格、统一换行），但要记录差异。
- **行号漂移**：方案的行号是基于 A 的版本，B 的行号可能因已有改动而偏移。**对策**：行号永远只作参考，以代码内容匹配为准。
- **同名函数多处**：`init()`、`handle()` 等通用方法名可能在 B 中出现多次。**对策**：必须叠加文件路径 + 类名/模块名上下文消歧。
- **IDE 自动格式化**：B 的编辑器可能有保存时自动格式化（Prettier/ESLint fix），导致代码格式与 A 不同。**对策**：以 B 的实际代码格式为准，替换时保持 B 的格式风格。

---

### 阶段 3：偏差检测与对齐（Reconcile）—— 健壮性核心
**目标**：检测方案描述（基于 A）与 B 实际代码之间的**所有偏差**，逐项决策如何处置。**B 即使初始等于 A，也可能因格式化、隐藏字符、版本漂移、局部预改而产生偏差。忽视偏差"强行套用"是漏改错改的主要根源。**

#### 3.1 偏差的常见类型

| 偏差类型 | 表现 | 风险 | 典型原因 |
|----------|------|------|----------|
| **格式偏差** | 代码逻辑相同但空格/缩进/引号/换行不同 | 🟡 匹配失败但逻辑无害 | 编辑器格式化、CRLF/LF 差异 |
| **内容偏差** | B 的代码与方案的 before 块有实质性内容差异 | 🔴 可能改错位置或 B 已被改过 | B 有独立的改动、版本不同步 |
| **路径偏差** | 方案引用的文件路径在 B 中不存在或位置不同 | 🔴 无法定位目标 | 目录结构差异、重命名 |
| **结构偏差** | 方案假设的函数签名/类结构/模块组织在 B 中不同 | 🔴 改动无法直接套用 | B 经历过重构 |
| **已存在偏差** | 方案的"改后内容"在 B 中已经存在 | 🟠 该变更可能已被应用 | B 已部分同步过 A 的改动 |
| **依赖偏差** | 方案用到的依赖/工具在 B 中未安装 | 🟡 运行时才暴露 | B 的 `package.json`/`pom.xml` 未同步 |

#### 3.2 偏差处置决策矩阵

| 偏差情形 | 处置策略 |
|----------|----------|
| **纯格式偏差**（逻辑一致） | 归一化后确认匹配，按 B 的既有格式风格执行替换，记录"格式已适配" |
| **B 已包含改后内容** | 标注"该变更疑似已在 B 中存在"，跳过并记录，请用户确认是否重复应用 |
| **路径/结构偏差可映射** | 找到 B 中的等价位置（如 `src/` ↔ `app/` 映射），记录映射关系后执行 |
| **内容偏差无法自动消解** | **暂停执行该变更**，列出方案 before vs B 实际代码的对照，请用户决策：①以 B 实际为准适配执行；②跳过；③用户提供更新后的方案 |
| **依赖缺失** | 标注为"前置工作"，在执行前提醒用户是否需要同步安装依赖 |

> **核心原则：偏差绝不被"默默吞掉"**。每一处偏差都必须在执行报告中显式记录：偏差是什么、如何决策、是否影响最终一致性。默默自适应 = 埋下不一致的隐患。

---

### 阶段 4：变更应用执行（Apply）
**目标**：在锚点匹配无误、偏差已消解的前提下，**逐项、忠实**地把每个原子变更应用到 B。**执行前必须获得用户确认（尤其是涉及删除文件、改依赖、改配置等破坏性操作）。**

#### 4.1 执行前确认（Gate）
在动手前，向用户呈现：
1. **原子变更清单总览**（共 N 处变更，涉及 M 个文件，类型分布）。
2. **锚点匹配结果**（每处变更在 B 中的定位）。
3. **偏差清单与处置决策**（有哪些偏差、打算怎么处理）。
4. **破坏性操作提示**（删除了哪些文件、改了哪些配置/依赖、需要哪些前置工作）。
5. 请用户确认后执行。

> **红线**：未经用户确认，**不得执行任何写入操作**。尤其是删除文件、覆盖大段代码、修改依赖清单、修改数据库/迁移文件——必须显式确认。

#### 4.2 逐项执行原则
- **一次一个原子变更**：按清单顺序逐项执行，每项执行后立即校验是否成功。
- **忠实复刻改后内容**：改后的代码**逐字符照搬方案给出的内容**，不自行调整命名、风格、逻辑。若方案给了完整的新代码块，整块替换，不做"智能合并"。
- **保持 B 的文件格式约定**：在"内容忠实"的前提下，文件的缩进风格、换行符遵循 B 的既有约定（避免引入格式噪音）。
- **不改方案外的代码**：替换时只替换方案指定的范围，不波及周边代码。**切忌"顺手"修正 nearby 的问题。**
- **执行顺序的依赖考量**：若变更间有依赖（如 C3 新增的工具函数被 C2 引用），先执行被依赖项；无依赖的可按文件分组批量执行。
- **保留可回溯性**：每处改动记录"方案出处 + B 文件:行号 + 改前摘要 + 改后摘要"。

#### 4.3 各变更类型的执行要点（详见第四章矩阵）
- 新增文件：确保目录存在，写入完整内容，注意文件编码与换行符。
- 删除文件：确认无其他文件引用后再删（若被引用，标注风险）。
- 修改内容：精确替换 before → after，替换后重新读取确认。
- 插入代码：定位插入点（函数体首行/指定行之后），插入方案给出的代码。
- 重命名/移动：更新文件路径，并检查是否需要同步更新 import 引用。

---

### 阶段 5：一致性校验（Verify）—— "B 即 A"的最终保障
**目标**：执行完毕后，**逐项核对**方案中的每个改动点是否都已在 B 中精确落地，并评估整体一致性。**不做校验就交付 = 无法保证 B 与 A 一致。**

#### 5.1 逐项落地核对（基于原子变更清单）
对清单中的**每个原子变更**，回答：
1. **是否已执行？**（已应用 / 跳过 / 失败）
2. **B 中的改后代码是否与方案描述一致？**（读取 B 的目标位置，与方案的 after 内容逐字比对）
3. **如有偏差，是否已记录并合理处置？**

| 编号 | 类型 | 目标文件 | 执行状态 | B 中改后内容核对 | 一致性 |
|------|------|----------|----------|------------------|--------|
| C1 | 替换 | `user.service.ts:88` | ✅ 已应用 | 与方案 after 块一致 | ✅ 一致 |
| C2 | 插入 | `user.controller.ts:42` | ✅ 已应用 | 与方案描述一致 | ✅ 一致 |
| C3 | 新增文件 | `rateLimit.ts` | ✅ 已创建 | 内容与方案一致 | ✅ 一致 |
| C4 | 删除代码 | `auth.module.ts:15` | ⚠️ 跳过 | B 中无匹配 before | ❌ 待确认 |

#### 5.2 整体一致性评估
- **改动完整性**：方案的 N 处变更，是否全部落地？（有跳过/失败的，列出原因）
- **无越界确认**：抽查方案**未涉及**的文件，确认未被改动（防止执行中"手滑"改了无关代码）。
- **引用完整性**：方案新增的文件/函数/依赖，是否被正确引用？方案删除的内容，是否有残留引用导致编译错误？
- **可编译性**（可选）：若条件允许，提示用户在 B 中执行构建/类型检查，确认改动不引入语法/编译错误。

#### 5.3 不满足时的处理
- **某变更未落地** → 分析原因（锚点未匹配/偏差未消解/执行报错），补改或标注"待处理"。
- **改后内容与方案不一致** → 重新执行该变更，确保逐字一致。
- **发现越界改动** → 回退越界部分，恢复 B 原状。
- **残留引用错误** → 补充修复引用（属于方案隐含的必要改动），并标注"为保持一致性所需的连带修正"。

## 四、按变更类型的执行策略矩阵（Pattern Library）

针对第二章的变更类型，给出**执行要点 + 注意事项 + 易错点**。这是经验沉淀的"速查表"。

### ① 新增文件
- **执行**：在 B 的指定路径创建文件，写入方案给出的完整内容。
- **注意**：确认父目录存在（不存在则创建）；文件编码统一 UTF-8；换行符遵循 B 既有约定。
- **易错**：文件路径拼错；遗漏创建中间目录；编码/换行符不一致导致 diff 噪音。

### ② 删除文件
- **执行**：删除 B 中的指定文件。
- **注意**：**删除前必须全局搜索该文件是否被其他文件 import/引用**；若被引用，标注风险并请用户确认（可能需要同步清理引用，但这属于方案隐含改动，需显式声明）。
- **易错**：删了文件但残留 import 导致编译失败；误删路径相似的文件。

### ③ 修改内容（替换 before → after）
- **执行**：在 B 中定位 before 代码块，替换为方案给出的 after 代码块。
- **注意**：替换**必须是精确的**——只替换 before 块覆盖的范围，不多不少。替换后重新 Read 确认。
- **易错**：before 块匹配到了错误位置（多处匹配时选错）；替换范围过大或过小；after 内容"自适应"改写。

### ④ 插入代码
- **执行**：在 B 的指定位置（函数体首行/某行之后/某标记之前）插入方案给出的代码片段。
- **注意**：插入点的精确定位（用锚点而非行号）；插入代码的缩进与上下文保持一致。
- **易错**：插入位置偏移（插到了错误的作用域）；缩进层级不对导致语法错误。

### ⑤ 删除代码
- **执行**：在 B 中定位要删除的代码段，移除之。
- **注意**：删除后检查上下文是否仍合法（如删除了一个变量声明，检查该变量是否还被引用）。
- **易错**：多删或少删（删除范围边界不清）；删除后留下语法残留（如悬空的逗号、括号）。

### ⑥ 重命名 / 移动文件
- **执行**：将 B 中的文件从旧路径移至新路径（或重命名）。
- **注意**：**必须同步更新所有引用该文件路径的 import/require 语句**（这通常是方案隐含的必要改动）。
- **易错**：只移了文件没改 import，导致大面积编译错误；移动后路径大小写敏感问题（Linux vs Windows）。

### ⑦ 配置变更
- **执行**：修改 B 的配置文件（`application.yml` / `.env` / `package.json` / `tsconfig.json` 等）。
- **注意**：配置文件多为结构化格式（JSON/YAML/INI），**尽量精确替换而非整体重写**，避免丢失 B 的既有配置项。注意 JSON 不允许注释/尾逗号。
- **易错**：覆盖了 B 独有的配置项；破坏了 JSON/YAML 语法；环境变量值含特殊字符未转义。

### ⑧ 依赖变更
- **执行**：在 B 的依赖清单（`package.json` / `pom.xml` / `requirements.txt` / `go.mod` 等）中新增/升级/移除依赖。
- **注意**：**版本号必须与方案完全一致**；修改清单后提示用户执行安装命令（`npm install` / `mvn install` / `pip install` 等）；注意 peer dependencies / 锁文件（`package-lock.json` / `yarn.lock`）是否需要同步。
- **易错**：版本号"取整"或用了 latest；只改了清单没提示安装；遗漏锁文件更新导致依赖未真正生效。

## 五、执行工具与操作速查

> 阶段 0-3 为**只读分析**（理解方案 + 定位锚点 + 检测偏差），优先使用内置搜索能力（Glob / Grep / Read）。阶段 4 进入**写入执行**，须在用户确认后按方案动码。阶段 5 为**只读校验**。

### 5.1 只读阶段（阶段 0-3, 5）的工具
- **Glob**：按模式找文件，核对方案引用的路径是否存在（`**/user.service.*`）。
- **Grep**：精确搜索 before 代码块 / 锚点字符串 / 函数名，定位变更位置。
- **Read**：读取 B 的目标文件，确认锚点匹配、检查改后内容、核对一致性。
- **LS**：列目录结构，确认 B 与方案的结构吻合。

### 5.2 写入阶段（阶段 4）的工具
- **文件编辑/创建**：使用 IDE/编辑器的文件编辑能力（SearchReplace / Write），精确执行替换、新增、删除。
- **命令执行**（仅在必要时、用户确认后）：如创建目录（`mkdir`）、移动文件（`mv`）。**禁止执行方案未要求的命令。**

### 5.3 操作安全约定
- **先备份意识**：建议用户在执行前确认 B 已纳入版本控制（Git），便于回退。大范围改动前提示"建议先 commit / 建分支"。
- **逐文件执行**：一个文件改完、Read 确认后，再改下一个，避免批量操作出错时难以定位。
- **禁止危险操作**：不执行 `rm -rf`、不强制覆盖、不改文件权限、不碰 `.git` 目录。

## 六、信息不完备时的降级策略（健壮性核心）

现实中，修改方案与 B 的实际状况往往**不完全吻合**。本 Skill 必须能优雅降级，而非卡死或乱改。

| 缺失/偏差项 | 降级策略 |
|-------------|----------|
| 方案只有文字描述、无代码块（形态④） | 推断最可能的改动位置与内容，**标注为"低置信度推断"**，执行前请用户确认推断是否正确 |
| before 代码块在 B 中零匹配 | ①检查格式差异（归一化后重试）；②搜索关键片段定位近似位置；③标注"锚点未匹配"请用户提供 B 的实际代码 |
| before 代码块多处匹配 | 用文件路径+函数名+上下文消歧；无法消歧则列出所有候选位置请用户指定 |
| 方案引用的文件在 B 中不存在 | ①检查是否路径映射差异（`src/`↔`app/`）；②确认是否为"新增文件"；③标注"路径偏差"请用户确认 |
| B 的代码与方案 before 有内容差异 | 暂停该变更，列出对照（方案 before vs B 实际），请用户决策：适配执行/跳过/更新方案 |
| 改后内容在 B 中已存在 | 标注"疑似已应用"，跳过并记录，请用户确认 |
| 方案不完整（缺 after 内容） | 标注"方案信息缺失"，追问用户补充，不自行编造改后内容 |
| 变更间有隐含依赖未在方案中说明 | 识别依赖关系（如新增函数被引用），将连带必要改动标注为"一致性所需连带修正"，显式说明 |
| 依赖/环境在 B 中缺失 | 标注为"前置工作"，提醒用户在执行前补齐 |

**置信度标注规则**：
- **高**：方案形态为精确代码块/diff + 所有锚点唯一精确匹配 + 无内容偏差 + 全部落地且逐字核对一致。
- **中**：大部分锚点精确匹配，个别变更存在格式偏差（已归一化）或锚点需上下文消歧；或有少量文字描述型变更经用户确认。
- **低**：方案含较多文字描述型变更 / 多处锚点零匹配或歧义 / B 与方案存在内容或结构偏差未完全消解 —— 必须显式声明"待验证"并列出所有不确定点。

## 七、防误判：常见陷阱（严谨性核心）

修改方案应用最怕"自信地改错"、"自作主张地多改"、或"漏改而不自知"。以下陷阱必须主动规避：

1. **擅自"优化"**：看到方案里的代码觉得"可以写得更好"，顺手改了——**坚决禁止**。方案怎么写就怎么改，哪怕方案的风格与 B 既有风格不一致。忠实 > 优雅。
2. **边读边改**：还没完整解析方案、没在 B 中定位锚点，就开始动手改——极易漏改、错改、改串。**必须先完成阶段 1-3，再进入阶段 4。**
3. **锚点选错**：before 代码块在 B 中多处匹配时，没做消歧就改了第一处——可能改错了位置。**必须确认唯一性。**
4. **行号迷信**：盲信方案给出的行号，不校验该行内容是否匹配——行号会漂移。**行号仅供参考，以代码内容匹配为准。**
5. **格式差异误判**：因 CRLF/LF、缩进、引号差异导致匹配失败，就判定"代码不存在"而跳过——**应先归一化格式再判断**。
6. **替换范围失控**：替换 before→after 时，多带或少带了行，导致周边代码被破坏。**精确界定替换边界。**
7. **遗漏附带改动**：方案正文中"顺便……""同时……"的附带说明容易被忽略。**通读全文，逐段提取。**
8. **臆造改后内容**：方案说"加了校验"但没给代码，自己编了一段——**应追问，不臆造**。
9. **连带改动不声明**：移动文件后改了 import、删除函数后清理了引用，但不标注——这些是方案隐含的必要改动，**必须显式声明为"一致性所需连带修正"**。
10. **不做最终校验**：改完就交付，不逐项核对——无法保证 B 与 A 一致。**阶段 5 不可省略。**
11. **越界扩大改动**：借执行方案之名，顺手修了 B 里的其他问题——扩大了不一致风险。**只改方案涉及的代码。**
12. **破坏性操作不确认**：直接删文件、改依赖、改 DB 迁移文件而不告知用户——**必须显式确认**。

## 八、安全与合规约束

- **先只读、后执行、再校验**：阶段 0-3 全程**只读**（不得修改 B 的任何文件）；阶段 4 须在**用户确认后**按方案执行；阶段 5 只读校验。
- **忠实执行，零越界**：只修改方案涉及的文件与代码；方案未提及的代码**一个字符都不动**；不执行方案未要求的命令。
- **破坏性操作显式确认**：删除文件、覆盖大段代码、修改依赖清单、修改配置/DB 迁移文件——执行前必须向用户显式提示并获得确认。
- **最小授权**：只访问方案涉及的文件与必要的配置；不主动读取无关的敏感数据（密钥、隐私数据）。
- **不越权**：不重启服务、不改生产配置、不触碰生产数据；不执行 `rm -rf` 等危险命令；如需运行时验证（构建/测试），由用户在其受控环境内执行并反馈。
- **可追溯**：每处改动附方案出处 + B `文件:行号` + 改前改后对照，便于用户逐项复核。
- **偏差透明化**：所有方案 vs B 的偏差、所有连带修正、所有跳过/失败的变更，必须在执行报告中显式记录，不隐瞒。
- **建议版本控制**：大范围改动前，主动建议用户对 B 做 commit / 建分支，确保可回退。
- **诚实标注边界**：当方案不完整、偏差无法消解、或一致性无法保证时，明确告知，不编造"已完成"的假象。

## 九、执行报告输出规范

## 十、工作流总览（速记）

```
1. 解析方案（形态识别 + 原子变更提取 + 完备度评估，歧义/缺失则追问）
        │
2. 阶段0  B 的技术栈/结构侦察 ──► 项目地图 + 路径核对（复用已有分析）
        │
3. 阶段1  方案解析与原子变更提取 ──► 原子变更清单（逐段提取，不漏不合并）
        │
4. 阶段2  B 中锚点定位与匹配 ──► 每个变更的精确位置(文件:行号) + 匹配结果分类
        │
5. 阶段3  偏差检测与对齐 ──► 偏差清单 + 处置决策（偏差绝不默默吞掉）
        │
6. [用户确认 Gate] 呈现清单/偏差/破坏性操作 ──► 获得确认
        │
7. 阶段4  变更应用执行 ──► 逐项忠实落地（不多改不少改不改错）
        │
8. 阶段5  一致性校验 ──► 逐项核对 + 无越界 + 引用完整 ──► 不满足则回退/补改
        │
9. 输出执行报告（变更清单 + 偏差记录 + 校验结果 + 待确认项 + 置信度）
```

**贯穿全程的红线**：方案即蓝图忠实第一、不漏不改不错、先定位后动手、原子化逐项核对、偏差显式化、执行可回溯可校验、破坏性操作必确认、零越界。

## 附录：使用本 Skill 的检查清单（自检）

每次输出执行报告前，逐项自检，全部为"是"方可交付高置信度结果：

- [ ] 是否**完整通读并正确理解了修改方案**，识别了每个段落的形态，无遗漏？
- [ ] 是否把方案拆解为**原子变更清单**，且反向核对"方案每段描述都对应到了清单"？
- [ ] 是否为每个变更在 B 中**精确定位了锚点**（代码内容匹配，非盲信行号），并确认匹配唯一性？
- [ ] 是否检测了**方案 vs B 的所有偏差**（格式/内容/路径/结构/已存在/依赖），并逐项记录处置决策？
- [ ] 歧义、缺失、无法消解的偏差，是否**已追问用户或显式标注**，而非臆造或默默跳过？
- [ ] 执行前是否**向用户呈现了完整清单并获得确认**（尤其是破坏性操作）？
- [ ] 执行时是否**逐项忠实复刻方案的改后内容**，未擅自优化/重构/加戏？
- [ ] 是否**只改了方案涉及的代码**，方案外的文件/代码零改动（无越界）？
- [ ] 方案隐含的**连带必要改动**（如 import 更新）是否已执行并显式声明？
- [ ] 执行后是否**逐项核对了每个变更的落地情况与逐字一致性**？
- [ ] 是否确认了**无越界改动**（抽查方案未涉及文件未被改动）与**引用完整性**？
- [ ] 执行报告是否**透明记录了所有偏差、连带修正、跳过/失败的变更**，未隐瞒？
- [ ] 是否标注了**置信度**与待确认项，未编造"B 已完全等于 A"的假象？
