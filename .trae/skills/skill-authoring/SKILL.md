# Skill Authoring / SKILL 编写与迭代维护指南

> 本 Skill 用于指导大模型以"技术文档架构师 + 方法论沉淀者"的双重视角，**编写**专业、正确、合规、严谨、健壮的 SKILL 文档（可复用能力的规范化封装），并在使用既有 SKILL 之后**回顾过程、收集经验、迭代补充**，使 SKILL 在其生命周期中持续自我完善、越用越准。
>
> 一句话定位：**SKILL 编写与维护的本质是"把可复用的专家经验固化成机器与人都能照着执行的规范文档"——写的时候要像资深工程师设计 SOP（标准作业程序），维护的时候要像给生产系统打补丁，每一次实战都要反向喂养规范，让它越用越稳、越用越准。**

---

## 一、技能定位与适用场景

### 1.1 适用场景（两类使命，缺一不可）

**使命 A —— 编写新 SKILL（从零到交付）**
- 用户要求"写一份 SKILL 文档""把某套流程封装成 Skill""创建一个 xxx 的 Skill"。
- 某项可复用能力（审查、生成、排障、部署、文档撰写等）需要被固化为规范，供后续反复调用。
- 用户给出一段需求描述，需要将其转化为符合本仓库规范的 `SKILL.md`。

**使命 B —— 迭代维护既有 SKILL（经验回灌）**
- 刚用完某个既有 SKILL 完成任务，需要回顾本次过程，把踩过的坑、遇到的问题、总结的经验补充进该 SKILL，以便下次更方便、快捷、准确地使用。
- 用户要求"复盘这次 Skill 使用""把经验沉淀到 Skill 里""更新 / 完善 / 补充 xxx Skill"。
- 发现既有 SKILL 在某场景下不够健壮（漏判、误判、降级失效、边界未声明），需要精准打补丁。

> **两类使命同属一个 SKILL 的完整生命周期**：编写是"建仓"，回灌是"补仓"。本仓库要求：**每使用完一个既有 SKILL，都应执行一次经验回灌**（若无新发现则不动，避免注水）。

### 1.2 不适用场景（边界）
- **调用某个既有 SKILL 去执行具体任务**（如做代码审查、生成修改方案、排障）→ 直接调用对应 SKILL，本 Skill 不替代其执行，只负责"写它"或"维护它"。
- **一次性、不可复用的任务**（无需固化为规范）→ 直接执行即可，不必封装成 SKILL。
- **纯概念解释 / 知识问答** → 直接回答。
- **修改业务项目代码** → 本 Skill 的产物是 SKILL 文档（规范），不是业务代码。

### 1.3 目标产物
- **使命 A**：一份符合本仓库规范的 `<skill-name>/SKILL.md`（含 frontmatter + 结构化正文）。
- **使命 B**：对既有 `SKILL.md` 的精准增补（新增 / 修订章节内容），交付时必须说明"补了什么经验、补在哪个文件的哪一节、为什么补"。

### 1.4 核心原则（心智模型）
- **规范优先**：产物必须符合本仓库既有 SKILL 的结构与风格（目录 / `SKILL.md`、frontmatter、章节骨架），与既有体系**一致而非另起炉灶**。
- **经验驱动，零臆造**：写方法论要基于真实工程经验与行业最佳实践；回灌经验要基于本次**真实发生的事实**，不可凭空编造"踩坑"。
- **可执行 > 可阅读**：SKILL 是给机器 / 人照着做的 SOP，每一条指令都要**可判定、可执行**，而非空话套话（"写高质量代码"不可判定；"新增 API 前必须先写 DTO 校验"可判定）。
- **健壮性内置**：必须设计"信息不完备时的降级""防误判""边界声明"，让 SKILL 在异常场景下也不崩溃、不乱猜。
- **最小侵入迭代**：回灌经验时**只增不改其义**，不破坏既有结构与其他场景的正确性。
- **证据可追溯**：回灌的每条经验都能追溯到"本次哪个具体问题触发了它"；编写的每条方法论都能找到最佳实践出处。

## 二、SKILL 文档的规范结构（地基）

> 任何 SKILL 必须先满足本节的硬性规范，这是"正确、合规"的底线。**编写前必读，维护时不可违反。**

### 2.1 文件与目录结构
- 每个 SKILL 是一个独立目录：`<skill-name>/SKILL.md`。
- 目录名 = skill name，全小写、kebab-case、语义化（如 `code-review`、`bug-localization`、`modification-plan-generation`）。
- 核心文件固定为 `SKILL.md`；如有配套资源（模板、配置示例），可放在同目录下作为辅助文件。
- 仓库根目录的 `targets.yml` 等全局配置不属于单个 SKILL，不在此范畴。

### 2.2 frontmatter 规范
`SKILL.md` 顶部必须有 `---` 包裹的 YAML frontmatter，且**只有两个必填字段**：

| 字段 | 必填 | 作用 | 约束 |
|------|------|------|------|
| `name` | 是 | 唯一标识，与目录名一致 | kebab-case，全小写，无空格 |
| `description` | 是 | **决定 SKILL 何时被触发的命脉**，见 2.3 | 含 what + when，含触发关键词 |

```yaml
---
name: "<skill-name>"
description: "<What it does>. Invoke when <trigger conditions / keywords>."
---
```

### 2.3 description 编写规范（最关键字段）
`description` 是模型判断"现在该不该调用这个 SKILL"的**唯一线索**。写不好 = 永远不被触发，或被错误触发。**这是 SKILL 能否被用起来的命脉，务必精雕。**

**必须包含两要素**：
1. **What（做什么）**：用一句话说清这个 SKILL 产出什么、解决什么问题。
2. **When（何时触发）**：明确列出触发场景与关键词，**包括用户可能说的口语化表达**（中英文均可），提升命中率。

**编写准则**：
- 以动词开头。
- 把高频触发词、同义表达、口语化说法写进去（如"整理 / 总结 / 复盘""review / CR / 过一遍"），让模型在用户随口一说时也能命中。
- 点明边界（"Invoke when X, not when Y"）以防误触发——可选但推荐。
- 语言：**默认英文**（与本仓库既有 SKILL 的 description majority 一致）；可在 when 部分内嵌中文触发词。

**好坏对照**：

| ❌ 差 | ✅ 好 |
|-------|-------|
| Reviews code.（只说 what，不说 when，不会被触发） | Systematically review code against multi-dimensional standards. Invoke when the user asks to perform a code review / CR / 过一遍代码 before delivery, merge, or release. |
| 帮我整理文档。（口语、无 what/when） | Author professional SKILL documents. Invoke when the user asks to write / create a SKILL (写一份 Skill / 封装), or to append lessons learned after using one. |

### 2.4 正文章节标准骨架
正文是 SKILL 的主体，应遵循与既有 SKILL 一致的结构。**推荐骨架**（章节可按需增删，但顺序与命名风格须保持一致）：

1. 标题：`# Skill Name / 中文名`（中英双语）
2. 开头 `>` 定位块：一句话说清本质 + 价值
3. **一、技能定位与适用场景**（含适用 / 不适用边界 / 目标产物 / 核心原则）
4. **二、规范结构 / 地基**（如适用）
5. **三、核心方法论**（阶段化、可执行的流水线，配文字版流程图）
6. **四、维度矩阵 / 知识库**（如适用，速查表 / 决策树形式）
7. **五、常见陷阱与防误判**
8. **六、健壮性与降级策略**（信息不完备时怎么办）
9. **七、执行约束**（只读 / 只写、合规红线）
10. **八、质量校验清单**（交付前自检）
11. **九、标准输出模板**（产物长什么样）

> **关键**：方法论必须"阶段化"——把任务拆成"阶段 0/1/2..."，每阶段有"目标 → 动作 → 产出 → 不满足则回退"，而非散文式叙述。参考 `code-review`、`project-init-context` 的写法。

## 三、核心方法论·上篇：编写一份高质量 SKILL

```
编写请求 ──► [阶段0 需求解析与定位] ──► [阶段1 调研与对齐]
                                              │
[阶段4 校验交付] ◄── [阶段3 内容编写] ◄── [阶段2 结构设计]
```

### 阶段 0：需求解析与技能定位
**目标**：搞清"这个 SKILL 到底解决什么问题、何时该被调用"——**定位错了，后面全错**。这是最容易被跳过、却最致命的一步。

**必须澄清的要素**（模糊则主动追问，给候选选项）：

| 要素 | 说明 | 追问话术 |
|------|------|---------|
| 能力边界 | 这个 SKILL 做什么、不做什么 | "除了 X，是否还包括 Y？" |
| 触发场景 | 什么情况下用户会用它 | "你会在什么时候想调用它？会怎么说？" |
| 输入 | 它需要什么前置信息 | "用之前需要准备什么？" |
| 产物 | 它输出什么、长什么样 | "用完之后你希望得到什么？" |
| 与既有 SKILL 的关系 | 是否重叠 | "这和现有的 xxx Skill 有什么不同？" |

**健壮性**：用户需求往往模糊。**宁可先追问对齐一句，也不要在错误定位上写完一整份文档再返工。** 若用户表述不清，主动给出 2-3 个候选定位让其选择。

### 阶段 1：调研与对齐（对齐既有体系）
**目标**：确保新 SKILL 与仓库既有体系一致，且方法论站得住脚。

**动作**：
1. **读既有 SKILL**：`LS` 仓库 SKILL 目录，挑 2-3 个最接近的（同类 / 同复杂度），`Read` 其 `SKILL.md`，吸收结构与风格——**这是"合规"的核心动作**。
2. **检查命名冲突**：新 `name` 不得与既有 SKILL 重名或高度歧义。
3. **核查关系**：是否与既有 SKILL 重叠？重叠则在 `description` / 边界章节**显式划清界限**（参考 `code-review` 与 `bug-localization` 的"关键区别"写法）。
4. **方法论溯源**：所写方法应基于行业最佳实践 / 工程共识，必要时可 `WebSearch` 核证，**不可凭空发明流程**。

### 阶段 2：结构设计（搭骨架）
**目标**：先定章节骨架与每节"要回答什么问题"，再填内容。

**动作**：
1. 套用 2.4 的标准骨架，按 SKILL 性质裁剪（生成类、审查类、排障类、流程编排类各有侧重）。
2. 每个章节用一句话写下"本节要回答的问题"。
3. 设计**阶段化方法论**：把主流程拆成阶段，画一张文字版流程图。
4. 设计**产物模板**：最终交付物长什么样（参考既有 SKILL 的"输出规范 / 输出模板"章节）。

### 阶段 3：内容编写（填血肉）
**目标**：把骨架填成可执行、严谨、健壮的规范。

**写作准则**：
- **指令体而非说明体**：用"必须 / 应 / 禁止"，而非"可以 / 一般来说"。
- **表格化知识**：决策点、指纹、分级标准用表格，便于速查。
- **给正反例**：关键准则配 ❌ / ✅ 对照。
- **内置健壮性**：每个方法论阶段补"信息不完备时怎么办""常见误判"。
- **清单化交付**：必读文件、执行动作、自检项用 checklist。

### 阶段 4：质量校验与交付
**目标**：交付前对照第八节清单逐项自检，未通过必返工。

**动作**：
1. 跑第八节"编写新 SKILL 专用"质量校验清单。
2. 创建目录 `<skill-name>/SKILL.md` 并写入。
3. 向用户交付 `computer://` 链接 + 一句话说明新增了什么。

## 四、高质量 SKILL 的设计准则

### 4.1 优秀 SKILL 的七个特征
1. **触发精准**：`description` 让模型在正确时机命中、错误时机不误触。
2. **边界清晰**：明确写"不做什么"，与相邻 SKILL 划界。
3. **阶段可执行**：方法论是流水线，每步有目标与产出，而非散文。
4. **知识表格化**：经验沉淀成速查表 / 决策树，而非长段落。
5. **健壮可降级**：信息不全时有明确的兜底策略，不卡死不乱猜。
6. **防误判内置**：主动列出"最易翻车的点"与规避法。
7. **产物有模板**：交付物有标准格式，保证每次产出一致。

### 4.2 方法论的组织原则
- **由表及里、依赖前置**：阶段顺序符合认知 / 执行的自然依赖（先侦察后深挖，先收集后判断）。
- **横向扫描 + 纵向精读结合**：既有全局模式扫描，也有逐项深挖。
- **去伪存真环节**：每个"发现 / 结论"都有验证 / 交叉验证步骤，防误报。
- **定级 / 定档客观**：用表格给标准，按实际危害而非主观偏好。

### 4.3 健壮性设计三件套（每个 SKILL 都必须内置）
1. **降级策略表**：缺 X 时怎么办（参考 `code-review` 第六节"信息不完备时的降级策略"）。
2. **常见陷阱清单**：最易翻车的 N 个点（参考各 SKILL 的"常见陷阱"节）。
3. **置信度标注规则**：高 / 中 / 低，何时标注"待确认 / 待人工核实"。

## 五、核心方法论·下篇：使用后经验回灌（迭代维护）

> SKILL 不是写完即死的文档，而是越用越准的"活规范"。**本仓库要求：每使用完一个既有 SKILL，都应执行一次经验回灌。** 这是让 SKILL 持续进化的唯一机制。

```
任务完成 ──► [阶段5 回顾触发判断] ──无新发现──► 不动（避免注水）
                          │有新发现
                          ▼
[阶段9 交付说明] ◄── [阶段8 精准补充] ◄── [阶段7 经验提炼与定级] ◄── [阶段6 经验收集]
```

### 阶段 5：回顾触发（何时回灌）
**触发条件**（满足任一即应回灌）：
- 本次使用中**实际踩了坑 / 绕了弯路**（SKILL 没覆盖到的场景）。
- SKILL 的某条指令**导致误判 / 漏判 / 执行偏离**。
- 发现了**更优的做法 / 更准的判断标准**。
- 遇到了 SKILL **未声明的边界**（信息不全时不知如何降级）。

**不回灌的情况**：本次使用完全顺利、无新发现 → **不必强行增补**，避免文档注水膨胀。

### 阶段 6：经验收集（收集什么、从哪收集）
**收集来源**（按可靠性从高到低）：
1. **本次对话的实际行动**（最可靠）：模型实际执行了什么、在哪一步出了问题、怎么修正的。
2. **报错 / 异常 / 与预期不符的输出**：客观事实，非主观感受。
3. **用户的纠正 / 反馈**：用户指出的偏差。
4. **最终的修正动作**：为了修正问题实际做了什么调整。

**收集的内容维度**：

| 维度 | 要回答的问题 |
|------|-------------|
| 场景缺口 | SKILL 没覆盖到的输入 / 场景是什么？ |
| 指令歧义 | 哪条指令被理解偏了？怎么偏的？ |
| 误判 / 漏判 | 哪一步误报或漏了？根因是什么？ |
| 降级失效 | 信息不全时，既有的降级策略够用吗？ |
| 更优解法 | 有没有比 SKILL 现有写法更好的做法？ |

### 阶段 7：经验提炼与定级
**提炼原则**：从"个例"提炼成"可复用规则"。
- ❌ "这次在第 3 步报错了" → 这是个例，换个项目就不适用，不可回灌。
- ✅ "当输入含 X 时，应先做 Y 校验，否则会在第 3 步失败" → 这是规则，可复用，应回灌。

**经验定级表**（决定补到哪个章节）：

| 经验类型 | 应补充到 | 补充形式 |
|----------|---------|---------|
| 新场景 / 新输入未覆盖 | 方法论对应阶段 或 维度矩阵 | 新增一行 / 一种情况 |
| 指令歧义 / 表述不清 | 原指令处 | 修订表述，加正反例 |
| 误判 / 漏判根因 | "常见陷阱 / 防误判"章节 | 新增一条陷阱 |
| 信息不全不知咋办 | "健壮性 / 降级策略" | 新增一行降级策略 |
| 更优判断标准 | 定级 / 分级表 或 知识库 | 修订标准 |
| 产物格式缺陷 | "输出模板" | 修订模板字段 |

### 阶段 8：精准补充（补到哪里、怎么补）
**补充准则**：
1. **先 Read 再改**：必须先 `Read` 既有 `SKILL.md` 全文，确认上下文与既有表述不冲突，再用**精确的局部编辑**增补——绝不凭记忆盲改。
2. **对号入座**：按阶段 7 的定级表，补到对应章节，**不堆到一个"杂项 / 其他"里**（否则不可检索）。
3. **最小侵入**：只改与本次经验相关的最小单元，**不顺手重构不相关内容**。
4. **保持风格一致**：新增内容的语气、表格风格、用词与原文档一致。
5. **不破坏既有正确性**：新增不能与既有指令矛盾；若矛盾，需**同步修订既有指令并说明**，不可只加不改导致自相矛盾。

### 阶段 9：交付与一致性维护
**动作**：
1. 增补后，跑一遍第八节"迭代维护专用"自检清单。
2. 向用户说明：**补了什么经验、补在哪个文件的哪一节、为什么补**。
3. 交付 `computer://` 链接。

## 六、常见陷阱与防误判

> 编写与维护 SKILL 最易翻车的点，主动规避。

### 编写类陷阱
1. **description 只写 what 不写 when** → SKILL 永远不被触发。**必须**写清触发条件与关键词。
2. **散文式方法论** → 不可执行。**必须**阶段化 + 每步有产出。
3. **照搬模板不复用** → 与既有体系割裂。**必须**先读 2-3 个既有 SKILL 对齐风格。
4. **没有边界声明** → 与相邻 SKILL 重叠误触。**必须**写"不适用场景"。
5. **指令全是空话**（"写高质量代码""注意性能"）→ 不可判定。**必须**给可判定的具体规则。
6. **没有降级策略** → 信息一不全就卡死或乱猜。**必须**内置健壮性三件套。
7. **知识不表格化** → 长段落难速查。**应**用表格 / 决策树。

### 维护类陷阱
8. **把个例当规则回灌** → 注水。**必须**提炼成可复用规则再补。
9. **回灌前不 Read 全文** → 与既有内容冲突 / 重复。**必须**先读后改。
10. **顺手大重构** → 破坏既有正确性。**坚持**最小侵入。
11. **经验堆进"杂项"** → 不可检索。**按类型**对号入座到对应章节。
12. **新增与既有矛盾却不同步修订** → 自相矛盾。**必须**同步修订并说明。
13. **无新发现强行增补** → 文档膨胀注水。**无新经验则不动**。

## 七、健壮性与降级策略

| 缺失 / 异常情况 | 降级策略 |
|----------------|----------|
| 用户需求模糊、定位不清 | 主动追问 + 给 2-3 个候选定位让其选择，不基于猜测硬写 |
| 不确定是否与既有 SKILL 重叠 | 先 `LS` + `Read` 既有 SKILL 比对，重叠则显式划界或合并 |
| 方法论无把握是否为最佳实践 | `WebSearch` 核证，无法核证的标"待验证"，不臆造流程 |
| 回灌时记不清本次具体问题 | 以本次对话实际行动 / 报错为客观证据，无证据则不回灌 |
| 不确定经验该补到哪一节 | 按阶段 7 定级表对号入座；实在无对应章节，再考虑新增小节并说明 |
| 新增内容与既有指令矛盾 | 同步修订既有指令，保留双方依据，不制造自相矛盾 |
| 既有 SKILL.md 结构陈旧、不合规 | 仅在用户明确要求"重构"时才大改；否则最小补丁 + 标注待重构 |

**置信度标注规则**（适用于回灌的经验条目）：
- **高**：本次有明确的报错 / 用户纠正等客观证据，且提炼的规则普适。
- **中**：有证据但普适性需更多场景验证，标注"建议在更多场景复核"。
- **低**：仅凭单次主观感受，一般**不回灌**；若必须记录，显式标注"疑似，待验证"。

## 八、执行约束（给大模型的硬性要求）

1. **先对齐后动笔**：编写前先解析需求、读既有 SKILL 对齐风格；维护前先 `Read` 全文。
2. **遵循既有体系**：结构、命名、风格与仓库既有 SKILL 一致，**不另起炉灶**。
3. **证据驱动**：回灌经验必须有本次事实支撑；编写方法论基于最佳实践，可核证。
4. **最小侵入**：维护时**只增不改其义**，不动无关内容。
5. **交付自检**：交付前跑第九节清单，未通过必返工。
6. **显式说明**：交付时说明"新增 / 修改了什么"，**不静默改动**。
7. **诚实标注边界**：能力 / 信息不足时明确告知，不臆造。
8. **语言一致**：文档语言与用户及仓库 majority 一致（本仓库以"中英双语标题 + 中文正文 + 英文 description"为主）。
