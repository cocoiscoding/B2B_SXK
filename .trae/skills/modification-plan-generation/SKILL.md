# Modification Plan Generation / 修改方案文档生成指南

> 本 Skill 用于指导大模型在一个**已经经过多轮对话、已被模型做了多次修改的项目**中，将所有已发生的修改**完整、准确、可追溯**地整理为一份**结构化的修改方案文档（Markdown 格式）**，使该文档能清晰回答"一共改了哪些步骤、每一步在哪里改的、每一步改了什么"。
>
> 一句话定位：**修改方案生成的本质是"忠实回溯"——从已发生的、散落在多轮对话中的修改事实中，逆向重建一份完整、精确、可被另一个模型或人类忠实复现的变更蓝图。**

---

## 一、技能定位与适用场景

### 1.1 适用场景
- 用户与模型在某个项目上**已经过多轮对话**，模型在对话过程中**对项目代码做了多次修改**（新增、删除、替换、移动文件、改配置、改依赖等）。
- 现在用户需要模型**回过头来**，把所有已经发生的修改**整理成一份完整的修改方案文档（md 格式）**。
- 文档必须能回答三个核心问题：**①一共改了哪些步骤？②每一步在哪个文件的哪里改的？③每一步具体改了什么（改前→改后）？**
- 生成的方案文档用途可能包括：交接给他人审阅、应用到另一个相同项目（交由 `modification-plan-application` 执行）、作为变更存档、用于 code review 等。

### 1.2 不适用场景（边界）
- **还未发生的修改**（如"帮我规划接下来要怎么改"）→ 属于设计/规划类任务，非本 Skill 范畴。
- **只需要理解某个功能怎么实现的**（不涉及变更整理）→ 交给 `feature-localization`。
- **只需要应用一份已有方案**（方案已存在、需要执行）→ 交给 `modification-plan-application`。
- **修 Bug / 排障**（以现象找根因）→ 交给 `bug-localization`。
- **功能仿照改造**（以 B 的思路改 A）→ 交给 `feature-pattern-porting`。
- 纯概念解释 → 直接回答，无需本 Skill。

> **与 modification-plan-application 的关键区别**：`modification-plan-application` 是"方案 → 代码"（拿现成方案执行到代码库）；本 Skill 是"代码 → 方案"（从已发生的修改逆向生成方案文档）。两者互为逆过程——**本 Skill 生成的方案文档，其格式与粒度应能被 `modification-plan-application` 直接消费与忠实复现**。

### 1.3 目标产物（修改方案文档）
经过本 Skill 的流程后，应输出一份结构化**修改方案文档（Markdown 格式）**，至少包含：
1. **修改概述**（一句话总结本次修改的主题与目的）。
2. **修改总览表**（共 N 处变更、涉及 M 个文件、类型分布）。
3. **分步变更清单**（按步骤逐一列出每一处修改，每步含编号、类型、目标文件、精确位置、改前内容、改后内容、修改说明）。
4. **前置依赖与连带变更**（修改所需的前置工作及连带必要改动）。
5. **完整性自检结果**（确认方案覆盖了所有已发生的修改，无遗漏）。
6. **置信度（高/中/低）**与待确认项 / 信息不确定点。

### 1.4 核心原则（心智模型）
- **忠实回溯，拒绝臆造**：方案中记录的每一处修改，都必须是**确实已经发生的**真实变更。不可凭推测"补全"未确认的修改内容，不可美化或重写已改的代码。**有据则录，无据则标注"待确认"，绝不编造。**
- **完整性第一（不遗漏）**：多轮对话中散落的每一次修改都必须被收集到，哪怕是小改动、附带改动、"顺便改的"。遗漏一处 = 方案不可信。
- **精确定位（不模糊）**：每处修改必须标注**精确的文件路径与位置锚点**（行号/函数名/before 代码块），使任何人或模型都能据此找到改动点。
- **改前改后对照清晰**：每处修改必须给出明确的"改前 → 改后"对照，而非笼统的文字描述。**对照越精确，方案越可复现。**
- **以最终状态为准（净变更原则）**：若同一位置被多次修改（改了又改），方案中记录的是**从最初到最终的净变更**，中间的反复过程不必逐次记录（除非用户明确要求完整时序）。
- **多源交叉验证**：修改信息来源于多个渠道（版本控制、对话历史、当前代码），**优先采用可靠性最高的来源**，多源之间交叉验证不一致时显式标注。
- **只读整理，不改代码**：本 Skill 的执行过程是**只读**的——只收集、整理、记录，**不修改项目的任何代码**（修改已经在之前完成了，本阶段只负责整理成文）。

## 二、第一步：修改信息的结构化收集（Input Collection）

拿到"整理修改方案"的请求后，**先不要急着动笔写方案**。先系统性地收集所有已发生修改的信息。**信息收集不全或失真，整份方案都会残缺或错误。**

### 2.1 修改信息的四种来源（按可靠性从高到低）

现实中，已发生的修改信息分散在多个渠道。**必须优先使用可靠性高的来源，低可靠来源的内容必须经高可靠来源交叉验证后方可采用。**

| 优先级 | 来源 | 可靠性 | 获取方式 | 说明 |
|--------|------|--------|----------|------|
| **①** | **版本控制（Git 等）** | 🟢 最高 | `git diff`、`git log`、`git status` | 若项目纳入了版本控制且修改前后有提交记录，这是**最权威、最精确**的变更来源——逐行级别的变更一目了然 |
| **②** | **当前代码 vs 初始/基线代码** | 🟢 高 | 对比当前文件与修改前的备份/初始版本 | 若有修改前的代码快照（初始版本、备份分支、首次对话前的状态），直接 diff 可得到精确变更 |
| **③** | **对话历史中的修改记录** | 🟡 中 | 回溯本轮会话中的历次文件编辑操作（工具调用记录） | 模型在对话中执行过的每次 Write/SearchReplace 等操作都是修改证据；但受上下文窗口限制，**早期修改可能已被截断遗忘** |
| **④** | **模型回忆 / 推断** | 🔴 低 | 模型基于记忆自行回忆改了什么 | **最不可靠**，存在记忆偏差、遗漏、臆造风险；只能作为线索，**必须用 ①②③ 交叉验证后才能采用** |

> **信息收集策略**：依次尝试 ① → ② → ③ → ④，**优先用高可靠来源补全低可靠来源的空白**。若 ① 可用，则以 ① 为主、③ 为辅（版本控制提供精确 diff，对话历史提供修改意图说明）；若 ①② 均不可用，则只能依赖 ③④，置信度必须降级。

### 2.2 必须提取的结构化字段

对收集到的**每一处独立修改**，提取以下字段，组装成"变更记录"：

| 字段 | 说明 | 必填 |
|------|------|------|
| **变更编号** | 自增序号（C1, C2, ...），用于追踪与核对 | ✅ |
| **变更类型** | 新增文件 / 删除文件 / 修改内容(替换) / 插入代码 / 删除代码 / 重命名·移动 / 配置变更 / 依赖变更 | ✅ |
| **目标文件** | 被修改的文件路径（相对路径，与项目根目录的关系明确） | ✅ |
| **精确位置** | 修改发生在文件中的具体位置：行号 / 函数名 / 类名 / before 代码块锚点 | 修改类必填 |
| **改前内容** | 修改发生前的原始代码（before 块） | 修改类必填 |
| **改后内容** | 修改发生后的目标代码（after 块 / 新文件完整内容） | ✅ |
| **修改说明** | 一句话说明此处修改的目的与作用 | ✅ |
| **修改来源** | 该记录的依据来源（Git diff / 对话历史第N轮 / 当前代码推断） | ✅ |
| **备注** | 特殊说明（如"此处经历了多次反复修改""附带产生的格式化变更"） | ⚪ 可选 |

> **"一处独立修改"的划分标准**：一次不可再分的、独立的修改单元（一处替换 / 一个文件的新增 / 一段代码的插入）。同一文件的多处不连续改动，拆为多个变更记录。**宁可拆细，不可合并**——拆细才能逐项核对不遗漏。

### 2.3 信息完备度评估与追问清单（健壮性保障）

现实中，修改信息**未必能完整收集**。遇到以下情况，**必须主动追问或显式标注**，而非基于残缺信息硬编方案：

| 问题 | 表现 | 处置 |
|------|------|------|
| **无版本控制** | 项目未纳入 Git，无法获取精确 diff | 追问：是否有修改前的代码备份？或请用户提供首次对话前的项目快照 |
| **对话历史截断** | 早期对话已被上下文窗口截断，最早的修改无记录可查 | 追问：是否还记得最初改了什么？或以当前代码与用户印象中的原始状态对比推断 |
| **修改记录不完整** | 对话历史中部分修改操作的工具调用记录缺失 | 标注"该修改记录不完整"，以当前代码的实际状态为准，标注为"推断" |
| **无法区分"模型改的"与"用户手改的"** | 项目中混入了非模型操作的修改 | 追问：请确认哪些修改是模型做的，哪些是您手动改的；或按用户指示只整理模型做的修改 |
| **修改已被外部撤销** | 对话历史中有修改记录，但当前代码中该修改不存在 | 标注"该修改疑似已被撤销"，请用户确认是否需要记录 |
| **不确定某处改动是否真实发生** | 模型回忆有某修改，但当前代码中找不到痕迹 | 以当前代码的实际状态为最终事实，标注"回忆中的修改未在当前代码中体现" |

> **原则**：宁可对齐一处不确定，也不要在残缺信息上编出一份"看似完整实则失真"的方案。修改方案的价值在于**可信**——一处编造，全篇存疑。

## 三、核心方法论：六阶段修改方案生成流水线

面对任何"整理修改方案"的请求，严格按以下六个阶段推进。**每个阶段都有明确的"要做什么、得到什么结论、不满足则回退"**。

```
已发生的多次修改 ──► [阶段0 项目技术栈与结构侦察]
                              │
                    [阶段1 修改信息全面收集] ──► 原始变更记录池
                              │
                    [阶段2 分类·去重·时序整理] ──► 结构化变更清单
                              │
                    [阶段3 修改方案文档生成] ──► 方案文档初稿
                              │
                    [阶段4 完备性校验] ──► 不满足则回退补收
                              │
                    [阶段5 一致性确认与交付] ──► 最终方案文档
```

### 阶段 0：项目技术栈与结构侦察（Recon）
**目标**：在整理方案前，先建立项目的"项目地图"，确认项目的结构、技术栈与涉及的文件范围。**若已有同项目的前序分析（project-init-context / project-bootstrap-guide / feature-localization 产出），优先复用，避免重复侦察。**

**侦察动作**：
1. 识别项目的技术栈指纹（`package.json` / `pom.xml` / `go.mod` / `requirements.txt` / `composer.json` / `Cargo.toml` 等）。
2. 确认项目的**目录结构与分层**（controller / service / repository / model / module / pages 等）。
3. 确认项目的**版本控制状态**：
   - 是否纳入 Git？是否有提交记录？
   - 当前 working tree 是否有未提交的改动（这些很可能就是模型做的修改）？
   - 是否有修改前的 commit / branch / tag 可用作基线对比？
4. 粗扫项目结构，了解**哪些目录/文件可能被修改过**（按文件修改时间排序、按 Git 状态筛选）。

**结论产出**：一份"项目地图"+ **修改范围初判**（哪些文件疑似被改过，从哪些来源可以获取精确变更）。

**常见陷阱**：
- 项目**未纳入版本控制** → 丢失了最可靠的信息来源（①），必须降级到来源 ②③④，并提示用户置信度会降低。
- 项目有版本控制，但**修改全部堆在一个未提交的 working tree 里** → 用 `git diff`（未暂存）和 `git diff --cached`（已暂存）均可获取精确变更。
- 项目有版本控制，且修改已**散落在多个 commit 中** → 用 `git log` + `git diff <base>..HEAD` 获取累计变更。

---

### 阶段 1：修改信息全面收集（Collect）—— 本 Skill 的基础
**目标**：运用第二章 2.1 的四种来源策略，**穷尽性地**收集所有已发生的修改信息，汇聚成"原始变更记录池"。**这是后续一切整理的基础——收集不全，方案必有遗漏。**

#### 1.1 按来源优先级依次收集

**来源 ①：版本控制（若可用）**
```
git status                    # 查看哪些文件被改了（新增/修改/删除）
git diff                      # 未暂存的逐行变更
git diff --cached             # 已暂存的逐行变更
git diff <基线commit>..HEAD   # 从基线到当前的累计变更
git log --oneline --stat      # 各次提交涉及的文件与行数
```
- 版本控制 diff 提供的是**逐行级别的精确变更**（`+`/`-` 行），这是构建 before/after 对照的最佳原料。
- 注意区分：**内容变更**（代码逻辑改了）vs **格式变更**（仅空格/缩进/换行变化）。格式变更若非用户有意为之，可在方案中合并或备注。

**来源 ②：当前代码 vs 基线代码（若 ① 不可用但有基线快照）**
- 若用户能提供修改前的代码快照（如初始版本压缩包、备份分支），用文件级 diff 工具逐文件对比。
- 重点关注**文件级变化**（新增了哪些文件、删除了哪些文件）和**内容级变化**（每个文件内部哪些行变了）。

**来源 ③：对话历史中的修改操作记录**
- 回溯当前会话中模型执行过的每次**文件写入操作**（Write / SearchReplace / Edit / Delete 等工具调用）。
- 每次操作都记录了：**改了哪个文件、改前是什么、改后是什么**——这是构建变更记录的直接素材。
- **注意上下文窗口限制**：若对话轮次很多，早期操作记录可能已被截断。此时需降级到来源 ④ 或来源 ①② 补充。

**来源 ④：模型回忆 / 推断（最后手段）**
- 在 ①②③ 均无法覆盖全部修改时，模型可基于自身记忆回忆"我大概改了什么"。
- **此类信息必须显式标注为"推断"，并尽量用当前代码的实际状态交叉验证**——若回忆与实际代码矛盾，以实际代码为准。

#### 1.2 收集过程中的关键注意事项

- **不遗漏小改动**：配置项的一行修改、一个 import 语句的增删、一个常量的重命名——这些"不起眼"的改动同样必须记录。
- **不遗漏附带改动**：对话中"顺便……""另外……"提到的附带修改（如移动文件后更新了 import、删除函数后清理了引用），必须逐项收集。
- **记录修改来源**：每条变更记录都必须标注其信息来源（Git diff / 对话第N轮 / 当前代码推断），便于后续校验时回溯。
- **收集改前内容**：对于修改类变更，**改前内容（before）至关重要**——它是方案的"锚点"，没有 before 就无法精确定位改动位置。改前内容优先从 Git diff 的 `-` 行、对话历史中的 old_str 获取。

#### 1.3 原始变更记录池（关键产出）
将收集到的所有修改汇总为一张原始记录表：

| 编号 | 来源 | 类型 | 目标文件 | 改前摘要 | 改后摘要 | 可靠性 |
|------|------|------|----------|----------|----------|--------|
| R1 | Git diff | 替换 | `user.service.ts:88` | `bcrypt.hashSync(...)` | `argon2.hash(...)` | 🟢 高 |
| R2 | 对话第3轮 | 插入 | `user.controller.ts:42` | — | 限流中间件调用 | 🟡 中 |
| R3 | Git diff | 新增文件 | `src/utils/rateLimit.ts` | — | 完整新文件 | 🟢 高 |
| R4 | 模型推断 | 删除 | `auth.module.ts:15` | `providers:[oldProvider]` | `providers:[]` | 🔴 低 |

**自检**：收集完毕后，**反向核对**——已知的修改文件是否都有对应的变更记录？对话历史中提到的每次修改是否都被收集到了？

### 阶段 2：分类·去重·时序整理（Organize）
**目标**：对原始变更记录池做**去重、合并、排序**，消除冗余与矛盾，形成干净、有序的"结构化变更清单"。**原始池中可能存在重复记录（同一处修改被多个来源各自记录）和时序冲突（同一位置改了又改），必须在此阶段消解。**

#### 2.1 去重与合并

| 情形 | 处理策略 |
|------|----------|
| **同一处修改被多个来源记录** | 以可靠性最高的来源为准（Git diff > 基线对比 > 对话历史 > 推断），合并为一条记录，保留最精确的 before/after |
| **同一位置在多轮对话中被反复修改** | 按"净变更原则"——只记录**从最初状态到最终状态**的变更，中间的反复过程合并为一条（除非用户要求完整时序记录） |
| **一条修改操作被拆分记录** | 若同一文件的连续改动在逻辑上属于同一修改目的，可合并为一条记录（但需确保 before/after 对照仍精确） |
| **格式化变更与内容变更混合** | 将纯格式变更（缩进/空格/换行）与内容变更分离；若格式变更是 IDE 自动产生而非有意修改，标注为"附带格式化变更" |

#### 2.2 变更排序

整理后的变更清单按以下优先级排序，使方案文档逻辑清晰、便于阅读与执行：

1. **按变更类型分组**：前置工作（依赖安装、DB 迁移）→ 新增文件 → 修改文件（按文件路径字母序）→ 删除文件 → 配置/依赖变更。
2. **同文件内按位置排序**：同一文件的多处改动，按代码中的**从上到下顺序**（行号从小到大）排列。
3. **尊重依赖关系**：若变更 A 是变更 B 的前提（如 A 新建的工具函数被 B 引用），A 排在 B 前面。

#### 2.3 净变更识别（"改了又改"的处理）

现实中，模型在多轮对话中可能对同一位置反复修改：
```
第2轮：把 bcrypt 改为 argon2
第5轮：又把 argon2 的调用方式调整了
第8轮：又加了盐值参数
```

**处理原则**：
- 方案中记录的是**第1轮的 before → 第8轮的 after**（净变更）。
- 中间的中间态（第2轮的 after、第5轮的 after）**不必逐次记录**——用户要的是"最终改了什么"，不是"修改的折腾过程"。
- 若用户明确要求"记录完整的修改时序"，则按时间线逐次列出每次修改。

#### 2.4 结构化变更清单（关键产出）

| 编号 | 类型 | 目标文件 | 精确位置 | 改前(before) | 改后(after) | 修改说明 | 来源可靠性 |
|------|------|----------|----------|-------------|------------|----------|-----------|
| C1 | 替换 | `user.service.ts` | L88, 函数 `hashPassword()` | `return bcrypt.hashSync(pwd, 10)` | `return argon2.hash(pwd, {salt: 32})` | 密码哈希算法升级 | 🟢 Git diff |
| C2 | 插入 | `user.controller.ts` | L42, 函数 `login()` 内 | — | `rateLimit.check(ctx.ip)` | 登录接口增加限流 | 🟡 对话记录 |
| C3 | 新增文件 | `src/utils/rateLimit.ts` | 整文件 | — | (完整文件内容) | 限流工具模块 | 🟢 Git diff |
| C4 | 删除代码 | `auth.module.ts` | L15 | `providers: [oldProvider]` | `providers: []` | 移除废弃 Provider | 🟢 Git diff |

### 阶段 3：修改方案文档生成（Generate）
**目标**：将结构化变更清单转化为一份**格式规范、内容完整、可读性强**的修改方案文档（Markdown 格式）。**文档的质量决定了它能否被另一个模型或人类正确理解与忠实复现。**

#### 3.1 文档结构规范

方案文档按以下结构组织（详细模板见第九章）：

1. **标题与概述**：方案名称、修改主题一句话概述、生成时间。
2. **修改总览**：变更总数、涉及文件数、类型分布统计表。
3. **涉及文件清单**：列出所有被修改的文件及其修改类型。
4. **分步变更详情**：逐一列出每个变更（C1, C2, ...），每步包含：
   - 变更编号与类型
   - 目标文件与精确位置（`文件:行号` 或函数名/锚点）
   - 改前代码块（before，用代码块标注）
   - 改后代码块（after，用代码块标注）
   - 修改说明（一句话目的）
5. **前置依赖与连带变更**（若有）：列出方案执行前需完成的前置工作，以及为保证一致性所需的连带改动。
6. **备注与已知不确定项**：标注信息来源可靠性、待确认项。

#### 3.2 before/after 对照的书写规范

before/after 对照是方案文档的**核心载体**，其精确度直接决定方案的可复现性：

- **修改类（替换）**：给出完整的 before 代码块和 after 代码块，两者精确对应（before 是要被找到并替换的原代码，after 是替换后的新代码）。
- **插入类**：before 标注插入位置（如"函数 `login()` 的第一行之后"），after 给出插入的代码块。
- **删除类**：before 给出要删除的代码块，after 标注为"（删除）"或展示删除后该位置的上下文。
- **新增文件**：无 before，after 给出完整文件内容。
- **删除文件**：before 标注文件路径，after 标注"（文件删除）"。

> **关键约定**：before 代码块必须是**足以在代码库中唯一定位**的代码片段（至少 3-5 行，包含足够的上下文），而非孤零零的一行——否则另一个模型在执行方案时可能无法精确匹配。

#### 3.3 修改说明的书写原则

每处修改的"修改说明"应回答**为什么改**（目的），而非仅仅重复**改了什么**（那是 before/after 已经展示的）：

- ✅ 好："将密码哈希从 bcrypt 升级为 argon2，提升安全性并符合 OWASP 最新建议"
- ❌ 差："把 bcrypt 改成了 argon2"（只是重复了代码变更，未说明目的）

---

### 阶段 4：完备性校验（Verify）—— "无遗漏"的保障
**目标**：方案文档生成后，**逐项核对**是否覆盖了所有已发生的修改，并验证每处记录的精确性。**不做校验就交付 = 无法保证方案的完整性。**

#### 4.1 完整性核对（基于原始变更记录池）
对阶段 1 收集的**每条原始记录**，回答：
1. **是否已纳入方案文档？**（已纳入 / 遗漏）
2. **方案中的记录是否与原始信息一致？**（before/after 是否准确）
3. **是否有原始记录在去重/合并过程中被错误丢弃？**

| 原始记录 | 是否在方案中 | 方案对应编号 | 一致性 |
|----------|-------------|-------------|--------|
| R1 (Git diff, user.service.ts) | ✅ | C1 | ✅ before/after 一致 |
| R2 (对话第3轮, controller) | ✅ | C2 | ✅ 一致 |
| R3 (Git diff, rateLimit.ts) | ✅ | C3 | ✅ 一致 |
| R4 (推断, auth.module.ts) | ✅ | C4 | ⚠️ 来源为推断，待确认 |

#### 4.2 精确性核对
- **before 代码块核对**：方案中的 before 是否与实际的改前代码**逐字一致**？（从 Git diff 的 `-` 行 / 对话历史的 old_str 验证）
- **after 代码块核对**：方案中的 after 是否与当前的**实际代码**逐字一致？（读取当前代码文件验证）
- **位置锚点核对**：方案标注的文件路径和行号/函数名是否与实际代码吻合？

#### 4.3 逆向可复现性验证
- **模拟执行**：假设另一个模型拿到这份方案，能否据此在原始代码上精确复现每一处修改？
- **锚点可定位性**：每个 before 代码块是否足以在原始代码中唯一定位？（太短 → 可能多处匹配；太长 → 冗余但安全）
- **无歧义性**：修改说明是否清晰无歧义？是否有"改那个东西"这类模糊表述？

#### 4.4 不满足时的处理
- **某修改被遗漏** → 补充到方案文档中，重新编号。
- **before/after 不准确** → 修正为与实际代码逐字一致的内容。
- **锚点不可定位** → 扩充 before 代码块，增加上下文行，使其唯一可匹配。
- **信息来源不可靠** → 降级标注置信度，列出待确认项。

---

### 阶段 5：一致性确认与交付（Deliver）
**目标**：向用户呈现方案文档，确认其完整性与准确性，最终交付。**方案文档是给用户（或另一个模型）使用的，必须确保用户认可其完整性。**

#### 5.1 交付时向用户呈现
1. **方案文档概览**（共 N 处变更、M 个文件、类型分布）。
2. **信息来源说明**（方案基于哪些来源生成、各来源的可靠性）。
3. **待确认项**（哪些记录来自低可靠性来源、哪些修改存在不确定）。
4. **请用户确认**：方案是否覆盖了所有修改？有无遗漏或错误？

#### 5.2 用户反馈后的处理
- 用户指出遗漏 → 回到阶段 1 补充收集 → 阶段 2-4 重新整理 → 更新方案。
- 用户指出错误 → 核实并修正。
- 用户确认无误 → 交付最终方案文档。

## 四、按修改类型的整理策略矩阵（Pattern Library）

针对第二章的变更类型，给出**整理要点 + before/after 书写规范 + 易错点**。这是经验沉淀的"速查表"。

### ① 新增文件
- **整理**：在方案中列出完整文件路径与完整文件内容。before 为空（标注"新建文件"）。
- **书写**：after 给出文件完整内容（用代码块包裹，标注语言）。说明文件的作用。
- **易错**：只给了文件路径没给完整内容；遗漏了文件所在的目录创建步骤。

### ② 删除文件
- **整理**：列出被删除的文件路径。after 标注"（文件已删除）"。
- **书写**：before 可标注文件曾存在及其用途；说明删除原因。
- **易错**：遗漏了"删除文件后需清理其他文件中的 import/引用"这一连带变更。

### ③ 修改内容（替换 before → after）
- **整理**：给出精确的 before 代码块（≥3 行，含足够上下文以唯一定位）和 after 代码块。
- **书写**：before 和 after 用成对的代码块呈现，明显标注"改前""改后"。标注文件路径与行号/函数名。
- **易错**：before 代码块太短导致无法唯一定位；before/after 与实际代码有细微差异（空格、引号）；遗漏了同一文件的其他改动。

### ④ 插入代码
- **整理**：标注精确插入位置（"函数 X 的第一行之后"/"第 N 行之后"/"某注释标记之前"），给出插入的代码块。
- **书写**：before 标注插入点的上下文代码（插入位置的前后各几行），after 展示插入后的完整上下文。
- **易错**：插入位置描述模糊（"在函数里加一段"——加在函数的哪里？）；插入代码的缩进与上下文不一致。

### ⑤ 删除代码
- **整理**：给出被删除的代码块（before），after 展示删除后该位置的上下文。
- **书写**：明确标注被删除的代码范围；说明删除原因。
- **易错**：删除范围边界不清（多删或少删了几行）；删除后留下的语法残留（悬空逗号/括号）未处理。

### ⑥ 重命名 / 移动文件
- **整理**：列出旧路径 → 新路径。**必须同时记录所有因路径变更而需更新的 import/require 引用**（这是连带变更）。
- **书写**：明确标注"文件从 A 移至 B"；列出所有需要同步修改的引用文件与引用处。
- **易错**：只记录了文件移动，遗漏了所有引用该文件路径的 import 更新——导致方案执行后大面积编译错误。

### ⑦ 配置变更
- **整理**：精确到配置项级别（哪个配置文件的哪个键值改了）。
- **书写**：before 给出原配置片段，after 给出新配置片段。注意 JSON/YAML 的语法正确性。
- **易错**：覆盖式描述（"把整个配置文件改了"）而非精确到键值；破坏了 JSON/YAML 语法。

### ⑧ 依赖变更
- **整理**：精确到依赖名称与版本号。区分"新增""升级""移除"。
- **书写**：before 给出原依赖片段，after 给出新依赖片段。**版本号必须精确**（不用 `latest`/`^` 模糊化）。
- **易错**：版本号模糊化（"升级了 react"——升级到什么版本？）；遗漏了锁文件（package-lock.json / yarn.lock）的同步更新提示。

## 五、修改信息收集工具与操作速查

> 本 Skill 全程为**只读操作**（收集、整理、生成文档），**不修改项目的任何代码**。

### 5.1 版本控制信息获取（来源①，优先使用）
- **`git status`**：查看哪些文件被修改/新增/删除。
- **`git diff`**：未暂存的逐行变更（模型修改最常见的状态）。
- **`git diff --cached`**：已暂存的逐行变更。
- **`git diff <commit>..HEAD`**：从指定基线到当前的累计变更。
- **`git log --oneline --stat -<N>`**：最近 N 次提交涉及的文件与行数统计。
- **`git log --diff-filter=AMD --name-only`**：只看新增(A)/修改(M)/删除(D)的文件清单。

### 5.2 代码内容读取（来源②③④的补充）
- **Read**：读取当前文件内容，确认 after 代码块与实际代码一致。
- **Grep**：搜索特定代码片段，验证 before 代码块在原始代码中的存在性与唯一性。
- **Glob**：按模式查找文件，核对涉及文件清单的完整性。
- **LS**：列目录结构，确认文件的新增/删除。

### 5.3 对话历史回溯（来源③）
- 回溯当前会话中模型执行过的每次文件编辑操作（工具调用记录）。
- 重点关注：操作类型（Write / SearchReplace / Edit / Delete）、操作目标（文件路径）、操作内容（old_str / new_str）。

### 5.4 操作安全约定
- **全程只读**：不修改、不删除、不新增项目中的任何代码文件。本阶段只整理文档。
- **方案文档输出位置**：将生成的方案文档保存到用户指定的位置或项目根目录，文件名建议含日期或版本标识（如 `modification-plan-20260622.md`）。

## 六、信息不完备时的降级策略（健壮性核心）

现实中，修改信息**往往无法 100% 完整收集**——对话可能被截断、项目可能无版本控制、模型可能遗忘早期操作。本 Skill 必须能优雅降级，而非卡死或编造。

| 缺失/不确定项 | 降级策略 |
|---------------|----------|
| **项目无版本控制（来源①不可用）** | 降级到来源②③④；追问用户是否有修改前的代码快照；以当前代码为 after、对话历史为 before 补全；整体置信度降级 |
| **对话历史被截断（来源③部分缺失）** | 以来源①②为主、来源④（推断）为辅；对推断的修改内容标注"待确认"；追问用户是否还记得被截断的修改 |
| **改前内容(before)无法获取** | 追问用户是否有原始代码；或从 Git 历史提取；若均不可得，标注"before 内容待确认"，给出当前代码(after)与修改说明 |
| **同一位置改了又改（多次反复修改）** | 按净变更原则：只记录最初 before → 最终 after；在备注中标注"此处经历了多次反复修改" |
| **无法区分模型修改与用户手动修改** | 追问用户确认；或按用户指示只整理模型做的修改；标注"以下变更均基于模型操作记录" |
| **部分修改疑似被撤销** | 以当前代码的实际状态为最终事实；在方案中标注"对话第N轮曾做此修改，但当前代码中未体现，疑似已撤销" |
| **修改记录与当前代码矛盾** | 以当前代码为准（它是最终事实）；标注矛盾点，请用户确认实际意图 |
| **修改总量极大（几十个文件）** | 按模块/功能分组整理，先给出文件级总览，再逐文件展开详情；考虑分多份方案文档 |

**置信度标注规则**：
- **高**：方案基于版本控制 diff 生成（来源①），所有变更均有精确的 before/after 代码块，且与当前代码逐字核对一致。
- **中**：方案主要基于来源②③，大部分变更有精确的 before/after，个别变更的 before 来自推断或对话回忆（已标注）。
- **低**：方案主要依赖来源③④（对话回忆 + 推断），多个变更的 before/after 不确定，或对话历史截断导致可能有遗漏——必须显式声明"方案可能不完整"并列出所有不确定点。

## 七、防误判：常见陷阱（严谨性核心）

修改方案整理最怕"遗漏而不自知"、"编造而不标注"、"记录失真而不校验"。以下陷阱必须主动规避：

1. **遗漏小改动**：只记了大改动（新增文件、重写函数），忽略了小改动（一个 import、一个常量值、一行配置）。**必须逐文件逐行核对 diff，不可只凭印象。**
2. **臆造改前内容**：记不清 before 是什么样的，就"编"一段合理的代码当作 before。**坚决禁止——before 必须有据可查（Git diff / 对话记录 / 用户确认）。**
3. **混淆中间态与最终态**：把对话中间某一轮的代码状态当成最终 after。**after 必须与当前实际代码逐字核对一致。**
4. **遗漏附带/连带改动**：移动文件后改了 import、删除函数后清理了引用——这些连带改动容易被忽略。**必须全局搜索被改文件的引用处。**
5. **before 代码块不足以唯一定位**：只给了一行 `return null` 作为 before——代码库中可能有几十处。**before 至少 3-5 行，含足够上下文。**
6. **格式变更混入内容变更**：IDE 自动格式化（Prettier/ESLint fix）产生的空格/引号变化被当成"修改"记录。**应区分格式变更与内容变更。**
7. **依赖版本模糊化**：记录"升级了 react"而不写明版本号。**版本号必须精确到具体版本。**
8. **只凭模型回忆不做校验**：模型说"我改了 XX"，就直接写入方案而不核对当前代码。**回忆必须用当前代码交叉验证。**
9. **去重时误删有效记录**：把两处不同的修改误判为重复而合并丢失。**去重时必须确认 before/after/位置三者完全一致才合并。**
10. **方案文档自身有歧义**：修改说明写成"改了一下登录逻辑"——怎么改的？before/after 在哪？**说明必须具体、可定位、无歧义。**
11. **不做最终校验**：方案写完就交付，不核对完整性、不验证精确性。**阶段 4 的校验不可省略。**
12. **忽视修改时序中的撤销**：对话中先加了某段代码、后来又删了——方案中不应记录这个"加了又删"的无效操作（除非用户要求完整时序）。**只记录净变更。**

## 八、安全与合规约束

- **全程只读**：本 Skill 的执行过程**不修改、不删除、不新增**项目中的任何代码文件。修改已在之前完成，本阶段只负责整理成文。唯一的"写入"是生成方案文档（md 文件）本身。
- **忠实记录，不美化不编造**：方案记录的是**已真实发生的修改**，不可为了"好看"而美化、重写、补全未确认的内容。有据则录，无据则标注。
- **最小授权**：只访问与修改相关的代码与配置；不主动读取无关的敏感数据（密钥、隐私数据）。
- **不越权**：不重启服务、不改配置、不触碰生产环境。本 Skill 只产出文档。
- **可追溯**：每处变更标注信息来源与可靠性，便于用户复核与验证。
- **偏差透明化**：所有不确定的、推断的、可能遗漏的修改，必须在方案中显式标注，不隐瞒。
- **诚实标注边界**：当信息不完整、来源不可靠、或无法保证方案完整性时，明确告知用户，不编造"方案已完整覆盖所有修改"的假象。
- **建议版本控制**：若项目未纳入版本控制，主动建议用户纳入，便于未来的变更追踪与方案生成。
