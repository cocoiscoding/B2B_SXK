# Feature Localization / 功能定位与解析指南

> 本 Skill 用于指导大模型在一个**庞大、相对陌生（或部分陌生）的项目代码库**中，根据用户提出的**一个功能名称或功能描述**，以系统架构师视角，**严谨、可追溯、不臆测**地定位到该功能的**实现位置**，并完整解析其**涉及的类、方法、架构分层、数据流向**，最终向用户清晰、简洁地讲清楚"这个功能是什么、在哪里、怎么跑通的"。
>
> 一句话定位：**功能定位的本质是"由果索因的结构化考古"——从一个模糊的功能意图出发，正向建立"入口→逻辑→数据"的实现全貌，而非零散罗列相关文件。**

---

## 一、技能定位与适用场景

### 1.1 适用场景
- 用户在大型项目（单体 / 前后端分离 / monorepo / 微服务）中想了解某个功能"是怎么实现的"。
- 用户提出形如："登录功能在哪个文件？""订单支付的流程是什么？""导出报表功能涉及哪些类？""帮我讲讲权限校验是怎么做的"。
- 需要**定位**功能入口、**梳理**核心类与方法、**还原**架构分层、**追踪**端到端数据流向。
- 既要找到"代码在哪"，更要回答"它如何协作完成这个功能"。

### 1.2 不适用场景（边界）
- **修 Bug / 排障**：以"现象 + 报错"定位根因 → 交给 `bug-localization`。
- **跑起来项目**：以"如何启动"为目标 → 交给 `project-bootstrap-guide`。
- **生成项目全景上下文文档**：以"全局架构与规范"为目标 → 交给 `project-init-context`。
- **开发新功能 / 重构 / 性能优化** → 非本 Skill 范畴。
- 纯概念解释、非本项目代码问题 → 直接回答，无需本 Skill。

> **与 bug-localization 的关键区别**：bug 定位是"逆向因果溯源"（从错误现象找根因）；功能定位是"正向结构考古"（从功能意图重建实现全貌）。两者方法论相似但方向相反。

### 1.3 目标产物（功能解析报告）
经过本 Skill 的分析后，应输出一份结构化**功能解析报告**，至少包含：
1. **一句话功能定义**（这个功能做什么）。
2. **功能边界**（包含什么、不包含什么，避免越界误解）。
3. **入口点清单**（前端入口 / 后端入口 / API 端点 / 触发器）。
4. **核心类与方法清单**（按分层列出，每项附 `文件:行号`）。
5. **架构分层与职责**（该功能在分层架构中的位置）。
6. **端到端数据流向**（从前端到数据库的完整链路）。
7. **涉及的数据实体与存储**（表 / 集合 / 缓存键）。
8. **关键依赖与外部交互**（第三方服务、消息队列、定时任务等）。
9. **置信度（高/中/低）**与不确定点 / 待确认项。

### 1.4 核心原则（心智模型）
- **只读分析（Read-Only）**：定位与解析阶段**不修改任何项目代码**，理解职责与改动职责分离。
- **证据驱动，拒绝臆测**：每一个"某个类负责某事"的结论都必须有代码或配置证据支撑；无证据的推断必须标注为"假设"。
- **入口优先，逐层下钻**：先锚定功能入口，再沿调用链展开，避免在无关代码里大海捞针。
- **意图先行，边界清晰**：动手前先和用户对齐"这个功能的确切含义与范围"，防止答非所问。
- **全链路视角**：前端 / 后端 / 数据 / 中间件 / 异步任务都可能是一个功能的环节，不要只盯一处。
- **相关性 ≠ 实现主体**：搜到某关键词不代表它就是该功能的核心实现，需区分"核心 / 调用方 / 边缘"。

## 二、第一步：功能需求的结构化解析（Intent Triage）

拿到用户的功能描述后，**先不要急着搜代码**。先把模糊的功能意图拆成结构化字段，并对齐边界。这一步决定了后续定位的方向是否正确。

### 2.1 必须澄清的 6 个要素

| 要素 | 说明 | 示例 / 追问话术 |
|------|------|----------------|
| **功能名称/意图** | 用户口中的功能到底是什么 | "登录"指账密登录？还是含 OAuth/扫码/短信？ |
| **触发方式** | 谁触发、从哪触发 | 用户点按钮 / 定时任务 / API 被调 / 消息消费 |
| **目标范围** | 关心前端 / 后端 / 全链路 | 用户要"前端流程"还是"后端逻辑"还是"全链路" |
| **技术栈侧重点** | 是否已知前端或后端在哪 | 若项目庞大，先确定从哪端切入 |
| **深度要求** | 要"概览"还是要"逐方法精读" | 概览式回答 vs. 完整调用链 |
| **已知线索** | 用户已知的文件名 / 类名 / API 路径 | 任何已知锚点都能大幅加速定位 |

### 2.2 功能分类（决定定位策略）
将目标功能归入以下类别之一，后续定位矩阵（第四章）按此分发：

- **A. 用户交互型功能**：有 UI 入口（点击/提交），如登录、下单、导出报表。
- **B. 接口/API 型功能**：以 API 端点为入口，如"获取用户列表接口"。
- **C. 后台/异步型功能**：定时任务、消息消费、批处理，如"每日对账""订单超时取消"。
- **D. 横切/基础设施型功能**：鉴权、限流、日志、权限校验，贯穿多处。
- **E. 数据/领域型功能**：以数据实体为核心的逻辑，如"库存扣减""积分计算"。

### 2.3 信息完备度评估与追问清单（健壮性保障）
现实中用户的功能描述往往**模糊、歧义**。遇到歧义，**必须先主动追问**，而不是基于猜测硬找。追问时给出具体、可执行的指引：

- 功能名称有歧义 → 给出 2-3 种可能的解释让用户选择（"这里的'支付'是指收银台流程，还是支付回调处理？"）。
- 不确定前端还是后端 → 请用户说明关注点，或明确"我会先做全链路概览"。
- 功能词太泛（"用户管理"覆盖面极大）→ 请用户聚焦到某个具体操作（增/删/改/查/某个子流程）。
- 用户给了中文功能名但项目可能用英文 → 先翻译/猜测可能的英文关键词，结合搜索验证。

> **原则**：宁可先对齐一句"你要的到底是什么"，也不要在错误的功能范围里跑得很远。功能范围一旦跑偏，整份报告都将作废。

## 三、核心方法论：五阶段功能考古流水线

面对任何功能定位请求，严格按以下五个阶段推进。**每个阶段都有明确的"要做什么、得到什么结论、不满足则回退"**。

```
功能意图 ──► [阶段0 技术栈与结构侦察] ──► [阶段1 锚定功能入口] ──► [阶段2 核心逻辑追踪]
                                                                        │
       [阶段4 验证与确认] ◄── [阶段3 架构与数据流串联] ◄────────────────────┘
```

### 阶段 0：项目技术栈与结构侦察（Recon）
**目标**：在搜代码前，先建立"项目地图"，确定用什么手段去搜、去哪里搜。**若已有同项目的前序分析（project-init-context / project-bootstrap-guide 产出），优先复用，避免重复侦察。**

**侦察动作**：
1. 识别技术栈指纹（`package.json` / `pom.xml` / `go.mod` / `requirements.txt` / `composer.json` / `Cargo.toml` 等）。
2. 判定**前端框架与路由约定**：
   - 文件式路由：Next.js（`app/`、`pages/`）、Nuxt（`pages/`）→ 文件路径即路由。
   - 配置式路由：Vue Router / React Router 的路由表（搜 `routes`、`path:`、`createRouter`）。
3. 判定**后端框架与路由分发方式**（Spring 的 `@RequestMapping` / `@GetMapping`、Express 的 `app.get`、Django 的 `urls.py`、FastAPI 的 `APIRouter`、Gin 的 `router.GET`、NestJS 的 `@Controller` + `@Get`）。
4. 判定**分层架构与目录职责**（见 3.x 速查表）：识别 controller / service / repository / model / module / domain 等分层位置。
5. 判定**API 风格与通信约定**（REST / GraphQL / gRPC / tRPC）、**统一返回结构**（`{code, msg, data}`）、**鉴权方式**。

**结论产出**：一份"项目地图"——前端入口在哪、后端入口在哪、路由如何组织、分层如何划分、API 如何命名。后续阶段据此选择搜索策略与切入层。

---

### 阶段 1：锚定功能入口（Anchor）
**目标**：把"功能意图"映射到**一个或多个具体的代码入口**。这是整个流水线最关键的起点，**入口锚错，全盘皆错**。

按**功能类型（第二章 2.2）**选择主路径，再按**可靠性从高到低**依次尝试定位策略，命中即停：

| 优先级 | 策略 | 手法 | 适用 |
|--------|------|------|------|
| 1 | **已知线索直查** | 用用户提供的文件名/类名/API 路径直接定位 | 用户给了任何锚点 |
| 2 | **API 路径串联** | 搜 API 路径（如 `/api/login`），前端调用处 ↔ 后端定义处一举打通 | A/B 类，**最强抓手** |
| 3 | **文案搜索** | 用 UI 可见文案（按钮/标题/提示）全局精确搜索 | A 类用户交互型 |
| 4 | **路由反查页面** | 由 URL/路由定位页面组件，再找功能入口 | A 类文件/配置路由 |
| 5 | **领域关键词** | 搜领域名词的英文/拼音变体（login/signin/auth） | B/E 类无 UI 入口 |
| 6 | **触发器反查** | 搜定时任务（cron/`@Scheduled`/celery）、消息消费者、事件监听 | C 类异步型 |
| 7 | **横切点反查** | 搜中间件/拦截器/守卫/装饰器（`@UseGuard`/`@PreAuthorize`/`middleware`） | D 类横切型 |

**关键产出**：
- 该功能的**前端入口**（页面组件 + 触发元素的事件处理器，如 `onClick` / `@click` / `onSubmit`）。
- 该功能的**后端入口**（Controller 方法 / 路由处理函数 / GraphQL resolver / 消息处理器）。
- 若是 API 型，记录**请求方法 + 路径 + 入参 + 返回**。

**常见陷阱**：
- 文案来自**国际化 i18n 文件**（搜不到原文）→ 先到 `locales/` / `i18n/` 找 key，再用 key 反查使用处。
- 入口在**通用组件 / 高阶组件 / 循环渲染**中 → 沿 props/slot 向上追溯到具体调用方。
- 同名功能存在**多套实现**（旧版/新版/不同端）→ 全部列出，标注差异，请用户确认要哪一套。
- 关键词命中过多（"login" 出现几十处）→ 叠加路由 / API 路径 / 目录分层缩小范围。

### 阶段 2：核心逻辑追踪（类与方法）
**目标**：从入口出发，沿**控制流与数据流**逐层追踪，识别实现该功能的**核心类与方法**，建立"调用链"。

**追踪要点（按分层下钻）**：
1. **Controller / Handler 层**：读入口方法，提取它调用了哪些 service / 用例 / 依赖。
2. **Service / UseCase 层**：这是业务逻辑核心，逐方法阅读——记录每个核心方法的**职责、关键分支、调用的下游**。
3. **Repository / DAO / ORM 层**：识别数据访问方法与涉及的实体（Entity / Model）。
4. **领域 / 工具层**：识别领域服务、策略类、计算逻辑、第三方封装。

**关键产出（核心类与方法清单）**，每一项标注 `文件:行号`：
| 分层 | 类/方法 | 文件:行号 | 职责 |
|------|---------|-----------|------|
| 入口 | `AuthController.login()` | `auth.controller.ts:42` | 接收登录请求、参数校验 |
| 业务 | `AuthService.login()` | `auth.service.ts:88` | 校验密码、生成 token |
| 数据 | `UserRepository.findByUsername()` | `user.repository.ts:31` | 查询用户记录 |
| 工具 | `JwtUtil.sign()` | `jwt.util.ts:15` | 签发 JWT |

**追踪要点（不要漏掉的非主线环节）**：
- **参数校验**：DTO / `class-validator` / `zod` / `pydantic` / `@Valid`，校验发生在哪。
- **鉴权 / 权限**：守卫 / 拦截器 / 中间件 / 注解（`@PreAuthorize` / `@UseGuard`）。
- **异常处理**：自定义异常 / 全局异常过滤器 / 错误码定义在哪。
- **事务边界**：`@Transactional` / `unitOfWork` / 手动事务，保证数据一致性的位置。
- **状态变更**：前端 `state` / `store` / `context` 的读写、后端缓存读写。

**常见陷阱**：
- **同名方法 / 多重载**：确认追踪的是入口实际调用的那个重载，勿张冠李戴。
- **接口与实现分离**：入口依赖的是接口（`IAuthService`），需找到其实现类（DI 容器 / `@Injectable` / `@Autowired` / `provides`）。
- **元编程 / 反射 / 装饰器魔法**：NestJS / Spring 大量依赖注解装配，行为不在方法体内而在框架，需理解框架约定。
- **过度追踪第三方库**：到第三方包边界即停，不要深入 `node_modules` / 第三方 jar。

### 阶段 3：架构与数据流串联（Assembly）
**目标**：把阶段 1-2 的散点串成**一条端到端的实现链路**，并阐明该功能在整体架构中的位置。

#### 3.1 分层归属（功能在架构中的位置）
明确该功能横跨哪些层、各层职责如何分工：
```
触发层(UI/API/Task) → 接入层(Controller/路由) → 业务层(Service/UseCase)
        → 数据层(Repository/ORM/DB) →（可选）缓存/消息/第三方
```

#### 3.2 端到端数据流向（最核心产出）
画出贯穿前后端/异步链路的完整数据流，标注每一步的**数据形态变化**：

```
用户操作(前端) → 事件处理 → 构造请求(URL/参数) → HTTP/消息
   → 后端路由 → Controller(参数校验) → Service(业务/事务)
        → Repository → DB（表A / 表B）
   → 返回 DTO → 响应 ← 前端处理 ←────────────────────┘
        → 更新 state/store → 重新渲染
```

**必须交代清楚**：
- 请求的**路径、方法、入参字段、返回字段**。
- 经过哪些**数据实体 / 表**，是否有**写操作**（增删改）、是否在**事务**内。
- 是否有**缓存**参与（先读缓存？写后失效？）、是否有**异步副作用**（发消息 / 发邮件 / 写日志）。
- 数据的**形态转换点**（Entity ↔ DTO、序列化/反序列化、加密/解密）。

#### 3.3 同步链路与异步链路分离
- **同步主链路**：用户请求 → 即时返回的部分（如登录返回 token）。
- **异步副作用**：消息队列消费、定时补偿、Webhook 回调（如登录后异步发欢迎邮件、支付成功后异步通知）。
- 二者要分开讲清楚，避免把异步逻辑误当成主流程的必经环节。

**产出**：一条"该功能的完整实现链路图（文字版）"+ 分层职责说明 + 数据实体清单 + 关键依赖清单。

---

### 阶段 4：验证与确认
**目标**：把"追踪到的链路"升级为"已确认的实现全貌"，避免遗漏核心环节或纳入无关代码。

**确认标准（尽量同时满足）**：
1. **闭环自洽**：从入口到数据落地再到（可能的）返回，链路每环都有 `文件:行号` 支撑，无断裂。
2. **职责不重不漏**：核心类/方法覆盖了该功能的主要逻辑，没有被边缘代码带偏。
3. **与功能定义吻合**：解析出的实现确实能完成用户所说功能；若发现实现与"功能名"的字面含义有出入（如"登录"实际含单点登录分支），应如实说明。

**验证手法（只读）**：
- 沿链路代入**一次典型调用**做"心智执行"，看每一步衔接是否成立。
- 检查是否有**未被纳入的分支**（不同角色、不同状态、异常路径）需要补充。
- 交叉验证：用第二个入口线索（如同时搜前端调用与后端定义）互证链路完整。

**不满足时的处理**：
- 若链路有断裂（某环节找不到实现）→ 标注"待确认"，列出可能的几种走向，请用户提供线索或确认。
- 若发现功能实际比描述更复杂（多分支/多入口）→ 如实列出全部入口与分支，按主次排序。

## 四、按功能类型的定位策略矩阵（Pattern Library）

针对第二章的功能分类，给出**定位主路径 + 对应手法**。这是经验沉淀的"速查表"。

### A. 用户交互型功能（有 UI 入口）
**定位主路径**：UI 文案/元素 → 页面组件 → 事件处理器 → API 请求 → 后端。
**手法**：
1. 用按钮/标题/提示文案精确搜索定位页面组件。
2. 在组件内找事件绑定（`onClick` / `@click` / `onSubmit`），进 handler 看发起的请求（`fetch` / `axios` / `useQuery`）。
3. 用请求路径在后端反查 Controller，进入阶段 2。

### B. 接口/API 型功能（以 API 为入口）
**定位主路径**：API 路径 → 路由注册 → Controller → Service。
**手法**：
1. 搜 API 路径字符串（`/api/xxx`），前端调用处与后端定义处一并找到（**最强抓手**）。
2. 后端按框架路由约定定位 handler（`@GetMapping` / `app.get` / `APIRouter` / `@Controller`）。
3. 看 Swagger / OpenAPI / 接口文档（若存在）快速获得入参出参契约。

### C. 后台/异步型功能（定时/消息/批处理）
**定位主路径**：触发器 → 任务入口 → 业务逻辑 → 数据落地。
**手法**：
1. 搜定时任务调度（cron 表达式 / `@Scheduled` / celery `@app.task` / bullmq / quartz / xxl-job）。
2. 搜消息消费者（`@RabbitListener` / `@KafkaListener` / `channel.consume` / `subscribe`）。
3. 搜事件监听器（Spring `@EventListener` / NestJS `@OnEvent` / EventEmitter）。
4. 注意：这类功能**没有 UI 入口**，易被忽略，需主动从调度/消费层切入。

### D. 横切/基础设施型功能（鉴权/限流/日志/权限）
**定位主路径**：框架扩展点 → 中间件/拦截器/守卫/装饰器 → 全局生效点。
**手法**：
1. 搜横切组件注册处（`app.use(middleware)` / `@UseInterceptors` / `app.module providers` / Spring `WebMvcConfigurer` / `SecurityFilterChain`）。
2. 搜装饰器/注解定义（`@RequirePermission` / `@Roles` / `@RateLimit`）及其处理逻辑。
3. **此类功能往往贯穿全局**，需说明其生效范围（哪些路由/方法会经过它）。

### E. 数据/领域型功能（库存/积分/计算）
**定位主路径**：领域实体 → 领域服务/策略 → 持久化。
**手法**：
1. 从实体/模型（`@Entity` / `schema` / `model`）出发，找操作它的领域服务。
2. 关注**事务边界**与**并发控制**（锁 / 乐观锁版本号 / 原子操作）。
3. 关注**一致性约束**（扣减后不能为负、积分变动有流水记录）。

## 五、定位工具与命令速查（均为只读安全）

> 优先使用 IDE / 大模型的内置搜索能力（Glob / Grep / Read）。**禁止执行任何写入、删除、网络写操作。**

### 5.1 全局精确搜索（定位入口的核心武器）
- **按 API 路径串联前后端**：搜 `/api/xxx`，前端调用处 ↔ 后端定义处一举打通。
- **按文案找页面**：精确匹配按钮/标题文案（注意 i18n key 转译）。
- **按路由找页面**：搜路由 path（`/system/user`）、路由 name。
- **按领域关键词**：搜功能的英文/拼音变体（login/signin/auth、pay/payment/charge）。
- **按注解/装饰器找入口**：`@Controller`、`@GetMapping`、`@Scheduled`、`@RabbitListener`、`@UseGuard`。

### 5.2 结构化检索（找文件而非找文本）
- **Glob 按命名约定找文件**：`**/auth/**`、`**/*controller*`、`**/*service*`、`**/*.entity.*`、`**/dto/**`。
- **目录职责推断**：先 LS 一层，依目录名（controller/service/repository/model/module）判断分层。
- **入口文件定位**：`**/main.{ts,js}`、`**/Application.java`、`**/manage.py`、`**/main.go`、`**/app.py`。

### 5.3 辅助证据来源（只读）
- **接口文档**：Swagger / OpenAPI（`swagger.json` / `/swagger-ui` / `@Api` / FastAPI 自动文档）。
- **数据库 schema**：`*.prisma` / `migrations/` / `*.sql` / `@Entity` 类，佐证数据实体与字段。
- **路由注册表**：前端路由配置、后端路由汇总文件。
- **配置文件**：`application.yml` / `.env.example`，佐证外部依赖（Redis/MQ/第三方 Key）。

## 六、信息不完备时的降级策略（健壮性核心）

现实请求中，用户的功能描述、项目信息往往不全。本 Skill 必须能优雅降级，而非卡死或乱猜。

| 缺失项 | 降级策略 |
|--------|----------|
| 功能含义有歧义 | 给出 2-3 种解释请用户选择，或先按"最常见含义"做概览并标注假设 |
| 不知前端还是后端 | 先做全链路概览，分别标注前端/后端部分，再请用户指定深入方向 |
| 无任何已知线索 | 从功能名翻译英文关键词 + 路由/API 路径搜索切入；命中少时扩展同义词 |
| 功能词命中过多 | 叠加路由 / 目录分层 / API 路径过滤；优先看 controller/service 层命中 |
| 找不到入口 | 换关键词变体 / 从数据实体反查使用方 / 检查是否异步触发型功能 |
| 链路某环断裂 | 标注"待确认"，列出可能的走向，请用户提供线索或日志佐证 |
| 功能跨多服务/多仓库 | 先画出涉及的服务/仓库边界，逐个定位其内部链路，再串联跨服务调用 |

**置信度标注规则**：
- **高**：入口明确 + 完整调用链有代码证据 + 数据流闭环。
- **中**：入口与主线清晰，但个别分支/异步副作用缺证据。
- **低**：仅找到部分相关代码，链路未闭环——必须显式声明为"待验证"并列出待确认项。

## 七、防误判：常见陷阱（严谨性核心）

功能定位最怕"自信地讲错全貌"或"把无关代码当核心"。以下陷阱必须主动规避：

1. **同名歧义**：`login` 可能指账密登录、OAuth、SSO、扫码，需结合上下文与边界确认。
2. **接口/实现混淆**：追踪到接口就停，没找到真正实现类，结论悬空。
3. **核心与边缘不分**：把校验/日志/调试代码当成功能核心，喧宾夺主。
4. **新旧实现并存**：大型项目常有重构遗留，新旧入口都在，需确认当前生效的是哪套。
5. **跨服务边界断裂**：微服务架构下一个功能可能跨多个服务，只看单服务会漏掉关键环节。
6. **过度展开目录**：把整个模块文件全列出既冗长又无用，应聚焦功能直接相关的类/方法。
7. **忽视异步链路**：只讲同步主链路，漏掉消息消费/定时补偿，导致功能全貌不完整。
8. **臆造不存在的调用**：方法名像但实际没被调用，必须沿真实调用链验证，不可凭命名猜测调用关系。
9. **版本/分支差异**：当前代码可能与线上版本不同，结论基于"当前代码库现状"，必要时请用户确认版本。
10. **解读≠改动**：本 Skill 产出是**理解性解析**，不要在解析未完成时就动手改代码（合规约束）。

## 八、安全与合规约束（Read-Only）

- **只读**：定位与解析全程**不得修改、删除、新增**项目文件；不得执行写库、写文件、发外部请求的命令。
- **最小授权**：只访问解析所必需的代码与配置；不主动读取与功能无关的敏感数据（密钥、个人隐私数据）。
- **职责分离**：解析报告中的"优化/改进建议"明确标注为**可选参考**，是否实施由用户决定。
- **不越权**：不重启服务、不改配置、不触碰生产环境数据；如需运行时信息，由用户在其受控环境内获取并反馈。
- **可追溯**：所有结论附 `文件:行号` 或代码片段证据，便于用户复核。
- **诚实标注边界**：当功能超出单一代码库范围（如跨仓库、跨团队）或信息不足时，明确告知，不编造。

## 十、工作流总览（速记）

```
1. 解析意图（6要素 + 功能分类 + 边界对齐，歧义则追问）
        │
2. 阶段0  技术栈/结构侦察 ──► 建立项目地图（复用已有分析）
        │
3. 阶段1  锚定功能入口（API路径/文案/路由/触发器…）──► 前端入口 + 后端入口
        │
4. 阶段2  核心逻辑追踪（Controller→Service→Repository，类/方法+行号）──► 调用链
        │
5. 阶段3  架构与数据流串联（分层归属 + 端到端数据流 + 同步/异步分离）
        │

        │
7. 输出功能解析报告（定义 + 入口 + 类方法 + 数据流 + 置信度）
```

**贯穿全程的红线**：只读、证据驱动、入口优先逐层下钻、意图先行边界清晰、相关性≠实现主体、信息不全先追问。
